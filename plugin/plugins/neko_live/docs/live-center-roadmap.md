# NEKO Live 路线图

> 更新：2026-07-18。本文只记录“现在做到哪、接下来做什么、哪些事延期”。架构和历史过程不在这里重复。

## 当前阶段

NEKO Live 已从头像锐评切片发展为直播互动中心。B 站主链路、Independent Mode、可信支持事件调度、观众档案、四区 Hosted UI 与运行态解释已经形成完整闭环；实验性抖音只读 bridge 已接入 provider-neutral 边界。

当前目标不是继续堆功能，而是把现有能力整理为可维护、可分发、可真机验证的产品基线。

## 已完成的产品骨架

- 账号与直播间分离，查询后再次确认房间；
- 控制台完成登录、房间、主题、节奏与开始/结束直播；
- 新观众首评、后续弹幕接话、暖场、冷场和主动营业；
- Gift / SC / Guard 可信证据、去重、连击、优先队列与短句致谢；
- 队列硬上限、会话代际和切房 worker 回收；
- 本场统计、观众档案、重置/删除与隐私控制；
- Runtime Timeline、Live Status、最近结果和开发者沙盒；
- B 站登录凭据加密与显式无账号兜底；
- Hosted UI 四区结构、开发者工具条件显示和低频可见性刷新；
- 永久插件 ID `neko_live` 与旧 `neko_roast` 身份迁移入口。

完成状态不等于发布验收。真实直播质量仍以 [产品计划](independent-mode-product-plan.md#发布验收) 和 [试播指南](solo-stream-test-guide.md) 为准。

## 当前工作顺序

### 1. 文档与分发收口

- 建立单一事实来源，删除重复历史流水与易失真数字；
- 保证 README、插件内 guide 和开发文档链接完整；
- 验证导入、同 ID 升级、旧 ID 迁移、失败回滚与 profile 保留；
- 输出可供另一台机器安装的测试包，并明确版本和回滚方法。

### 2. 离线可靠性回归

- 支持事件：伪礼物、重复 provider ID、迟到连击、优先级与队列上限；
- 会话竞态：重复开始、连接中换房、断线重连、切房和旧 worker 回收；
- UI 状态：弹窗草稿、保存快照、刷新失败、按钮 pending 与直播状态；
- 隐私：日志、audit、viewer profile、dashboard 和 monitor 的敏感字段负例；
- 分发：`panel_compat.tsx`、8 locale、bridge 生命周期和插件 CLI check。

### 3. 真机测试恢复后

- 先做 B 站 dry-run 与低频真实输出；
- 再做 30 分钟 Independent Mode 验收；
- 复盘支持事件致谢、低价值弹幕选择、延迟、TTS 长度和后半段复读；
- 抖音只验证 bridge-only 连接、只读弹幕、安全礼物摘要与停止后无残留；
- 通过后再扩大到友好主播和小范围封闭测试。

## 下一阶段候选

以下事项不应自动开工，需在当前发布基线通过后重新排序：

1. **可信支持事件账本**：脱敏、有限容量、保留期明确的复盘数据；先做独立设计，不复用旧原始流水 schema。
2. **主播账号身份保护**：只从已验证登录 UID 派生并显式确认；不把 owner/master 关系写入 prompt。
3. **旧 `bilibili_danmaku` 退役**：按 [迁移矩阵](bilibili-danmaku-migration-matrix.md) 关闭前置项，再从最新 `main` 建独立删除 PR。
4. **通用 B 站内容/写工具**：只有维护者确认需求后才分别设计独立插件；写操作必须预览、确认、执行分离。
5. **抖音普通主播登录与分发**：单独评审账号风控、隐私、依赖、二进制更新和回滚成本。

## 延期或明确不做

- 拍案式大型直播装修、复杂背景素材与高成本视觉功能；
- 平台写操作、自动私信、自动发弹幕或自动运营；
- 贡献榜、无限期支持明细、原始弹幕历史；
- 插件内抖音 protobuf 直连、JS 签名和浏览器自动化；
- 为未来能力预建空模块、配置或一级导航；
- 为控制文件大小而继续拆没有维护收益的模块。

## 已知限制

- 真机测试暂停期间，离线测试不能证明直播话术自然度和 TTS 体验；
- 抖音接入依赖非官方本地 bridge，协议和分发风险高于 B 站；
- B 站无账号连接只是受限兜底，不等价于正式登录；
- Hosted UI 受插件中心运行时限制，兼容入口仍需单文件生成；
- 观众档案是安全派生记忆，不是完整用户行为数据库。

## 开工判断

新任务只有同时满足以下条件才进入实现：

- 属于当前工作顺序或经维护者重新拍板；
- 能说明所在产品 Slice 和负责模块；
- 不破坏 Pipeline / SafetyGuard / Dispatcher / Store 边界；
- 成本、降级、回滚、观测和测试可回答；
- 可以从最新目标分支建立独立、非堆叠 PR。
