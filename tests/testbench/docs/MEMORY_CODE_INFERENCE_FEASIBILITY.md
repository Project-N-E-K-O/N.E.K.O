# 由记忆分析问题反推主程序记忆代码缺陷 — 可行性裁决

> **受众**: 开发者 (非 tester)。本文件**不进**公共 `/docs` 白名单。
>
> **定位**: 回应用户 2026-07-15 的提问 —— "是否能通过我们记忆分析出的问题, 进一步反推 (主程序的) 记忆系统的代码中可能存在哪些问题。我对此表示一定的怀疑, 要求先调查可行性。" 本文件是**可行性裁决**, 不是实现规格。P30 本期**不落任何反推实现**。
>
> **一句话结论**: **部分可行, 且严格分层**。记忆分析发现里的**机械不变量类**问题可以作为"值得排查主程序某模块/某不变量"的**线索** (弱证据, 需人工确认); 而**内容质量类**问题 (矛盾/冗余/归因漂移/晋升语义偏离) 是**数据与模型行为的产物**, **不能可靠反推**到具体代码缺陷。把二者混为一谈会制造大量假阳性, 误导开发者去改没有 bug 的代码。

---

## 1. 为什么需要先裁决 (而不是直接做)

testbench 的记忆分析 (P27 溯源 / P28 向量空间 / P29 概况) 消费的是**磁盘数据形状 (WHAT)**: 某角色此刻的 facts/reflections/persona/对话在盘上长什么样。它**看不到运行期过程 (HOW)**: 抽取时喂了什么 prompt、embedding worker 何时跑、evidence 半衰期如何 decay、写入调用栈是谁。

"反推代码缺陷"本质是从 **WHAT 的异常** 倒推 **HOW 的 bug**。这条链条只有在"异常与代码路径存在**近乎唯一的因果对应**"时才成立。对**结构完整性不变量**, 这种对应通常存在 (违反不变量几乎一定是写入/迁移代码的锅); 对**内容质量**, 同一个"坏现象"可能来自模型能力、用户数据、配置阈值、prompt 设计等**多个非代码因素**, 反推会退化成猜测。

---

## 2. 可 / 不可反推分类表

> finding 代码沿用 P29 蓝图 §4.2。"可反推性"指"该发现能否**较可靠地**指向主程序某段代码/不变量"。

### 2.1 可反推 (机械不变量类 —— 违反几乎必是代码/迁移 bug)

| finding | 现象 | 反推指向 (主程序) | 置信 |
|---|---|---|---|
| D2 悬空引用 | `source_fact_ids` / persona `source_id` / `merged_from_ids` 指向不存在 id | 写入/删除路径未维护引用完整性: `memory/reflection/*` 晋升写入、`memory/facts.py` 删除、迁移脚本 | 高 |
| D5 断裂晋升 | persona `source==reflection` 但 `source_id` 缺失/悬空 | 晋升代码 (`reflection`→`persona`) 落 `source_id` 的分支漏写 | 高 |
| E3 多向量空间 | 同角色混入不同维度/模型的向量 | `memory/embeddings.py` 换模型未重嵌/未打 `embedding_model_id`; 迁移未清旧向量 | 高 |
| E4 向量损坏 | 向量维度/数值非法 | embedding 写入未校验; 序列化/反序列化 bug | 高 |
| E2 向量过期 | 改文后 `embedding_text_sha256` 与当前 text 不符 | 编辑记忆的写路径未触发/未标记重嵌 | 中高 |
| (事件流) 游标非单调 | event log / cursor 回退或跳变 | `memory/event_log.py` / `outbox.py` 游标推进逻辑 | 高 (需事件数据, 见 §3) |
| D1 游离事实 (部分) | fact 从未被任何反思引用 | *可能*是抽取/反思管道断链, 但**也可能只是还没跑反思** → 需结合时间/产出率, 非纯代码信号 | 中 |

### 2.2 不可靠反推 (内容质量类 —— 多因, 主因常非代码)

| finding | 现象 | 为什么不能反推代码 |
|---|---|---|
| A1/A2/A3/A4 冗余重复 | 近重复对/簇/冗余代价 | 去重是**阈值 + 模型相似度**的产物; "重复多"多半是**阈值设置**或**用户反复说同一件事**, 不是去重代码有 bug |
| B1/B2/B3/N1 矛盾 | 矛盾/待核对候选/未解决纠正 | 矛盾是**数据内容层**现象 (用户改主意/模型抽取不一致), 非代码缺陷; N1 "纠正未生效"最多提示**纠正应用逻辑**值得看, 但也可能是用户没跑纠正 |
| C1/C2/C3 归因质量 | 反思漏源/虚源/脱节 | 归因是**语义近邻 vs 声明来源**的差异, 受 embedding 质量 + 反思 prompt 影响, 不是单段代码 |
| G1 晋升语义漂移 | persona 与来源反思 cosine 低 | 晋升本就允许"抽象/改写"; 低相似**未必是 bug**, 可能是设计预期 |
| F1-F5 流水线比率 | 吸收率/晋升率/否决率/抽取产出率异常 | 比率异常是**运营信号**, 受数据量/阈值/模型影响; 只有**极端 0 值**才弱提示某道工序完全没跑 (仍需运行期证据确认) |
| H1/H2/H3 留存质量 | 重要性错配/低质事实/僵尸事实 | 数据与模型抽取质量问题, 非代码不变量 |

> **铁律**: §2.2 类发现**永不**输出"主程序代码 X 有 bug"的断言; 至多作为"记忆运营/配置调参"建议, 与代码排查严格分离。

> **P32 订正 (2026-07-15, 落地时核对真实代码后修正本表)**:
> - **"D5 断裂晋升"行删除**: 直查 persona `source_id` 与 P29 概况 **D2** 冗余, 且会重现"合并晋升人设 (`source_id` 已在合并中消解、真实来源在 `merged_from_ids`)"的**假阳性** (P39 O5 已守)。P32 直接复用概况 D2 (它用溯源图自己的边定案, LESSONS §7.25「别重算 chokepoint 已定关系」), 不再单独查 `source_id`。
> - **"游标非单调"行修正**: 主程序 `events.ndjson` 记录形状为 `{event_id: uuid4, type, ts, payload}`, **没有单调序号/游标字段** —— 真正的游标持久化在 `cursors.json` (`memory/cursors.py`)。故 P32 **不做**游标单调性检查, 改做 **EVT-DUP (重复 `event_id`)**: uuid4 不可能自然碰撞, 重复即 `memory/event_log.py` 的 append/reconcile 写入 bug。游标单调性留待"主程序补结构化钩子"的未来受控 phase。
> - **新增 ID-DUP**: 同一记忆文件内主键 id 重复 (facts/reflections 数组元素 `id`、`persona[entity].facts[].id`), 高置信机械信号, 指向对应写入模块的去重逻辑。
> - 表内"D2 悬空引用 / E3 多向量空间 / E4 向量损坏"的**指向模块方向仍成立**; P32 落地时 finding 码按 P29 实际发出的码集对齐 (概况 **D4**=悬空来源引用、**E3**=向量损坏、**E4**=多向量空间)。

## 3. 证据面缺口 (为什么现在连"可反推类"也只能给弱线索)

要把 §2.1 的"线索"升格为"可下手排查", 还缺运行期证据, 而这些 testbench **当前拿不到**:

- **无写入调用栈 / 来源**: 悬空引用出现了, 但不知道是**哪次写入**产生的 (晋升? 手动编辑? 导入迁移?)。
- **无 embedding worker 轨迹**: E2/E3/E4 能看到"结果坏了", 看不到"何时嵌入、用哪个模型、是否失败重试"。
- **无 evidence/decay 时间线**: 无法区分"陈旧僵尸事实"是 decay 逻辑 bug 还是正常老化。
- **事件流不保证在 sandbox**: `events.ndjson` 导入的真实角色未必带全 (故 P32 的 EVT-DUP 在无文件时诚实回报 `evt_status="unavailable"`, 绝不当"通过")。且它只含 `event_id`/`ts`, **无单调游标** —— 游标真身在 `cursors.json`, 单调性检查需主程序补钩子 (见 §4)。
- **无跨快照 diff**: 没有"上一版记忆 → 这一版"的变更记录, 无法定位"是哪一步引入了不变量违反"。

结论: **可反推类**目前只能产出"**这类不变量被违反了, 建议人工排查主程序 <模块> 的写入/迁移路径**"这种**导航级**线索, 不能自动定位到具体行/具体 bug。

## 4. 未来受控设计草图 (若要认真做, 需要什么)

若未来要把"反推"做成一个**可信**能力 (而非猜测生成器), 建议走**受控、需主程序配合**的路线:

1. **主程序侧补结构化不变量校验钩子**: 在 `memory/*` 的写入 chokepoint 落**结构化事件** (谁、何时、写了什么、引用了哪些 id), testbench 侧据此做**引用完整性 / 游标单调性 / 向量一致性**的**确定性校验**, 违反即精确定位到"最后一次触碰该 id 的写入操作"。
2. **只覆盖 §2.1 机械不变量, 明文排除 §2.2 内容质量**: 反推报告只列"不变量违反 + 指向的写入操作 + 建议排查的模块", **绝不**对矛盾/冗余/漂移下代码结论。
3. **弱证据分级 + 人工确认闸门**: 每条线索带"证据强度"与"需人工确认"标记, 永不自动改主程序 (延续 testbench 只读边界)。
4. **跨快照 diff 作为定位主力**: 用"引入违反的那次变更"而非"当前快照"来指认可疑代码路径。

> 该路线的前提是**主程序愿意补钩子**; 没有运行期证据, 反推的可信度上限就是 §3 描述的"导航级线索"。是否推进由记忆系统相关人员按投入产出决定。
