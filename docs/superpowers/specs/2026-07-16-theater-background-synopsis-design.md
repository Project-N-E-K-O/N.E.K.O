# 小剧场电影式背景简介设计

## 目标

把三份正式小剧场 Story 的顶层 `background` 改写为 350–500 字电影式剧情简介。简介应让玩家在选剧阶段理解人物处境、主要冲突、题材看点、风险升级与情感悬念，同时保持不剧透结局、不混入生成规则。

## 内容合同

每份正式简介按以下顺序组织为一个连续段落：

1. 交代时代、地点、双主角当前关系与初始失衡。
2. 呈现触发演出的意外事件和双方眼前目标。
3. 选取两至三个最能兑现题材承诺的升级压力，不罗列全部节点。
4. 点明双方方法或价值观之间的冲突，让猫娘保持共同主角的主动性与边界。
5. 以尚未回答的剧情问题收束，邀请玩家进入，不公布关键反转、关系结果、救援结果或正式结局。

长度以 `len(background.strip())` 计算，必须处于 350–500 个 Unicode 字符之间，标点计入长度。

## 三份剧本的宣传重点

- 《约会清单最后一项》：秘密清单、双方各自准备的邀请、纪念祭体验，以及暴雨使预演失效后必须亲自作答的关系问题。
- 《明天还会有两只杯子》：十二年后的时间观测员、恶化的个人未来、未来工具的诱惑、雪原救援与返航倒计时；核心是帮助与控制、自主与陪伴的冲突。
- 《等轨道灯再亮一次》：十四天后的回收窗口、只能维持九天的生命保障、硅尘风暴、地表远征与双人专业协作；核心是能否在资源和信任同时见底前让轨道重新看见他们。

## 字段与运行时边界

- 继续使用顶层 `background` 作为唯一公开背景真源，不增加 `synopsis`，不重新启用 `summary` 或 `scenario_card.brief`。
- `background` 只写剧本已有公开事实和悬念，不写 Story、Scene、Node、Choice、Goal、模型、规则、护栏等系统语言。
- 不修改初始 Scene、静态节点、Choice、事实、Goal、结局或 Session 合同。
- 不提升 `story_revision`：本次不改变可恢复剧情状态和路由合同。简介必须避免新增世界事实，以降低现有模型上下文变化。
- 现有介绍卡已支持独立滚动，本次不修改 HTML、CSS 或 JavaScript 布局。

## 修改范围

1. 更新 `config/theater/stories/*.json` 中三份正式 Story 的 `background`。
2. 更新 Story 单元测试和 Chromium 公开投影测试，锁定 350–500 字并证明前端未截断正文。
3. 在两份小剧场架构文档中记录正式 Story 的电影式简介长度与内容边界。
4. 更新个人 Skill `neko-theater-story-writer`：在核心工作流中加入公开简介检查，并在 `references/story-package.md` 中同步当前公开卡字段与 350–500 字合同。
5. Skill 的名称、触发描述和 `agents/openai.yaml` 不变，因为本次只补充既有 Story 编写能力。

## 测试与验收

- RED：现有正式 Story 长度测试至少对短简介失败；Skill 基线测试证明旧说明不能稳定产出 350–500 字、无剧透的公开简介。
- GREEN：三份 Story 的背景长度、公开字段白名单、Story Loader、浏览器真实展示和 JSON 校验通过。
- Skill 更新后使用隔离子代理重跑相同任务，确认其主动检查长度、采用电影简介结构，并把私有约束留在服务端字段。
- 运行 Skill `quick_validate.py`、相关 Story/前端测试、三份 Story 离线校验以及完整小剧场回归。

## 非目标

- 不重写主线剧情、对白、Choice 或结局。
- 不为自定义用户 Story 在 Loader 中增加硬长度拒绝；350–500 字是正式发布剧本与生成 Skill 的作者规范。
- 不借本次文案扩写增加新角色发言、新设定或未在现有剧本中成立的情节。
