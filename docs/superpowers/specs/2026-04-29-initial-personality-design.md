# 初始人格选择功能设计

## 1. 背景

当前 `main` 分支没有“新用户首次选择人格”的完整闭环。旧分支 `feature/character-personality` 做过一版，但把多语言文件整体重写了，产生了过大的 diff，也不利于继续维护。

这次要在 `feature/initial-personality` 上把功能重新做干净，保留用户已经确认过的产品逻辑，同时避免再次污染 locale 文件和角色卡原始设定。

## 2. 目标

本次功能要实现以下能力：

1. 新用户首次进入项目时，按既定顺序看到：
   - 存储位置选择
   - 新手教程
   - 初始人格选择
2. 人格选择弹窗支持跳过，跳过后仍可进入主界面。
3. 人格选择采用两步流程：
   - 第一步选 3 个固定人格模板之一
   - 第二步预览/确认该人格
4. 手动选择的人格只作用于“当前猫娘”。
5. 手动选择的人格优先级高于角色卡自带人格。
6. 后续可以在设置入口重新打开人格选择，并支持恢复角色卡默认人格。
7. AI 最终拿到的是“当前猫娘的有效人格”，不是前端假展示。

## 3. 非目标

本次不做这些事情：

1. 不把人格模板做成可自由增删的动态列表。
2. 不改造角色卡编辑器的大表单交互。
3. 不做完整的人格模板管理后台。
4. 不重构整个教程系统或存储位置启动流程。
5. 不重排、不整段重写 `static/locales/*.json`。

## 4. 已确认的产品决策

### 4.1 进入顺序

新用户进入首页后的顺序固定为：

`存储位置 -> 新手教程 -> 人格选择`

### 4.2 首次人格选择交互

- 首次弹出时允许跳过
- 跳过后直接进入主界面
- 后面可在设置入口补选

### 4.3 人格与猫娘的关系

- 人格覆盖是“按猫娘”生效，不是全局生效
- 切换到别的猫娘后，读取那只猫娘自己的设置
- 如果那只猫娘没有手动人格覆盖，就使用角色卡自带人格

### 4.4 人格模板来源

本次固定使用桌面文档 `猫娘设定.docx` 中的 3 个模板：

1. 经典元气猫娘
2. 傲娇毒舌小猫
3. 优雅全能管家

## 5. 备选方案对比

### 方案 A：单页选择后直接写回角色普通字段

做法：
- 首次弹窗只保留一个页面
- 用户点选后，直接把 `性格/口癖/雷点/隐藏设定/一句话台词` 等普通字段覆盖写回角色数据

优点：
- 实现快
- 复用现有角色保存接口最直接

缺点：
- 会覆盖角色卡原始人格内容
- “恢复默认人格”需要额外备份原值，否则无法恢复
- 用户以后看角色面板时，分不清哪些是原卡设定，哪些是后选人格

### 方案 B：两步选择 + 单独保存人格覆盖（本次采用）

做法：
- 首次流程分为“选择”与“预览确认”两步
- 角色卡原始人格字段保留不动
- 手动选择的人格单独保存为该猫娘的 override
- 运行时计算“有效人格”：有 override 就用 override，没有就退回角色卡原始人格

优点：
- 满足“恢复默认人格”需求最自然
- 不污染角色卡原始设定
- 和“每只猫娘各自维护人格”这个需求完全一致

缺点：
- 需要补一层“有效人格”合成逻辑
- 前后端都要新增少量专用代码

### 方案 C：全局只选一次人格

做法：
- 首次进入时只选一次，之后所有猫娘共用这一套人格

优点：
- 状态最简单

缺点：
- 不符合“人格落到当前猫娘上”的要求
- 和角色卡自带人格的默认回退机制冲突

### 结论

采用 **方案 B**。

## 6. 数据设计

### 6.1 角色卡原始人格字段

角色已有的人格相关普通字段继续保留为“原始人格来源”，例如：

- `性格原型`
- `性格`
- `口癖`
- `爱好`
- `雷点`
- `隐藏设定`
- `一句话台词`

这些字段继续来自角色卡或用户原有编辑内容，本次功能不直接覆盖它们。

### 6.2 新增：每只猫娘的人格覆盖

在每只猫娘的 `_reserved` 下新增一个人格覆盖结构，建议命名为：

`_reserved.persona_override`

结构目标：

```json
{
  "preset_id": "classic_genki",
  "selected_at": "2026-04-29T12:34:56Z",
  "source": "onboarding",
  "profile": {
    "性格原型": "经典元气猫娘",
    "性格": "......",
    "口癖": "......",
    "爱好": "......",
    "雷点": "......",
    "隐藏设定": "......",
    "一句话台词": "......",
    "用户称呼": "你/人类"
  }
}
```

说明：

- `preset_id` 用于标识选中的是哪个固定模板
- `profile` 存储当次选择的人格快照
- 保留角色卡原始字段不变
- 恢复默认人格时，直接删除 `_reserved.persona_override`

这样做的原因是：

1. 不破坏角色卡原始人格
2. 云存档可以跟着角色数据一起走
3. 以后模板文案调整时，不会反向污染老用户已经选过的人格

### 6.3 新增：首次弹窗状态

首次弹窗是否已经处理过，不放到浏览器 `localStorage` 作为唯一真相，而是单独放到本地状态文件。

建议新增状态文件：

`state/initial_personality_prompt.json`

建议字段：

```json
{
  "version": 1,
  "status": "pending",
  "handled_at": "",
  "last_presented_catgirl": ""
}
```

状态含义：

- `pending`：还没处理过首次人格流程
- `completed`：已完成首次选择
- `skipped`：首次弹窗已跳过，不再自动弹出

说明：

- 这是“首次流程是否已经走完”的全局状态
- 不是人格内容本身
- 人格内容仍然落在每只猫娘的角色配置里

## 7. 运行时有效人格规则

对任意猫娘，运行时都计算一个“有效人格”：

1. 如果 `_reserved.persona_override.profile` 存在，优先使用它
2. 否则使用该猫娘原始角色字段里的默认人格信息

后端在组装最终系统提示词时，必须基于“有效人格”而不是原始字段直出。

这意味着：

- 新用户首次选完后，当前猫娘立刻生效
- 以后切换猫娘时，每只猫娘按自己的 override/default 独立生效
- 恢复默认人格后，AI 会重新退回角色卡原始人格

## 8. 前端交互设计

### 8.1 首次弹出时机

首页加载后，只有在以下条件都满足时，才启动人格流程：

1. 存储位置启动屏障已经结束
2. 首页新手教程已经结束或被跳过
3. 已拿到当前猫娘名称
4. `initial_personality_prompt` 状态仍为 `pending`

这样可以保证用户看到的顺序就是：

`存储位置 -> 新手教程 -> 人格选择`

### 8.2 两步交互

#### 第一步：选择人格

展示 3 张人格卡片：

1. 经典元气猫娘
2. 傲娇毒舌小猫
3. 优雅全能管家

在“设置入口补选”与“当前角色重选”这两种场景下，第一步标题区下方额外展示一个注意事项提示块，用来明确告诉用户：

- 切换人格后，会清空当前角色近期对话上下文
- 这个影响只作用于当前角色
- 首次 onboarding 场景不展示这条提示，避免给新用户增加不必要负担

页面动作：

- 选择某张卡片后进入第二步
- 点击“跳过”则关闭弹窗并把状态写为 `skipped`

#### 第二步：预览确认

展示内容：

- 该人格的名称
- 一段简短人格说明
- 一句示例打招呼/对话预览
- 在“设置入口补选”与“当前角色重选”场景下，第二步确认区继续重复展示同一条注意事项，确保用户点击确认前能再次看到上下文清空影响

页面动作：

- `确认使用`：保存人格覆盖，并把首次状态写为 `completed`
- `返回重选`：回到第一步

### 8.3 设置里的补选入口

由于人格覆盖是按猫娘生效，这个入口不应做在全局 API 设置里，而应做在“当前猫娘对应的设置区域”。

建议入口放在现有角色面板的设置区，表现为：

- 一个“选择人格模板”按钮
- 一个“恢复角色默认人格”按钮
- 一个当前状态摘要，例如“当前：经典元气猫娘（覆盖）”或“当前：角色卡默认人格”

这样用户在切换猫娘后，能明确看到自己正在改的是哪一只猫娘。

## 9. 后端设计

### 9.1 人格模板定义

后端维护一份固定 3 个人格模板的 canonical 数据，用于：

- 校验 preset id
- 生成保存到角色配置中的 profile 快照
- 统一给前端提供可选模板数据

模板内容来源于桌面文档 `猫娘设定.docx`。

### 9.2 建议接口

建议补充专用接口，而不是继续把这套逻辑塞进通用角色更新接口：

1. `GET /api/characters/persona-presets`
   - 返回 3 个可选人格模板的元数据
2. `GET /api/characters/catgirl/{name}/persona-selection`
   - 返回该猫娘当前是默认人格还是 override
3. `PUT /api/characters/catgirl/{name}/persona-selection`
   - 为该猫娘设置人格 override
4. `DELETE /api/characters/catgirl/{name}/persona-selection`
   - 删除 override，恢复角色卡默认人格
5. `GET /api/characters/persona-onboarding-state`
   - 获取首次人格流程状态
6. `POST /api/characters/persona-onboarding-state`
   - 写入 `completed` 或 `skipped`

### 9.3 切换猫娘后的行为

当前项目切换猫娘已经走 fast path，并会刷新当前角色的运行态。人格功能接入后需要保证：

- 当前猫娘选中人格后，立即刷新该猫娘会话配置
- 非当前猫娘被设置人格时，也要刷新对应角色的运行态缓存
- 下次切换到该猫娘时，AI 读到的是它自己的有效人格

## 10. 提示词拼装设计

后端现有角色提示词拼装逻辑里，需要增加一层“有效人格提取”：

1. 先拿到角色原始配置
2. 判断是否存在 `_reserved.persona_override`
3. 如果存在，则用 override profile 构造人格 prompt block
4. 如果不存在，则继续使用角色卡原始人格字段

本次不改基础大 prompt，只在人格补充块这一层收口。

## 11. 多语言策略

### 11.1 UI 文案范围

这次只给 UI 层补必要的国际化文案：

- 弹窗标题、副标题、按钮文本
- 3 张人格卡片的名称/一句话简介
- 第二步预览页的标题、说明、确认/返回文案
- 设置区入口和状态摘要文案
- “切换人格会清空当前角色近期上下文”的注意事项文案

### 11.1.1 弹窗文案语气要求

这次弹窗内用户可见文案，统一调整为轻量、自然的二次元喵娘语气，但仍需保持可理解和可审查：

- 语气可以软萌、有陪伴感
- 风格要克制，不能为了卖萌牺牲信息清晰度
- 风险提示仍然必须直白表达“会清空当前角色近期上下文”这个事实
- 多语言版本都必须保留同等语义，不允许只在中文里说明风险、其他语言弱化或缺失

### 11.2 严格约束

locale 文件必须遵守以下规则：

1. 只做最小增量修改
2. 只在对应 section 下插入新 key
3. 不整体重排 JSON
4. 不改无关 key 的顺序、缩进和内容
5. 所有 locale 文件同步补同样结构的 key

这次的重点是“结构对齐、局部补 key”，不是“重写语言文件”。

### 11.3 提示词语言

第一阶段里，AI 真正使用的人格 profile 以模板原文为准；UI 展示层走 i18n。

也就是说：

- UI 名称、按钮、说明可以本地化
- 后端保存到角色 override 中的人格快照保持模板原始文案

这样能先保证行为稳定，后续如果要做“按界面语言切换人格 prompt 文案”，再单独扩展。

## 12. 测试设计

### 12.1 前端流程测试

至少覆盖：

1. 新用户首次进入时，顺序为“存储位置 -> 教程 -> 人格选择”
2. 跳过后不再自动弹出
3. 选择人格后进入第二步，确认后关闭弹窗
4. 设置入口可以重新打开人格弹窗
5. 恢复默认人格后，状态摘要正确回退
6. `settings` / `manual_reselect` 场景会在第一步和第二步展示上下文清空提示
7. `onboarding` 场景不展示该提示

### 12.2 后端逻辑测试

至少覆盖：

1. 设定 override 后，角色配置正确写入 `_reserved.persona_override`
2. 删除 override 后，角色配置回到默认人格
3. 当前猫娘与非当前猫娘都能正确刷新运行态
4. 切换猫娘后，各自读取自己的有效人格
5. 首次 onboarding 状态文件的 `pending/completed/skipped` 流转正确

### 12.3 回归关注点

重点避免回归：

1. 教程完成事件后弹窗不触发
2. 切猫娘后人格串号
3. 恢复默认后 AI 仍然沿用旧人格
4. locale diff 再次膨胀

## 13. 代码规范与实现约束

本次实现除了满足功能要求，还必须满足项目现有代码风格约束。目标不是“先写出来再说”，而是“落地后的代码看起来就像这个项目里本来就该这么写”。

### 13.1 必须参考项目既有写法

实现时必须优先参考项目现有同类代码的组织方式，包括但不限于：

- `main_routers/*` 的接口组织方式
- `utils/*` 的辅助函数分层方式
- `static/js/*` 的前端模块写法
- 现有 onboarding / tutorial / character 相关模块的事件流与状态流

禁止为了实现这个功能，临时引入一套只属于本功能的个人写法。

### 13.2 不随意发明难维护结构

要避免这几类容易被代码审查指出的问题：

1. 不引入难以理解的魔法字符串、神秘标记块、临时协议格式。
2. 不用阅读成本很高的正则或字符串拼接，把角色原始数据和运行时数据强行混在一起。
3. 不为了省几行代码，把本该分层的数据结构和运行时行为揉成一团。
4. 不新增项目里其他模块没有类似先例的重型抽象。

如果确实需要新增结构，必须同时满足：

- 能明显降低复杂度
- 命名和分层方式与项目已有模式一致

### 13.3 必须分离原始设定与运行时生效逻辑

这次实现里要明确避免把以下两类东西写成一坨：

1. 用户或角色卡原始填写的人格设定
2. 系统为了让 AI 生效而在运行时组织出来的提示信息

具体要求：

- 角色卡原始人格字段保持原样，不直接被运行时逻辑改写
- 用户手动选择的人格覆盖独立存放
- AI 最终使用的“有效人格”通过明确规则计算出来
- 不能把“原始字段”和“系统拼接块”混进同一个难以追踪的持久化字段里

### 13.4 不要把几个人格字段硬拼成隐藏 prompt block

这次实现需要明确避免旧方案里那种写法：

- 从 `性格/口癖/爱好/雷点/隐藏设定/一句话台词` 等字段中逐个取值
- 再在代码里硬编码中文标签
- 然后拼成一大段隐藏字符串
- 最后再用开始/结束标记包起来，塞回 system prompt

原因是这种写法会同时带来几个问题：

1. 把结构化人格字段重新压扁成一坨文本，后续很难维护
2. 容易把“角色原始设定”和“系统运行时注入内容”混在一起
3. 为了替换这段文本，往往又会引入正则、标记块、特殊边界，进一步增加理解成本
4. 代码审查时会显得硬编码很重，可读性差

这次新实现应遵守：

- 不新增类似 `PERSONA_PROMPT_BLOCK_START / END` 这种仅服务于字符串拼装的私有标记协议
- 不围绕“先拼一坨字符串，后续再靠正则拆/替换”来设计主流程
- 以结构化数据保存人格来源与人格覆盖
- 运行时如果确实需要生成给模型看的内容，也应基于清晰的数据边界在单点生成，而不是把它反写进持久化字段

### 13.5 文本与编码要保持可读

实现中要避免产生“看起来像编码污染”的效果，例如：

- 无必要地把正常可读文本改造成很重的程序拼装痕迹
- 在现有中文字段附近引入难懂的中间层命名
- 把本来清晰的人格字段包进不直观的运行时壳子里

原则是：

- 持久化数据尽量保持直观、可读、可追踪
- 运行时逻辑尽量显式，不靠隐晦技巧隐藏数据来源

### 13.6 locale 文件继续严格最小改动

这条单独强调一次：

- 不重写整个 locale 文件
- 不重排 key 顺序
- 不修改无关段落
- 只在已有结构附近补必要 key
- 所有语言文件保持相同结构、相同改动范围

这次实现必须避免再次出现“大段删掉重写成一坨”的 diff。

### 13.7 新代码命名尽量避免 `catgirl`

本项目历史代码里已经有很多 `catgirl` 命名与路由，这部分旧接口为了兼容可以保留；但本次新增代码应尽量避免继续扩散这个命名。

约束如下：

- 新增变量、函数、前端状态、辅助方法命名时，优先使用 `character`、`persona`、`currentCharacter`、`selectedCharacter` 等更中性的表达
- 只有在复用现有接口、现有路由、现有返回字段、现有模块命名时，才保留 `catgirl`
- 不为了“统一风格”去大面积重命名项目已有接口；只要求新代码少引入新的 `catgirl` 命名

目标是：

- 新功能代码命名更自然
- 不制造额外兼容风险
- 不做无意义的大范围术语翻修

## 14. 风险与控制

### 风险 1：再次污染 locale 文件

控制：

- 只做局部补 key
- 修改前先定位插入 section
- 修改后跑 i18n 同步检查

### 风险 2：覆盖掉角色卡原始人格

控制：

- override 单独存到 `_reserved`
- 原始字段不直接改写

### 风险 3：前端显示和后端生效不一致

控制：

- 选择后必须走专用后端持久化接口
- 运行态刷新必须在保存成功后执行
- 设置页摘要读取后端实际状态
