# NEKO Live 稳定 ID 迁移与安全升级设计

## 背景

NEKO Live 当前使用 `neko_roast` 作为内部稳定 ID、插件目录、Python 包路径和配置根节。安装器在目标目录已经存在时默认把新包安装为 `neko_roast_1`，但插件入口仍指向 `plugin.plugins.neko_roast`，因此注册阶段会以 `PluginEntryDirectoryMismatch` 拒绝启动。

新版尚未正式发布，现有用户很少。本次选择在发布前完成一次干净的稳定 ID 迁移，同时修复所有插件都可能遇到的重复安装问题。

## 目标

1. 将 NEKO Live 的内部稳定 ID 从 `neko_roast` 统一迁移为 `neko_live`。
2. 使插件目录、Python 包路径、配置根节和 API 身份保持一致。
3. 显示名称继续使用 `NEKO Live`，分发包文件名使用 `neko-live-<version>.neko-plugin`。
4. 同一插件的本地重复导入不再生成无法加载的 `_1` 目录。
5. 重复导入采用显式确认、停止旧实例、备份、原位安装、验证、恢复运行和失败回滚的安全升级流程。
6. 不引入旧 ID 的长期兼容层，不允许新旧插件同时注册。

## 非目标

- 不改变直播功能、调度器行为、UI 信息架构或平台协议。
- 不提供 `neko_roast` 的运行时别名或自动配置、数据迁移。
- 不自动删除未知的用户目录或配置。
- 不使用 rebase、amend、squash 或 force-push。

## 稳定身份

迁移后的身份统一如下：

| 层级 | 值 |
| --- | --- |
| 插件稳定 ID | `neko_live` |
| 插件目录 | `plugin/plugins/neko_live` |
| Python 入口 | `plugin.plugins.neko_live:NekoLivePlugin` |
| 入口类 | `NekoLivePlugin` |
| 配置根节 | `[neko_live]` |
| 显示名称 | `NEKO Live` |
| 分发包名 | `neko-live-<version>.neko-plugin` |

内部 ID 使用下划线，确保它同时是合法的 Python 包名并与目录完全一致。面向用户的名称和包文件名使用空格或连字符。

## 插件迁移

### 源码与清单

- 将插件目录从 `neko_roast` 移至 `neko_live`。
- 更新 `plugin.toml` 中的插件 ID、入口和配置根节。
- 更新入口类名、插件 `name`、内部导入、动态导入错误判断、测试导入和工具脚本常量。
- 更新 Hosted UI、压测工具、文档、分发边界检查和本地制品路径中的稳定身份。
- 仅保留描述业务概念时确有意义的 `roast` 名称，例如“锐评强度”；不保留作为插件身份或包路径的 `neko_roast`。

### 旧安装处理

本次采用发布前干净断代：

- 不注册 `neko_roast` 别名。
- 不同时加载 `neko_roast` 与 `neko_live`。
- 测试用户升级时先停止并卸载旧插件，再安装新包。
- 若检测到旧目录，UI 给出明确卸载提示并中止新包安装，不把它静默改成 `_1`，也不自动删除。
- 旧配置和旧数据目录保持原样，测试用户如需保留内容可在卸载前手动备份；正式版本只认 `neko_live`。

## 本地导入的安全升级

### 冲突分类

导入前读取包内 manifest，并以稳定插件 ID 判断：

1. **无同 ID 插件**：执行普通安装。
2. **存在同 ID、目录和入口一致**：显示升级确认，执行原位安全升级。
3. **存在同目录但 ID 不同**：视为目录冲突，拒绝覆盖并提示用户处理。
4. **同 ID 存在多个实例或目录不一致**：视为异常安装状态，拒绝自动处理并列出冲突路径。

不得再通过追加 `_1`、`_2` 来解决可执行插件目录冲突。

### 升级事务

用户确认后按以下顺序执行：

1. 重新确认目标插件和导入包身份未变化。
2. 记录旧插件是否正在运行。
3. 正常停止旧插件；停止失败则中止升级。
4. 将旧目录原子重命名到同级唯一备份目录。
5. 把新包安装到原目录名，冲突策略固定为 `fail`。
6. 运行 manifest、入口目录和基础加载验证。
7. 若旧插件原本运行，则启动新插件。
8. 成功后异步清理备份；清理失败只告警，不回滚已成功升级。

任一步骤失败时：

- 删除未完成的新目录；
- 把备份恢复到原目录；
- 若旧插件原本运行，则尝试恢复运行；
- 返回可读错误，包含失败阶段，但不泄露敏感路径或配置内容。

### UI 行为

- 本地包导入不再默认选择“重命名副本”。
- 检测到同 ID 时，按钮和对话框明确使用“升级插件”。
- 确认框展示插件名称、当前版本、目标版本和是否需要短暂停止。
- 升级进行中禁止重复提交。
- 成功后刷新插件详情；失败后显示已回滚或需要人工恢复的准确状态。
- 仍允许用户取消；不提供会生成 `_1` 目录的选项。

## 数据与配置

- 配置读取只使用 `[neko_live]`。
- 插件存储、观众档案、审计数据和本地制品路径改用 `neko_live` 命名。
- 若数据目录由用户显式配置，保持该路径不变。
- 旧默认目录不自动搬迁或删除，避免把未发布测试数据误当成正式数据。
- 日志中不得记录原始对话、令牌、Cookie 或观众敏感信息。

## 兼容性与错误处理

- `neko_live` 是正式发布后的唯一稳定 ID，后续不得再次因品牌调整改变。
- 插件 ID 与 Python 包目录保持一一对应。
- 网页开发环境和 Electron 插件中心均复用同一导入/升级状态模型。
- Windows 路径占用、杀毒软件延迟和跨设备复制导致的失败必须进入回滚路径。
- 备份名称不参与插件发现，避免备份目录被重复注册。
- 安装器底层保留 `fail` 语义；旧 `rename` 行为若仍供非执行制品使用，不得用于插件安装入口。

## 验证

### NEKO Live

- manifest ID、目录和 Python 入口一致。
- 仓库中不存在作为稳定身份或导入路径的 `neko_roast` 残留。
- 配置默认值、Hosted UI、开发者模式、直播平台连接和调度器回归通过。
- 全部 NEKO Live 测试、Ruff、插件 CLI check、Hosted UI 测试和 locale parity 通过。
- 构建的 `.neko-plugin` 通过 inspect、verify，并可在干净环境启动。

### 安装器

- 首次安装成功。
- 同 ID 升级保持原目录名且不生成 `_1`。
- 运行中插件升级会停止并恢复。
- 安装、验证、启动三个阶段分别注入失败时都能恢复旧版本。
- 同目录不同 ID、同 ID 多实例和未知目录状态会拒绝覆盖。
- 用户取消不改变文件、配置或运行状态。
- 网页与 Electron 中的确认、忙碌、成功、失败和回滚状态一致。

### 发布检查

- 8 个 locale 的 key 与占位符保持一致。
- 构建产物名为 `neko-live-<version>.neko-plugin`。
- 在旧测试安装上验证迁移提示，在干净安装上验证首次安装，在同 ID 安装上验证安全升级。

## 交付顺序

1. 增加安装器安全升级的失败测试和事务测试。
2. 实现后端升级事务和冲突分类。
3. 更新插件中心导入 UI。
4. 迁移 NEKO Live 稳定身份及其测试、配置和文档。
5. 运行完整回归并构建、检查分发包。

先修安装器再迁移插件，可确保新 ID 的第二次导入从一开始就走正确的升级路径。
