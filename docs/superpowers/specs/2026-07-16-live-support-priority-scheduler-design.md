# NEKO Live 可信支持事件与实时回应调度设计

## 状态

本设计已于 2026-07-16 获得产品方向确认。实现必须增量保留当前工作区已有的会话隔离、防伪、优先级调度、文档与测试改动，不得覆盖或回退这些修改。

2026-07-16 真机测试收尾范围只交付可信投影、去重/连击状态机、有界优先调度、会话任务回收与内存聚合状态；不新增持久礼物账本或诊断事件日志。本文后续“本场可信账本”、拥塞状态机、公平摘要和熔断章节仍是长期设计目标，不得解读为本次提交已经实现。

2026-07-17 当前收尾范围已在 `plugin/plugins/neko_live/modules/live_support_events/scheduler.py` 落地，并由调度器聚焦测试和完整插件门禁覆盖；本文后续长期目标仍保持未实现状态，进入后续独立评审，不得从当前完成标记推导为已交付。

## 目标

NEKO Live 必须同时满足两个不同的可靠性目标：

1. **礼物事实准确**：只有平台结构化事件能够形成 Gift、Super Chat 或 Guard 事实；重复、乱序、连击更新和旧会话回调不得造成虚假或重复统计。
2. **口头回应实时**：LLM/TTS 来不及逐条处理时，可以合并、摘要、降级或放弃过期回应；任何负载都不得让队列、内存或任务数量无界增长。

核心原则是：**可信账本尽量准确，口头回应允许有损；宁可合并过期回应，也不能拖死直播。**

## 非目标

- 不引入 RabbitMQ、Kafka、Redis 或新的生产依赖。
- 不把所有支持事件持久化后补播。
- 不承诺在极端洪峰中逐条口头感谢。
- 不跨平台比较礼物金额，不新增主播维护的价格表。
- 不允许支持事件绕过 Pipeline、Safety Guard、dry-run 或 Dispatcher。
- 不修改宿主 N.E.K.O 的全局输出队列。
- 不中断已经开始执行的 pipeline 请求或 TTS 语句。
- 本阶段不扩大抖音生产接入，只维持 provider-neutral 契约和测试边界。

## 总体架构

```text
平台长连/桥接事件
    ↓
Provider 规范化与可信来源分类
    ↓
会话隔离、组合更新与时间窗口去重
    ↓
本场可信支持事件账本/计数
    ↓
有硬上限的回应规划器
    ↓
现有 handle_live_payload() → Pipeline → Safety Guard → Dispatcher
    ↓
串行 LLM/TTS 输出
```

接收、可信事实、回应规划和输出必须是清晰边界。平台接收回调只做有界、快速、非阻塞的规范化与提交，不等待 LLM 或 TTS。

## 可信来源分类

所有直播输入归入以下三类之一：

- `verified_support`：平台结构化 Gift、Super Chat、Guard 等支持事件，可以更新本场账本并进入回应规划器。
- `platform_notice`：平台系统通知，只能更新运行状态或审计，不能形成礼物事实。
- `unverified_chat`：普通弹幕和文本输入。无论文本写得多像礼物，都不能升级为支持事件。

普通弹幕中的“人气票×99999999”“送出大航海×999”或 `gift x 99999` 只能作为未验证声明。文本防伪规则可以阻止这些内容误导 LLM，但它不是礼物真实性的判断来源。

真实 provider 路径缺少有效的 `support_verified` 和受限 `support_evidence` 时必须 fail closed。测试和本地模拟只能使用显式 `manual_live_simulation` 来源。

## Provider-neutral 安全投影

支持事件在现有公共字段上增加以下受限字段：

- `support_verified: bool`
- `support_evidence: str`
- `provider_event_id: str`
- `provider_event_type: str`
- `provider_timestamp_ms: int`
- `combo_id: str`
- `combo_count: int`
- `combo_end: bool`
- `coin_type: str`

安全投影不得保留原始事件对象、cookie、token、签名、原始聊天文本或可持久关联的额外身份字段。平台时间戳只能辅助排序和诊断；在没有验证平台时钟语义前，不得仅因时钟偏差拒绝真实礼物。

## 会话隔离

每次开始监听、重连、切换直播间和停止直播都使用新的 `session_generation`。所有异步任务、连击计时器和输出提交必须携带所属 generation。

- 旧 generation 的迟到回调不得更新账本、队列或输出。
- 停止、切换房间或 teardown 时取消旧 worker、连击计时器和待输出回应。
- 去重状态按 provider、房间和会话隔离。
- reset/close 后不得残留后台任务或把旧房间事件带入新房间。

## 去重与连击更新

### 事件状态

去重项不是简单的 ID 集合，而是受限状态：

```text
(provider, room_ref, provider_event_id)
    → fingerprint, combo_count, value, first_seen_monotonic, last_seen_monotonic
```

同一 ID 按以下规则处理：

- 指纹和累计值完全相同：重复投递，忽略。
- 连击数量或累计值单调增加：合法状态更新，只应用新增差额，不重复创建回应。
- 计数倒退：迟到或乱序更新，忽略，不回退累计值。
- UID、礼物身份或事件类型改变：协议异常，隔离并记录固定原因码。

平台没有稳定事件 ID 时，才使用受限回退指纹。回退指纹包含 provider、房间、事件类型、脱敏 UID、礼物身份、累计数量/金额和短时间桶，只用于短时重复包保护，不作为长期身份。

### 时间窗口和容量

- 去重窗口默认 10 分钟，基于本地单调到达时间。
- 去重缓存必须同时具有 TTL 和容量硬上限，建议默认最多 65,536 项。
- 容量淘汰、TTL 淘汰和异常更新必须有聚合计数，但不得记录原始 ID。
- 会话结束时立即清空，不持久化到下一场直播。

### 连击收敛

- 有 `combo_id` 的更新写入同一 accumulator，不重复入回应队列。
- `combo_end=true` 时立即完成收敛。
- 缺少结束标志时，在最后一次增长后 1 秒无更新即收敛。
- 累计数量和价值只能单调增加。
- 相同 ID 的状态更新与不同包 ID 的同一 combo 都必须覆盖。
- 收敛后只生成一个规划候选，包含最终可信数量和价值。

## 本场可信账本

账本与回应队列分离。账本至少维护聚合结果：

- 已验证 Gift、Super Chat、Guard 的数量和受限价值统计；
- 每个脱敏观众关联 ID 的本场累计支持；
- 连击最终累计；
- 重复、乱序、异常、合并、过期和丢弃数量。

账本不依赖回应是否成功。回应被合并、过期或 TTS 失败时，真实礼物仍计入本场统计。账本不保存原始弹幕、原始 UID、凭证或完整 provider 包。

## 优先级

只保留四档，避免复杂且昂贵的多级队列：

1. `P0 milestone`：Super Chat、Guard 和显式里程碑。
2. `P1 high`：B 站 `gold` 且 `gift_total_coin >= 10000`。
3. `P2 medium`：B 站 `gold` 且 `1000 <= gift_total_coin < 10000`。
4. `P3 light`：其他礼物，包括 `silver`、免费礼物和缺失或较低金额。

同级事件默认保持稳定到达顺序。优先级只能调整尚未开始的回应，不能取消已经运行的 pipeline/TTS。

## 硬上限与准入

所有容器必须有真正的硬上限；**任何优先级都不得临时突破上限**。

建议默认值：

- Pipeline 待输出回应：沿用现有 `queue_limit`，当前默认 5 条。
- 已验证支持调度：`max(8, queue_limit)`，当前默认 8 条，且不突破配置允许的最大值 100。
- 普通弹幕候选摘要：最多 64 条，同时单独维护总候选数和最佳候选。
- 去重状态：10 分钟且最多 65,536 项。

队列满时按以下顺序处理：

1. 尝试合并同观众、同礼物、同短时间窗事件。
2. P0/P1 可以淘汰最旧的 P3，再淘汰最旧的 P2。
3. P2 只能淘汰或合并 P3。
4. P3 优先聚合进已有轻量摘要，否则拒绝最新事件。
5. 若队列中没有更低优先级受害者，新到事件不得突破硬上限；记录拒绝或汇总统计。

准入和淘汰必须在与上限相关的有界时间内完成。默认上限较小时允许扫描队列，但算法复杂度不得随被拒绝洪峰持续增长。

## 实时拥塞降级

拥塞依据最老待处理回应的等待时间（sojourn time）和队列占用共同判断，而不是只看长度。初始内部阈值：

- `NORMAL`：最老等待小于 2 秒，正常逐条规划。
- `BUSY`：2–5 秒，合并同观众和同礼物。
- `CONGESTED`：5–10 秒，P2/P3 与普通互动改为聚合摘要。
- `EMERGENCY`：超过 10 秒或达到硬上限，只保留能及时处理的 P0/P1 和一个聚合摘要；其他回应过期或拒绝。

状态回落使用滞回，避免在阈值附近频繁切换。阈值是 NEKO 的内部安全默认值，不增加小白必须理解的设置项；开发者页面可以只读显示状态和指标。

过期处理：

- 过期回应不得在高峰结束后补播。
- 真实礼物已经进入账本，因此过期只影响口头回应。
- P0/P1 可以使用较宽时间预算，但仍不得无限等待。
- 过期事件只记录固定原因码和聚合数量。

## 公平性与回应规划

严格优先级容易让普通互动永久饥饿，因此回应规划器同时应用：

- 同观众连续支持优先合并，避免单个观众占满队列。
- P3 跨观众聚合为安全感谢摘要，不虚构精确金额。
- 普通弹幕只保存最佳候选、总数和有界审计样本，不保存全部候选。
- 持续高负载时保留一个聚合互动槽位，使猫猫仍可间歇回应整体观众；P0 紧急里程碑仍可先行。
- 不承诺每位观众都获得独立回应，公平目标是避免单一来源无限垄断有限输出时间。

## 输出隔离与熔断

每个直播会话只有一个支持回应 worker。worker 每次只将一个安全 payload 交给 `ctx.handle_live_payload()`，等待返回后再处理下一项。

- 接收路径不等待 LLM/TTS。
- 支持事件 worker 不直接调用 Dispatcher。
- 单次 active dispatch 超过 30 秒属于软卡顿：不取消已经开始的 pipeline/TTS，但回应规划立即进入 `EMERGENCY`，停止积压新回应。
- 60 秒内连续 3 次 dispatch 异常时打开输出熔断 30 秒，停止创建必然过期的输出任务。
- 熔断期间继续接收、验证和统计真实礼物。
- 熔断窗口结束后只允许一个新鲜的 P0/P1 事件半开探测；成功则关闭熔断，失败则重新打开 30 秒。
- 恢复后只处理新鲜事件，不重放过期回应。
- 重试必须有界并带退避；不得形成重试风暴。
- LLM、TTS、平台接收和审计资源相互隔离，一个组件故障不得阻塞其他组件 teardown。

这些阈值是内部安全默认值，不新增用户可见设置。当前实现不得通过强制取消一个已经运行的宿主请求来模拟超时；soft deadline 只控制后续准入和状态。

## 防伪文本

文本规则只处理整句覆盖层式声明，例如：

```text
gift x 99999999
人气票×99999999
SC x 99999
```

规则必须锚定整句或明确的直接自称投喂结构，避免误伤“有人说 gift x 999 是什么意思”之类正常讨论。命中后标记为未验证支持声明并从礼物语义中隔离；它仍不能产生任何账本条目。

## 可观测性

状态只暴露聚合数字和安全枚举：

- 当前拥塞状态、最老等待时间桶和队列占用；
- 各优先级 pending 数量；
- 活跃 combo 数量；
- 去重、合法更新、乱序、异常、合并、淘汰、拒绝、过期和输出失败计数；
- LLM/TTS 熔断状态；
- 最近一次安全事件类型和调度优先级。

固定原因码至少包括：

- `support.duplicate_event`
- `support.combo_updated`
- `support.combo_regressed`
- `support.event_conflict`
- `support.combo_finalized`
- `support.queued`
- `support.aggregated`
- `support.lower_priority_evicted`
- `support.capacity_rejected`
- `support.response_expired`
- `support.dispatch_failed`
- `support.output_circuit_open`
- `support.stale_generation`

不得记录昵称、原始 UID、原始消息、原始礼物包、凭证或未经脱敏的异常对象。

## 测试策略

### 正确性

- 结构化支持事件形成可信事实，普通弹幕不能伪造。
- 同 ID 同快照只处理一次。
- 同 ID 递增连击只应用新增差额并最终生成一次回应。
- 递减或身份冲突更新 fail closed。
- 不同真实事件 ID 即使内容相同也不能误杀。
- session generation 变化后旧回调、旧 combo 和旧 worker 不得输出。

### 背压和性能

- 十万条纯 P0/P1 洪峰中，队列峰值始终不超过配置硬上限。
- 洪峰拒绝路径不得出现随累计输入增长的 O(n²) 行为。
- 两万条普通候选后摘要容器仍不超过 64。
- 去重缓存同时满足 TTL 和容量上限。
- 压测结束、reset 和 close 后无残留任务，内存可回落。

### 拥塞和公平性

- 四种拥塞状态按等待时间进入并带滞回退出。
- 同观众连续礼物合并，P3 能汇总，P0/P1 不被低优先级阻塞。
- 持续高优先级输入不会无限保存普通事件；普通观众以聚合摘要获得有限回应机会。
- 过期回应不补播，但账本统计保持准确。

### 故障

- dispatch 有界重试后继续处理后续事件。
- LLM/TTS 连续失败开启熔断，不阻塞接收和 teardown。
- 恢复后只处理新鲜事件。
- 取消、切房和重连与 worker 完成之间的竞态不会产生旧会话输出。

### 防伪

- 覆盖层式 `gift x 99999999` 被隔离。
- 正常讨论、引用和解释句不被误判为真实支持事件。
- 所有防伪命中都不产生账本或感谢 payload。

### 回归

- 运行插件完整 pytest。
- 运行 Ruff、插件 CLI 静态检查和 `git diff --check`。
- 若新增或修改用户可见 i18n key，同步并校验全部 8 个 locale。
- 检查隐私字段、system prompt 水印和现有 Pipeline/Safety Guard/dry-run/Dispatcher 链路。

## 验收标准

1. 十万条高优先级支持事件无法突破硬上限、拖成 O(n²) 或遗留任务。
2. 相同 provider ID 的重复包、合法连击更新和异常冲突被准确区分。
3. 真实礼物账本不因口头回应合并、过期或失败而丢失事实。
4. 普通弹幕文本无法伪造任何礼物事实或感谢。
5. LLM/TTS 故障不会阻塞平台接收、会话停止或切换直播间。
6. 高峰期间回应会自动合并和降级，恢复后不会补播陈旧内容。
7. 所有队列、候选、去重状态和后台任务都有明确上限与生命周期。

## 参考设计依据

- Twitch EventSub：至少一次投递、事件 ID 去重与重放保护。
- YouTube LiveChatMessages：结构化支持事件类型以及同一 ID 更新礼物连击数量。
- GStreamer Queue、Google Pub/Sub 与 Reactive Streams：有界流控和明确溢出策略。
- RFC 8289 CoDel：使用队列停留时间而非仅长度判断用户可见拥塞。
- RabbitMQ Priority Queues：限制优先级数量并防止严格优先级饥饿。
- Streamer.bot Action Queues：提醒和媒体动作串行执行，避免输出重叠。
- Bulkhead 与 Circuit Breaker：隔离 LLM/TTS 故障并避免重试风暴。
