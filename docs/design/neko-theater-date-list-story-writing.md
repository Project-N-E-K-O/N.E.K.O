# 《约会清单最后一项》剧本创作与转场修订

## 文档目的

本文记录 `date_list_last_item_story.json` 的剧本诊断、修改依据和验证结果。它只服务这一个故事，不替代 Theater Runtime、Story Package 协议或通用创作 Skill。

本轮保持以下边界不变：

- 仍为玩家与猫娘的双人甜蜜约会故事。
- 保留规整修正版的十五步主线；v2.4 在暴雨段增加作者隐藏语义边、支线中心、延迟汇流和替代结局，当前 Story Package 共十九个节点、十九条边和两个结局。
- 保留故事 ID、结局 ID、开场 Node ID 与两条开场 Choice ID，避免当前开场存档的按钮立即失效；其余节点和 Choice 以新稿为准。
- 推荐选项仍使用作者权威 ID、目标和 callback；隐藏语义边不显示按钮，只能由服务端在当前作者候选中解析。
- v2.4 同步修改 Theater Runtime 的自由输入路由，但不恢复旧动态图，也不允许模型创建节点或修改事实。
- 本轮不修改或同步本地 `neko-theater-story-writer` Skill；是否同步由用户在剧本体验稳定后另行决定。

## 当前规整修正版

本节是当前剧本真值；下文记录的是此前多轮诊断和修订历史，与本节冲突时以本节和当前 JSON 为准。

主线仍由十五次选择组成：接住袋口挂坠、接受纪念祭邀请、各自佩戴腕带、在人潮中调整同行位置、进入杂货铺、选择礼物、交出礼物、分食甜点、完成四格合照、写下长期观察、回应暴露的卡片、面对被雨毁掉的预写答案、接受路线外的屋顶邀请、脱稿确认关系、共同写下明天。玩家始终可以按这十五步进入原结局。

v2.4 另有一条不会显示成按钮的暴雨语义边：在 `node_honest_observation`，玩家若暂不回答口袋纸张的问题，连续坚持询问猫娘如何看待自己，前两次仍留在当前场景自然交谈；中途换话题会清零；第三次连续表达才进入 `node_deep_impression_conversation`。这个分支不是模型临时创作，而是作者静态图的一部分。

本版采用以下因果链：

1. 公开背景停在门廊现场：牛皮纸袋、两张普通入场券和正在滑落的手工挂坠都已可见，第一组选项只处理眼前挂坠。
2. 猫娘主动邀请同行；抵达入口后，纪念腕带、人潮、护城河木栈道、天气告示和风铃店铺均先出现，再提供相应选择。
3. 杂货铺先公开左侧铃铛、右侧书签与文具，礼物选择才出现；交付礼物的对白保持通用，既能承接星铃，也能承接木笔，不替未选分支说话。
4. 双色大福、四格照相亭、赠品纪念册、外面渐大的雨和口袋里的受潮纸张都在相关选项出现前公开。
5. 卡片落入积水后才变得无法辨认；猫娘在确认旧答案失效后主动邀请玩家走上导览图之外的屋顶，并直接询问关系。
6. 主线的两条告白选择都由玩家角色亲口提出建立恋爱关系；猫娘明确接受后，最后一组选项才共同写下明天。
7. 隐藏支线中，猫娘只依据已经发生的让路、分食和观察记录回答“如何看待玩家”，不猜玩家内心；之后玩家可以继续谈完并在屋顶汇回告白节点，也可以暂不为关系命名，只约明天不用清单再见。

当前关系边界是：确认关系前不牵手、不拥抱、不挽手臂、不勾衣物，也不使用“情侣练习”等预支关系的称呼。人潮段只通过站位、步速和路线表达照顾；共同书写发生在双方已经明确确认恋爱关系之后。

当前选项自检要求每条推荐边都满足“上一节点的公开 Scene、对白或已发生回调，足以解释下一组选项中的人物、地点、物品和动作”。测试会遍历所有推荐边检查该条件；隐藏边另行检查作者意图元数据、连续计数和前端不可见性。所有推荐项与回调继续禁止作者替玩家写入私心、冲动、自尊或其他未选择的心理活动。

## v2.4 隐藏语义支线修订

本轮修改针对“玩家不点击推荐按钮、也不复述近似按钮文案就永远无法推进”的结构问题。剧本和 Runtime 的职责拆分如下：

- 剧本在普通 `edges` 中声明 `visibility=latent`、稳定 `transition_id`、局部 `goal_id`、稳定 `intent_id`、语义说明、表达示例和两次拉回阈值。
- 大模型只判断本轮是否命中当前作者 `intent_id`；它不能选择目标节点、提交事实、写结局或生成自由文本意图档案。
- 服务端第一次和第二次命中只保存结构化连续次数并展示当轮自然回应；第三次才提交作者目标节点，并立即用目标节点重新演绎本轮输入。
- 普通聊天、不同意图、越界请求、未知 ID 或模型失败会清除当前连续计数，不会把分散输入累计成支线。
- 隐藏 ID、拉回次数和主支线规则不进入猫娘对白，也不经过公开投影。猫娘只能表现出迟疑、回答、保留或转移注意，不能向玩家解释系统正在“拉回”。

支线中心的两条可见选择都在猫娘上一句中得到铺垫：她先介绍侧窗外已经看见的屋顶木梯，也自然提出如果今晚不急着下结论，可以只约明天再见。选择去屋顶后，`node_side_rooftop_bridge` 必须明确承接刚才谈过彼此印象的事实，再询问关系；选择明天再见后，纪念册只写时间和旧街入口，关系名称保持空白。替代结局确认双方彼此重要，但不替玩家或猫娘提前确认恋爱关系，因此不是偏离推荐项后的惩罚结局。

当前自动化已验证：隐藏边不会出现在推荐按钮；同一意图的前两次连续输入不推进，普通岔题会清零，第三次连续输入当轮进入支线；支线可以保留 `mutual_impressions` 事实汇回告白节点，也可以结束于 `ending_tomorrow_without_label`。本轮仍不把这些尚在体验验证中的规则同步回本地编剧 Skill。

## 运行时事实

推荐选项并不是当前 Node 自己的 `suggestions`。实际显示顺序是：

1. 播放当前 Node 的回调、场景和对白。
2. 读取当前 Node 出边所指向目标 Node 的 `suggestions` 并展示。
3. 玩家选择后进入目标 Node。
4. 播放该 Choice 的回调，再播放目标 Node 的对白。

因此，对于边 `A → B`：B 的选项会在 B 的对白之前出现。B 选项用到的人物、地点、物品、规则、问题和动作前提，必须在 A 的可见演出结束前已经成立。B 的对白只能回应刚刚发生的选择，不能事后补问该选择已经回答的问题。

## 玩家角色视角规则

推荐选项不是作者给剧情下结论，也不是让玩家站在上帝视角选择正确剧本方向。它代表玩家角色在当前这一刻，根据自己已经经历和观察到的事情，可以自然说出的话或做出的动作。

推荐选项只允许使用以下信息：

- 当前及此前公开 Scene 中可观察到的环境、人物动作和道具。
- 玩家已经听到的猫娘对白，以及已经发生的 Choice 回调。
- 剧情已经公开建立的共同经历和玩家角色身份。玩家可以知道自己写过约会清单，但不能因此知道作者设计的主题、转折或结局。
- 当前对白明确提出的问题、邀请、规则和可执行目标。

推荐选项不得使用以下信息：

- Node 标题、`summary`、`runtime_generation_guide`、隐藏事实、未揭示线索、后续 Node 和结局条件。
- 作者对主题的概括，例如直接替角色说出“今天要留一个计划外选择”，除非角色已经用自然对白具体提出了这件事。
- 玩家角色尚未看见的物品、尚未抵达的地点、尚未发生的动作和对方尚未表达的意愿。
- 只有编剧才知道的最佳答案、伏笔用途、剧情任务编号或未来关系结果。

检查时要把作者信息暂时遮住，只保留实际呈现给玩家角色的 Scene、narration 和 dialogue，再问：“处在这个情境中的人，真的会想到这句话吗？”如果选项只有在读过剧本结构后才合理，就必须重写前置演出或选项。

例如，原推荐台词：

> “好。今天至少有一件事，我不提前看答案。”

虽然呼应了作者设定的主题，但玩家角色并没有一个需要“提前查看”的外部答案，“至少有一件事”也没有具体所指，因此像编剧替玩家总结主题。更符合角色视角的表达应建立在猫娘当下提出的具体请求上，例如：

> “好。如果中途想改路线，我们就一起商量。”

这句话只回应角色刚刚听见的请求，不预知后续转折，也不替整场故事宣布主题答案。

## 原稿问题

完整推荐路径自检发现，十八次转场中多数存在“答案先于问题”：

- 挂坠尚未滑落，选项已经要求接住；猫娘尚未说要重缝，选项已经要求阻止重缝。
- 双人券、腕带、人潮通道、礼物摊、甜品和照相亭都曾在选项中先于正文出现。
- “最后一口”“重新坐近”“另一条腕带”等指代缺少可见前件。
- 第六项挑战没有具体任务，直接跳到领取同步印章。
- 十分钟改道、掉落的卡片、屋顶目的地都由选项先行宣布。
- 猫娘尚未接受真正约会，玩家已经看到“一起写明天”的共同承诺。

这些问题不是单句不自然，而是作者数据结构与运行时显示顺序错位。

## 最新真实演绎记录诊断

本轮依据最新会话 `/Users/mac/Library/Application Support/N.E.K.O/theater/sessions/theater_4548a8dd-850d-48f5-91a7-6347fd5c2b1a.json` 定责，不以静态剧本推测实际表现。该会话的当前猫娘是霜瞳。

玩家接住星星后，霜瞳实际说的是：

> “哼……既然你非要留着那种歪歪扭扭的针脚，本小姐就勉为其难承认你的眼光还算凑合喵。”

随后界面立即显示：

> “好。中途想停还是想改，我们都一起商量。”

这条选项依赖猫娘先提出“中途可以停或改，并由两人商量”，但人格化对白把这项邀请完全省略，因此真实体验中确实是无前件选项。

定责结论：

- 主要是代码契约问题。剧情推进提示词没有向模型提供 `scripted_dialogue` 的完整语义，也没有提供对白后即将出现的推荐选项；模型只得到“猫娘得到肯定后开心”的短意图，所以能演出霜瞳人格，却不知道必须保留路线约定。
- 开场也是代码问题。原链路不调用演绎模型，直接把 `opening_dialogue` 当作猫娘对白返回，因此任何活跃猫娘都会读同一份作者台词。
- 固定台词偶发出现属于模型回退链路。旧代码没有记录回退原因，只能确认使用了作者文本，无法从日志区分模型超时、无效 JSON 或防复述拦截。
- 剧本也有次要缺口。`node_protect_charm` 的 `catgirl_raw_intent` 只写“得到肯定后开心”，没有记录必须提出共同商量路线；当时的 `node_theme_question` 还错误地把入场券写成了七项规则的载体。后者已在后续“唯一七项来源修正”中废止。

对应修正：

- 正式开场也交给当前活跃猫娘演绎；作者开场对白只作为必须保留含义的语义底稿。
- 剧情推进同时提供作者对白语义和下一轮真实推荐选项，明确要求人格化转述不得删掉选项所需的问题、邀请、道具和规则。
- Choice callback 与开场 Scene 旁白仍保持作者权威，模型只转述猫娘对白，不能改写已经发生的动作。
- 为模型配置缺失、调用失败、无效输出、复述玩家及纠错失败增加不含玩家隐私的诊断日志。
- 对“照读作者台词、缺少人格自称、把玩家选项当猫娘台词、把共同决定改成单方命令”等可机械判定的问题，只进行一次定向纠错重试；仍不合格时使用作者台词保住剧情交接并记录原因，不做开放式文风循环。
- 补全开场、保留挂坠和路线约定节点的猫娘演绎意图。
- 将原本像复述规则的推荐台词收口为角色直接回应：“好。有想改的地方就告诉我，我们一起决定。”

该阶段曾使用当前真实活跃猫娘“霜瞳”连续新建两场临时 Session。两次演绎都先由霜瞳以“本小姐”的自称提出“中途想停或换地方就当场商量”，然后才显示“一起决定”的推荐选项，证明人格转述与选项前件修复有效。但当时后续仍把双人券引向“阅读七项规则”；最新演绎已经证明这套剧情前提错误，因此该验证不再代表当前剧本完整通过。

## 推荐项停滞与惊喜泄露修正

最新会话 `/Users/mac/Library/Application Support/N.E.K.O/theater/sessions/theater_47c4220a-a1ec-4c83-8c54-ee075252c242.json` 复现了连续自由互动下的真实故障。当前猫娘希尔在玩家依次说出“先去哪里呢”“触发”“出发”后，分别回应了路线起点、只看过封面和准备同行，但推荐项始终保持为：

> “有你的份。我本来就想约你；途中谁想改，我们就一起决定。”

存档中的 `choice_label_overrides` 也一直保留同一旧值。与此同时，希尔在玩家询问去处时再次主动说出“七项安排”，使本应由玩家保留的小惊喜变成双方反复讨论的任务标题。

这次问题由代码契约和早期剧情出口共同造成：

- 提示词虽然要求未命中时改写每一个当前选项，但解析层接受缺失、原样返回和只改标点的 `choice_rewrites`，没有把它们判为无效。
- 回合服务把新改写合并到旧 `choice_label_overrides`；模型只要有一轮漏写，上一轮按钮就会永久保留，后续对白即使改变也无法更新界面。
- 自由互动每轮都会读取完整背景、限制和主线目标。旧提示没有强调这些信息只是世界边界，小模型便把显眼的清单标题和项目数量当成安全话题反复回扣。
- `node_protect_charm` 的唯一出口要求玩家先确认清单里有猫娘。即使玩家已经明确说“出发”，也不能以当前意图推进，只能被旧按钮拉回清单话题。

当前修正如下：

- 未命中的自由互动必须为每个当前 Choice 返回一条与“当前显示文案”和“作者原文”均有实质差异的新文案；缺项、类型变化、原样返回或仅改标点会触发一次定向纠错。
- 每次自由互动直接替换上一轮按钮覆盖值，不再合并旧覆盖。纠错后仍无有效改写时回到作者原文，至少不会继续展示与当前上文无关的旧按钮。
- 角色互动提示明确规定：背景、标题和主线只用于防越界，不是必须讨论的话题；玩家没有主动询问时，猫娘不得复述清单名称、项目数量或关系前提，应优先回应最近对话和眼前事件。
- 开场挂坠节点不再追问“安排里有没有我”，而是由猫娘直接发出纪念祭邀请并询问是否一起出发；推荐台词收口为“好，那就一起出发吧”。因此玩家说“出发”已经能等价完成当前 Choice，不需要先复述作者预设的清单说明。
- 接受邀请后先收下一张普通入场券并前往入口。猫娘只表示玩家可以在那里分享愿意公开的路线，不想说的部分可保留为惊喜；玩家主动递出路线后，她才回应已经公开的内容，且不再报“七项”或“前六项”的总数。
- 剧本限制新增明确知识与话题边界：在玩家主动公开路线前，猫娘只能在开场说明一次自己看见过封面，之后不得主动重复“七项安排”或追问项目数量。

推荐项的“关联上文”不等于随意改写核心剧情。模型只能重新表达尚未完成的当前意图，稳定 ID、Choice 类型、目标节点、作者 callback 和权威状态仍由静态图控制。自由聊天本身仍不建立新剧情事实；只有玩家明确说出或实施当前 Choice，才允许推进。

## 小剧场规则不进入角色对白

最新会话 `/Users/mac/Library/Application Support/N.E.K.O/theater/sessions/theater_bcd6e970-02ce-4b4b-be75-273cce4ec46b.json` 中，玩家主动在入口展示愿意公开的路线后，希尔说出了“每一段皆可跳过，节奏亦需由我们共同商定”。这不是人格文件中的心理活动泄漏，而是作者底稿“每一段都能跳过，名字和节奏也要由我们一起决定”的直接人格化转述。旁白已经展示路线，猫娘却继续解释可跳过和共同决定，听感像在宣读小剧场规则。

当前产品决定是让玩家通过演绎和推荐项自由探索，不由猫娘或开演前故事卡讲解系统机制：

- `scenario_card.rules` 保持为空，开演前不再展示回合数、推荐项用途、自由输入用途或行动与对白分组。
- 顶层公开摘要不写“十八个推进回合”；十八轮只作为作者结构和自动化验收事实。
- 猫娘读完路线后只回应玩家的准备、尊重折起的惊喜，并自然询问最想去的第一站；不说“每一段可跳过”“节奏共同商定”等机制式免责声明，也不使用提前定义关系的称呼。
- 下一条推荐台词直接进入第一站，不再让玩家复述“途中想停、想改”的规则。
- 亲密边界不能因此消失，但要通过具体情境自然呈现。例如拥挤处由猫娘主动走在人潮外侧并约定走散后的会合点；这属于角色行动与照顾，不是系统规则说明。
- `restrictions`、内部主线目标、稳定 Choice ID 和十八轮结构继续保留给作者与运行时，不因为前台隐藏规则而失去因果和安全边界。

## 正式确认关系前不安排情侣式接触

前一版本虽然写明双方尚未正式确认关系，却在路线公开后由猫娘提出“情侣练习”，又把清单第一项设计成牵手或勾袖口；屋顶告白前的 callback 还让猫娘主动牵住玩家。这样把作者知道的终局关系提前当成角色已经拥有的关系，破坏了从长期搭档到正式交往的递进。

当前关系顺序收口为：

1. 开场至入口：双方只是互相在意的长期搭档，猫娘发出纪念祭同行邀请，但不为关系命名。
2. 第一项：二人各自戴好普通纪念腕带，通过并肩位置、步速和走散后的会合点穿过人潮，不发生牵手、十指相扣、勾袖口或手背接触。
3. 中段：礼物、甜点、照片和长期观察逐步证明彼此在意；亲密来自细节回应，不来自预先安排的身体接触。
4. 屋顶前：即使猫娘已经先说出喜欢，玩家尚未回应关系问题时，二人仍只并肩上楼，不牵手。
5. 屋顶告白：两条玩家 Choice 都明确提出“正式交往”；猫娘在目标节点明确回答“我愿意从今天起和你正式交往”，关系到这里才成立。
6. 关系成立后：终场通过共同握住一支笔的两侧写下明天完成亲密升级，本剧不再强行补一段牵手。

`choice_hold_hands`、`choice_link_sleeves`、`choice_accept_practice_date` 等旧 ID 继续作为存档兼容标识保留，但玩家可见标签、callback 和节点演出不得按旧名称解释。测试必须扫描关系确认前的所有 Scene、对白、推荐项和 callback，确保旧语义没有残留。

## 当前输入晚一轮与自由演绎越界修正

最新正式会话 `/Users/mac/Library/Application Support/N.E.K.O/theater/sessions/theater_cf030da7-ba89-4853-b37a-c2a07bc6cf73.json` 复现了连续两轮错位：玩家说“出发吧，目标星灯祭”时，糖糖仍先回应上一轮对星星挂坠的称赞；玩家下一轮问“我们先去哪里？”时，糖糖才回应“出发”，又把“第一站去哪里”原样反问玩家。旁白还写成“糖糖顺势挽住你的手臂”，越过了尚未正式确认关系的边界。

定责结果是运行时顺序问题，不是模型没有收到玩家输入：

- 自由输入调用先在旧 Node 上生成完整演出，模型返回 `matched_choice_id` 后，服务端才提交目标 Node。原代码只更新状态、Scene 和 Trace，却继续展示旧 Node 已经生成的台词，因此自然语言推进必然有机会晚一拍。
- 提示词已经把“本轮唯一回应目标”放在上下文末尾，但没有对“把同一个问题反问玩家”做确定性拦截；模型即使看见当前输入，也可能用反问规避回答。
- 剧本限制写明告白前不得牵手，但原展示前校验只扫描 Node 的禁用对白词，既不扫描旁白，也不执行剧本级、分阶段的关系护栏。

当前实现按以下边界修正：

- 第一次自由输入模型调用只承担稳定 Choice 路由。命中后先提交作者目标节点，再用目标 Node、目标 callback、提交后的事实和下一组真实推荐项重新生成正式演出；旧节点生成的台词不再显示。
- 玩家提出直接问题时必须先回答当前已知内容，不能把同一问题换一种说法反问。高置信去向反问会触发一次纠错；若纠错反而破坏首版已经合格的对白，保留首版对白并让推荐项回到作者原文，不用坏的第二版覆盖。
- 去向回答中出现公开上下文从未出现的书名号式命名地点时视为幻觉，例如真实重放里曾出现的“星愿风铃”摊。纠错仍失败时，从当前公开 Choice 提取最近已知目的地作保守回答；本段剧情会回答先到纪念祭入口，不会发明新摊位。
- `runtime_guardrails.conditional_output_guards` 同时扫描 narration 和 dialogue。牵手、十指相扣、勾袖口、挽手臂等词在“玩家明确选择正式关系”的作者事实提交前禁止出现；该事实提交后阶段护栏自动解除，避免误伤确认关系后、具有当下同意的自由互动。
- 上述代码均保留了中文注释，说明二次生成、首版安全对白保留、剧本阶段护栏和去向保守回退的业务原因。

## 内部规则与角色台词分离

最新正式会话 `/Users/mac/Library/Application Support/N.E.K.O/theater/sessions/theater_48e6d846-3e11-4d2e-91dd-709567812e40.json` 显示，糖糖在玩家接受邀请后说出“入场券只是进门凭证”“今天怎么过全看心情”“路线可以给她看”“惊喜可以藏着”“她绝对不偷看”等整段说明。每句话单独看都符合边界，但合在一起像角色向玩家宣读小剧场规则，既啰嗦，也把本来应由运行时默默执行的限制暴露到台词中。

当前框架将提示词明确分成两个变量：

- `内部规则（只执行，不复述）`：保存作者限制、禁止假设、输出硬边界、主线目标、本轮演绎指令、作者意图和 Choice 路由语义。它只控制模型如何演，不属于猫娘知识或对白素材。
- `公开演绎上下文`：只保存角色、背景、当前 Scene、最近对话、当前节点演出和推荐项。猫娘只能依据这一部分回应玩家。

系统提示与纠错提示都明确要求：内部规则只能通过“不发生越界行为”来执行，不能在 narration、dialogue 或推荐项中引用、解释、概括、承诺遵守或换一种说法复述。剧本还用 `runtime_guardrails.forbidden_output_patterns` 对本故事最常见的规则说明句做展示前检查，例如“中途停或换时商量”“入场券不会安排今天”“不想说就留作惊喜、不会追问”。

作者台词也同步收口：

- 保留挂坠后，猫娘只说明自己带了两张票、原本就是来邀请玩家，并直接问是否一起出发；删除途中停止、改路线和共同商量的说明。
- 玩家答应后，猫娘只说先去旧街入口，到达后想看看玩家挑的第一站；删除入场券用途、路线公开范围、惊喜和不追问声明。
- 玩家展示路线后，猫娘回应已经看见的路线，并把折起的背面交还玩家收好；不再口头承诺“不会偷看”。

这些边界仍然存在于内部规则和确定性校验中，只是不再由角色说给玩家。纪念祭摊位自身必须公开的玩法条件仍属于剧情事实，不与运行时内部规则混为一谈。

## 玩家公开背景补全

最新会话 `/Users/mac/Library/Application Support/N.E.K.O/theater/sessions/theater_4c43f1d7-db5c-497d-bbb3-a33bd12d36e9.json` 暴露出新的信息断层：开场只让猫娘说自己看见“七项”两个字，下一回合却直接谈“今天别按清单来”。剧本内部虽然记录她只看过清单封面，但玩家看不见这条内部事实，也无法判断她为何知道纸张是清单、为何知道清单与今天有关。该回合又因模型超时使用作者台词回退，使断层直接暴露在公开对白中。

页面真实显示链路为 `scenario_card.brief`，只有该字段为空时才回退到顶层 `summary`；因此只修改内部 `background` 对玩家没有作用。该字段曾为了补足信息断层扩写到七百字以上，但后续版本把玩法规则、剧情走向和玩家需要做什么也写进了开演前背景，造成说明过度。

当前产品决定是让公开背景只回答“你们是谁、为什么来到这里、开场有哪些东西”，自然停在演绎开始的位置：

- 玩家知道自己为了纪念祭告白，准备了路线、礼物、意外预案、备用台词和未公开的告白。
- 清单封面明确写着“星灯纪念祭·七项安排”。
- 当前活跃猫娘也想邀请玩家参加纪念祭，提着一个牛皮纸袋来到门口；袋里装着两张普通双人入场券和折好的纪念祭导览，袋口系着她亲手缝制的星星挂坠。
- 背景到“针脚有些歪的星星挂坠”结束。猫娘看见封面和后续剧情留给正式演绎；路线修改权、亲密边界等内部规则由运行时直接执行，不在背景或猫娘台词中说明。
- JSON 使用 `{{lanlan_name}}` 保存角色位置；故事列表接口在返回页面前以服务端当前活跃猫娘名替换。不得把某次验证时的“小葵”或其他名字写死在剧本里。

模型使用的 `background`、`world_seed`、开场 Scene、作者开场对白和生成意图仍保留完整知识边界。公开背景可以简短，但静态剧本仍必须保证纸袋、入场券、导览和挂坠在开场 Choice 出现前可见，不能依赖模型临场补充。

当前字段职责如下：

| 字段 | 当前用途 | 必须保持的事实 |
| --- | --- | --- |
| `scenario_card.brief` | 玩家开演前实际看到的背景正文 | 动态替换角色名后不少于 350 字；只交代双方关系、玩家清单和猫娘提袋到门口，不说明后续玩法或玩家该做什么。 |
| `background` | 每回合演绎模型优先读取的故事背景 | 与玩家公开背景采用同一人物关系和知识边界，不能让模型误以为猫娘读过内页。 |
| `world_seed` | 作者级完整前提与后续维护依据 | 记录玩家准备清单、猫娘独立准备普通入场券，以及双方各自不知道什么。 |
| `opening_dialogue` | 开场人格转述的作者语义底稿，也是模型失败时的安全回退 | 猫娘明确只看见封面标题、没看内页，再提出接住星星挂坠。 |
| 初始 Scene | 玩家开场时可观察的现场证据 | 封面标题可见；牛皮纸袋、袋内入场券与导览、袋口挂坠都有明确位置，猫娘没有碰清单内页。 |

为防止模型从作者背景中泄漏猫娘不该知道的内容，`seed.forbidden_assumptions` 还明确禁止四种假设：猫娘读过清单内页、猫娘知道路线/礼物/备用台词、猫娘读过准备好的告白、普通入场券包含编号任务或挑战。后续改写背景或对白时，这四条边界必须与公开简介一起检查，不能只看文案是否顺畅。

## 唯一七项来源修正

最新会话 `/Users/mac/Library/Application Support/N.E.K.O/theater/sessions/theater_7b3149be-ad32-4496-be8d-a77ed34c2806.json` 暴露出剧本的核心前提冲突：玩家公开背景一方面说清单由玩家亲手写，实际演绎却让猫娘说“两张券上写的七项挑战规则”，后续又通过印章、计分和奖励推进。这样同时存在“玩家私人七项”和“纪念祭官方七项”两套系统，玩家无法判断每一项究竟是谁安排的，猫娘也像凭空知道尚未公开的清单内容。

当前唯一真值收口为：

- 只有玩家亲手写的私人约会清单包含七项。前六项是玩家从纪念祭可自由选择的摊位和去处中排出的相处路线，第七项是折在背面的告白准备稿。
- 猫娘带来的两张券只是普通双人入场券，只负责进入旧街；券面没有编号任务、挑战规则、印章、计分、奖励或天气挑战。
- 猫娘开场只看见封面“星灯纪念祭·七项安排”，不知道清单是否为她而写。她说明自己本来就是来邀请玩家，提出途中停止或改路线都由两人商量，然后直接询问玩家是否同行；开场后不再主动追问清单。
- 玩家抵达入口后只向猫娘公开愿意分享的路线，藏着告白的一面始终折起。猫娘只回应实际看见的路线，不主动报项目总数，也不知道折起部分的告白内容或备用回答。
- 后续所有“第一项”至“第七项”都必须明确或可追溯地指向玩家清单。腕带只是纪念品，纪念册来自照相亭，阵雨由入口天气牌提前铺垫，三者都不承担任务进度。
- 猫娘仍是共同发起者：她准备门票、提出同行、在人潮中主动走外侧、分享自己的观察，并在第六项后提出清单外散步。删掉官方挑战和提前亲密不等于把她改成被动跟随者。

这次修正保留十八个 Node、剧情边和既有 Choice ID，以免破坏已有会话兼容；`choice_read_pair_tickets`、`choice_receive_sync_stamp` 等旧 ID 仅作为内部稳定标识保留，不再代表玩家可见剧情语义。玩家可见标题、Scene、对白、选项和回调不得再按旧 ID 解释。

| 当前输入对齐 | 自然语言命中 Choice 后在目标 Node 重新生成当轮演出；禁止把当前去向问题反问玩家，也不允许回答中凭空出现未公开的命名地点。 |
| 动态关系护栏 | 告白前的身体接触禁词同时约束旁白和对白；正式关系事实提交后自动解除阶段限制，不把前半段规则永久带入结局后的自由互动。 |
| 静默规则 | 作者限制、禁止假设、输出边界和主线目标独立存入内部规则变量；猫娘只消费公开演绎上下文，不向玩家解释规则。 |
| 作者完成表达 | `choice_promise_one_surprise` 声明“出发、出发吧、出发，目标星灯祭、走吧”；服务端唯一命中后先提交目标节点，再生成当轮演出，不让旧节点重复邀请。否定句、疑问句和歧义命中仍留给自由互动。 |
| 问号语气 | 普通文本末尾的问号不再被模型单独当作询问证据；回应刚刚邀请时，“那就……出发？”按无否定、无具体疑问词的即时承接判断为接受，“为什么出发？”“现在出发吗？”等真实问题仍停留在当前节点。 |

## 创作检查规则

以后每条边 `A → B` 都执行以下检查：

- B 的每个推荐选项能否只凭 A 结束时的公开信息理解？
- 隐去 Node 标题、作者摘要、生成指引和后续剧情后，玩家角色是否仍会自然想到这个选项？
- 选项是在回应眼前的人和事，还是替作者总结主题、提示正确路线或预告未来？
- 选项内每个名词是否已经出现，代词和“另一、再次、最后”等词是否有明确前件？
- 选项要求回应的问题、邀请或规则是否已经由 A 提出？
- B 的回调是否写玩家刚完成的动作，B 的对白是否对此产生反应？
- 涉及亲密或共同承诺时，对方是否已经表达当下同意？
- 每个“第几项”是否都能追溯到玩家已经公开的清单内容，猫娘是否只知道当下有权知道的部分？
- 是否误把入场券、腕带、照相亭纪念册或天气牌写成任务、印章、计分或奖励系统？只要出现即属于前提回退。
- 自由互动后的每条推荐文案是否实质承接了玩家刚说的话和猫娘刚作出的回应，而不是旧覆盖值、作者原文或仅改标点的伪更新？
- 背景、标题和主线是否只用于约束，还是被猫娘反复当成话题？玩家没有主动询问时，不得为了“留在主线”机械复述清单名称和项目数量。
- 当前推荐项的核心意图是否仍符合玩家最近表达？如果玩家已经明确完成“出发”等现有 Choice，就应推进而不是强迫其补说另一段作者台词。
- 这句对白是在追求、试探、邀请、拒绝或决定，还是在替系统解释“可跳过、如何推进、怎样选择”？后者应移回内部约束或直接删除。
- 同意边界是否落在具体动作与当下回应里？不要为了隐藏规则而删除边界，也不要把边界写成角色宣读的通用玩法说明。
- 是否因为作者知道最终会相爱，就在关系确认前写入牵手、十指相扣、勾袖口、情侣称呼或其他只有终局才成立的亲密？只要出现就必须后移或改成并肩同行等中性动作。
- 关系确认节点是否由玩家明确提出正式交往、猫娘明确答应，而不是用“真正的约会”“练习结束”含糊代替关系结论？
- 自然语言已经完成当前 Choice 时，展示的是否是目标 Node 对本轮输入的回应，而不是旧 Node 生成后被延迟一轮的台词？
- 玩家提出“先去哪里”等直接问题时，猫娘是否先回答已知目的地或明确说明尚未知晓；是否把原问题反问回来，或发明从未公开的命名摊位？
- 关系阶段限制是否同时检查旁白与对白，并且只持续到对应作者事实提交，而不是整场永久禁用？
- 这句话是在回应眼前的人和事，还是把“允许什么、禁止什么、可以怎样调整”解释给玩家？后者应删除台词，只保留在内部规则变量。
- 去掉规则说明后，剧情所需的公开事实、邀请、目的地和下一步是否仍然完整？不能用“静默规则”掩盖缺失的剧情铺垫。
- 当前 Choice 是否有少量、明确且不含歧义的 `completion_phrases`？只声明已经完成该动作或答复的自然短句，不把否定、疑问、假设或包含匹配当成确定性推进依据。
- 作者完成表达是否只在唯一命中时推进？若多个当前出口声明相同表达，必须交回上下文路由，不能由服务端猜选剧情。

## 验收标准

- JSON 可解析，Story Loader 可加载。
- 完整推荐路径连续推进十五轮并抵达既定结局。
- 同一作者隐藏意图连续表达一次、两次时停留，第三次进入支线；中途换话题后重新计数。
- 隐藏边、意图 ID、计数和分支承诺不出现在页面推荐项或猫娘台词中。
- 暴雨支线既能保留已谈事实汇回告白主线，也能抵达“明天再把话说完”的替代结局。
- 每轮推荐选项的关键前提，都能在上一轮公开演出中找到。
- 每个推荐选项都能从玩家角色的当下已知信息推出，不依赖作者视角或未来剧情。
- 不出现未介绍的地点、道具、规则和关系结论。
- 七项只来自玩家私人清单；普通入场券在所有玩家可见文本与模型生成约束中都不承载任务。
- 猫娘在玩家主动公开路线之前不作确定性判断；公开后只知道纸面实际展示的路线，折起部分的内容仍保密。
- 猫娘只在开场说明自己看见封面标题；玩家主动公开路线前，不得再次主动说出“七项安排”或围绕数量追问。
- 连续自由互动后，推荐项不得保留上一轮的旧覆盖值；未命中时必须实质更新，模型失败时回到作者原文。
- 玩家以“出发”等等价表达接受邀请时，能够命中当前 Choice 并离开开场话题，不要求补充清单解释。
- 开演前公开规则列表为空，摘要不出现回合数、推荐项用途或自由输入说明。
- 猫娘对白和玩家推荐台词不宣读“每一段可跳过”“节奏共同商定”“途中想停、想改”等通用机制；具体亲密选择仍通过场景动作自然建立。
- 正式确认关系前，玩家可见 Scene、对白、推荐项和 callback 不出现“情侣练习”“假装情侣”、牵手、十指相扣、勾袖口或手背试探。
- 屋顶前双方只并肩同行；两条告白 Choice 都明确提出正式交往，猫娘目标对白明确答应正式交往。
- 自然语言“出发”等价命中后，当轮对白必须立即进入目标节点，不再回应星星挂坠；下一轮“我们先去哪里？”必须回答当前已知去向或说明信息不足，不得反问相同问题。
- 去向回答不得引入背景、Scene、已公开状态、最近对话和当前推荐项中从未出现的命名地点。
- 告白前关系护栏同时覆盖 narration 与 dialogue；正式关系事实提交后解除，保证前后阶段语义一致。
- 提示词中的内部规则与公开演绎上下文必须是两个独立字段；公开演绎上下文不得包含作者限制、禁止假设、输出硬边界或主线目标。
- 猫娘回复不得宣读或改写内部规则。开场邀请与前往入口的演出中不得出现“中途停或换时商量”“入场券不会安排今天”“惊喜可以不说、不会追问”等规则性说明。
- 玩家输入“出发”“出发吧”“出发，目标星灯祭”或“走吧”时，只调用一次目标 `node_theme_question` 的演出生成并在同一回合推进；不得先调用旧 `node_protect_charm` 生成重复邀请。
- 玩家输入“先别出发”或“为什么出发”时不得确定性推进；同一作者表达同时命中多个当前 Choice 时也不得擅自选路。
- 玩家用“那就……出发？”回应刚刚的同行邀请时，模型不得仅因句末问号返回空 `matched_choice_id`；含“为什么、吗、你想”等实际疑问语义的输入仍不得命中。
- 开演前背景只持续到猫娘提袋来到门口，不提前说明玩家行动、玩法规则或后续剧情；页面不出现 `{{lanlan_name}}` 或“当前猫娘”等占位称呼。
- 开演前背景在动态替换当前猫娘名后不得少于 350 字；测试还要用单字角色名覆盖最短替换情况，不能只按 JSON 模板长度判断。
- 纸袋必须先作为装有普通入场券和纪念祭导览的牛皮纸袋出现，挂坠明确系在袋口，之后才能写“接住袋口的星星”。
- Story、Runtime、混合输入和浏览器相关回归测试通过。
- 剧本体验稳定后，只有在用户明确要求“同步 Skill”时，才把本轮已经验证的规则同步到本地 `neko-theater-story-writer` Skill。
