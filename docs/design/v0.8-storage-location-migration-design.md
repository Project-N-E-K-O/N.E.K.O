# v0.8 存储位置首启迁移设计（按 611ac 提交前发布基线与后续代码线校正）

> 目标：在 `v0.8` 引入“存储位置选择 + 可选迁移”，同时不破坏现有 phase-0 cloudsave 启动链路。
>
> 评估结论：
>
> - 以 `611ac1816864a4d3c97f13afb0381f1c433b0ef2` 的父提交 `c28a968b31aa61df8f685dae56867789dd1570b8` 作为“当前发布版”基线时，存储仍是 `Documents/N.E.K.O` 主根；Windows 仅在 `Documents` 不可写时才回退到 `%LOCALAPPDATA%/N.E.K.O`，更低优先级才是打包目录或 `cwd` 下的 `N.E.K.O`。
> - `611ac1816864a4d3c97f13afb0381f1c433b0ef2` 及之后的后续待上线代码线，才逐步演进出“旧根目录一次性导入/修复 + cloudsave 维护态/回滚 + 运行时真源路径”等基础设施；这些能力属于本文第 1.3 节起的后续代码线现状，不是第 1.1-1.2 节判断发布版原始存储位置的依据。
> - 无论按当前发布版还是 `611ac...` 之后的后续代码线来看，项目都还没有 `storage_policy`、`anchor_root / selected_root` 双根模型、首启迁移窗口、自定义根持久化与恢复机制。
> - 真正的实现难点不在 copy 本身，而在“启动时机、`ConfigManager` 单例路径绑定、导入时缓存路径的模块”。

## 0. 关联文档与约束

本设计与以下文档保持一致：

- `docs/design/cloud-save-sync-optimization-plan.md`
- `docs/design/v0.8-storage-location-migration-implementation-guide.md`

当前实施状态补充：

- 截至当前分支，实施指导文档中的 `Stage 1` 已完成首轮落地
- 已落地内容仅覆盖“launcher phase-0 首启前端骨架 / probe gate / 事件与状态打点”
- 该阶段仍是 `probe_only`，不代表 `storage_policy`、双根绑定与真实迁移已经完成

一致性说明：

- 当前 cloudsave 基线文档口径是：`cloudsave/` 位于运行时根目录下。
- 若本方案采用“方案 A：`cloudsave_root` 固定在 `anchor_root`”，必须在同一 PR 内同步更新 cloudsave 基线文档中的路径口径、测试清单与 Auto-Cloud 推荐规则。

本设计约束：

- 不破坏 Steam Auto-Cloud 现有主链路。
- 不支持运行中热切换根目录。
- 不在 `v0.8` 一次性重写全部历史调用点，只覆盖启动、迁移、关键读写链路。
- 首启存储位置选择不能依赖“服务完全启动后再弹 Web 首屏”，必须早于运行时根路径绑定。
- 正式发布后的主验收路径是“Steam 启动”与“打包后的桌面启动器启动”，不是源码模式。

### 0.1 发布态启动口径

文档必须区分以下几种启动方式，否则“能否正常”会被混淆成不同问题。

正式发布后的主路径：

- `frozen + Steam 启动`
- `frozen + 桌面启动器启动`

开发/兼容路径：

- `source + launcher.py`
- 手动三服务模式：`main_server.py` / `memory_server.py`

术语约定：

- “Steam 启动”：指 Steam 客户端拉起打包后的 launcher / 桌面壳
- “桌面启动器启动”：指用户直接双击打包后的桌面应用入口
- “源码 launcher 启动”：指开发环境执行 `python launcher.py`

本文后续若写“桌面端启动”而未特别说明，默认指“打包后的桌面启动器启动”，不是源码模式

当前项目已有基线说明：

- 正式发布主路径仍是 `Steam Auto-Cloud 同步本机 cloudsave/`
- `download_cloudsave_bundle_from_steam()` / `upload_cloudsave_bundle_to_steam()` 的 RemoteStorage bundle helper 只服务于 source 联调兼容路径
- 非 source launch 下，这些 helper 会返回 `reason = "not_source_launch"`

因此本设计的正确口径应是：

- `v0.8` 存储位置迁移功能必须首先保证 `frozen + Steam/桌面启动器` 正常
- source 模式是否也能工作，属于开发便利性目标，但不是发布成败的首要判据

### 0.2 对“正式发布后能否正常”的结论

基于第 1.1-1.2 节的当前发布版存储事实，以及第 1.3 节起后续代码线 / 当前工作区里可复用的基础设施，若按本文新增约束实现，正式发布后通过 Steam 或桌面启动器启动应当可以正常工作，前提是以下条件被满足：

1. 存储布局解析发生在 launcher phase-0，早于任何服务进程首次 `get_config_manager()`
2. 子进程在 import `main_server.py` / `memory_server.py` 前能拿到同一份 storage layout
3. `cloudsave_root` 在发布态继续固定到 `anchor_root`
4. 首启选择/迁移 UI 由 launcher 或桌面壳负责，而不是依赖主服务 Web 首屏

如果这四项没有落地，则正式发布后最容易出现的问题是：

- Steam 启动能进程序，但服务进程先绑定到默认根，导致首启选择失效
- 桌面端看到了迁移 UI，但主服务已经按旧根初始化，产生读写分裂
- Steam Auto-Cloud 仍同步 `anchor_root/cloudsave`，而运行时误迁到了 cloudsave 同层或子层，造成路径混乱

结论可以写得更明确一些：

- 当前文档方向是对的，但还需要显式写明“发布态主路径是 frozen launcher，不是 source helper”
- 如果不补这层说明，文档描述还不够适合正式发布验收
- `v0.8` 至少要兼容两类升级来源：
  - `c28a968...` 为代表的当前发布版：`Documents/N.E.K.O` 主根
  - `611ac...` 之后的后续待上线代码线：运行时主根已转向平台标准应用数据目录，并叠加了 cloudsave bootstrap / legacy import 等机制

## 1. 现状基线（区分发布版口径与后续代码线能力）

本章分成两层口径：

- `1.1-1.2`：按 `611ac1816864a4d3c97f13afb0381f1c433b0ef2` 的父提交 `c28a968b31aa61df8f685dae56867789dd1570b8` 记录“当前发布版”的历史存储方式，用来回答 Steam 发布版本原本把数据存在哪里。
- `1.3-1.5`：按 `611ac...` 之后直到当前工作区的后续代码线记录 `v0.8` 可直接复用的实现能力，用来回答后续改造时能继承什么基础设施。

### 1.1 历史发布侧根目录解析与启动顺序

在历史发布侧基线 `c28a968b31aa61df8f685dae56867789dd1570b8` 里，`ConfigManager.__init__()` 会直接调用 `_get_documents_directory()`，然后立即绑定：

- `self.app_docs_dir = self.docs_dir / self.app_name`
- `self.config_dir = self.app_docs_dir / "config"`
- `self.memory_dir = self.app_docs_dir / "memory"`
- `self.plugins_dir`、`self.live2d_dir`、`self.vrm_dir`、`self.mmd_dir`、`self.workshop_dir`、`self.chara_dir`

也就是说，当时的运行时根不是“平台标准应用数据目录优先”的新模型，而是 `ConfigManager` 初始化时当场解析出的单根模型。

历史代码的真实解析顺序是：

1. Windows 发布侧主候选仍然是 `Documents`
   - 系统 API 返回的“我的文档”路径
   - 同盘符下的 `文档` / `Documents` / `My Documents` 替代路径
   - 注册表 `Personal` 路径
   - 同盘符根下枚举出的 `documents` / `文档` / `my documents`
   - `~/Documents`
   - `~/文档`
2. Windows 仅在 `Documents` 不可写时，才把 `%LOCALAPPDATA%` 作为优先回退位置
3. 若仍不可用，再回退到：
   - 打包态：`Path(sys.executable).parent`
   - 开发态：`Path.cwd()`
4. 若以上候选都失败，最终兜底仍是 `Path.cwd()`

补充说明：

- macOS 历史基线是 `~/Documents`，再退到 `cwd`
- Linux 历史基线是 `XDG_DOCUMENTS_DIR`（若设置）或 `~/Documents`，再退到 `cwd`

因此，Steam 发布版本对应的“原始存储位置”应更准确地写成：

- 默认主根是 `Documents/N.E.K.O`
- Windows 在 `Documents` 不可写或受 CFA/反勒索防护影响时，回退写入到 `%LOCALAPPDATA%/N.E.K.O`
- 更低优先级才是 `Path(sys.executable).parent / "N.E.K.O"` 或 `Path.cwd() / "N.E.K.O"`

补充说明：

- 历史代码里已经有 CFA 兼容逻辑：若第一个可读候选与最终可写候选不同，会保留 `_readable_docs_dir`，使模型等读取仍可命中原始 `Documents/N.E.K.O`
- 这意味着“原位置”在 Windows 上不仅是默认主根，也是 CFA 场景下的只读来源
- `get_config_manager()` 在该历史基线下仍是单例，首次初始化时选中的根会被后续模块沿用；当时并不存在 `storage_policy` 或独立的 launcher phase-0 存储布局协商

历史发布侧代码默认把以下内容绑定在单一 `app_docs_dir` 下；这里的 `app_docs_dir` 在默认 Windows 场景下就是 `Documents/N.E.K.O`。

### 1.3 后续代码线已具备的迁移与保护能力

以下内容从本节开始切回 `611ac...` 之后直到当前工作区的后续代码线，用于评估 `v0.8` 可以直接复用哪些基础设施；它们不代表历史 Steam 发布版本当时已经具备这些能力。

这批能力主要由 `a802abb0` 的 cloudsave/runtime 基建、`8562ebeb` 的 startup bootstrap 兜底、`739555cb` / `4c3196d1` 的日志体系重构，以及 `6e0af346` 的运行时真源路径收敛逐步带入当前工作区。

当前代码已经具备可复用的基础设施：

- `import_legacy_runtime_root_if_needed()`：扫描 `get_legacy_app_root_candidates()`，对旧根执行一次性导入/修复。
- 旧根修复不是纯覆盖：
  - 若目标根已有 seeded 内容，会合并 `characters.json`、`user_preferences.json`、`core_config.json`、`voice_storage.json`、`workshop_config.json`
  - 会保留 `character_tombstones.json` 约束
- 旧根导入使用临时目录 + `os.replace(...)` 切换，并支持 backup 目录回滚。
- `bootstrap_local_cloudsave_environment()` 会：
  - 确保 `cloudsave/` 结构存在
  - 确保 `root_state` / `cloudsave_local_state` / `character_tombstones` 存在
  - 恢复 stale write-blocking mode
  - 初始化或修复 manifest
- `cloud_apply_fence()` 提供跨进程互斥 + `root_state.mode` 写栅栏。

与本方案最相关的现有状态字段：

- `root_state.mode`
- `root_state.current_root`
- `root_state.last_known_good_root`
- `root_state.last_migration_source`
- `root_state.last_migration_result`
- `root_state.last_migration_backup`
- `root_state.last_successful_boot_at`

`v0.8` 若把“旧数据保留后可见、可清理”纳入正式能力，建议在 `root_state` 中新增：

- `legacy_cleanup_pending`

额外评估结论：

- 当前已有 `ROOT_MODE_DEFERRED_INIT`，这非常适合在“`selected_root` 不可用但又不能静默切回默认根”时作为只读恢复态使用。

### 1.4 后续代码线读写路径语义

当前存在两套配置路径 API：

- `get_runtime_config_path(filename)`：始终返回运行时真源路径
- `get_config_path(filename)`：若运行时文件不存在，会回退到项目目录 `config/`

已确认相对收敛的链路：

- `ConfigManager.save_characters()`：写 `get_runtime_config_path("characters.json")`
- `ConfigManager.save_workshop_config()`：写 `get_runtime_config_path("workshop_config.json")`
- `ConfigManager.save_json_config()`：默认写运行时配置目录
- `memory/recent.py`、`memory_server.py`、`main_routers/config_router.py` 中 `core_config.json` 的关键读取点已使用运行时真源路径
- `utils.preferences` 写路径已固定到运行时真源

仍保留回退读取语义、需要在 `v0.8` 重点治理的链路：

- `ConfigManager.load_characters()` 默认读 `get_config_path("characters.json")`
- `utils/config_manager.py` 中 core config 的环境读取逻辑仍读 `get_config_path("core_config.json")`
- `utils/preferences.py` 读取仍可能命中项目配置回退
- `utils/autostart_prompt_state.py`
- `utils/tutorial_prompt_state.py`
- `utils/holiday_cache.py`

结论：

- 现阶段最大的“迁移后读写分裂”风险不是保存路径，而是这些回退读取路径。

## 2. 对原设计稿的差距评估

### 2.1 已有基础比原稿更强

原稿中“启动导入现状”“维护态栅栏”“旧目录导入/修复”这几块方向是对的，但当前代码已经比文稿描述更具体：

- 不是简单“旧目录 copy 一次”，而是带 seeded target 修复与配置 merge 的导入流程。
- 不是抽象“维护态”，而是已存在跨进程锁 + `root_state.mode` 栅栏。
- 启动顺序不是泛指“启动阶段会执行”，而是 launcher phase-0 中明确先 fence，再 bootstrap，再 cloudsave import。

### 2.2 原稿低估了一个隐藏风险：路径绑定时机

当前项目大量模块会在 import 阶段获取 `get_config_manager()` 或缓存路径，例如：

- `main_server.py`
- `memory_server.py`
- `utils/preferences.py`
- 多个 memory 子模块

因此：

- 若存储位置选择发生在服务已经启动之后，绝大多数模块已经绑定旧路径。
- 所以“首屏初始化后显示阻断式 `StorageMigrationDialog`”这一表述不适合当前架构。
- 首启窗口必须前移到 launcher/pre-bootstrap 阶段，或者引入高风险的全局路径重绑定方案。

### 2.3 原稿低估了单根假设的渗透范围

当前不仅 `ConfigManager` 用单根模型，以下逻辑也默认 `cloudsave` 和运行时根同根：

- `ConfigManager.cloudsave_dir`
- `ConfigManager.local_state_dir`
- `acquire_cloud_apply_lock()` 的锁文件位置与 mutex 命名
- `LEGACY_RUNTIME_DIR_NAMES`
- `_copy_runtime_root_entries()`
- `_replace_runtime_root()`
- 大量测试断言

这意味着：如果 `v0.8` 采用“方案 A：cloudsave 固定在 `anchor_root`”，现有旧根导入辅助函数不能原样复用，至少要把“可迁移运行时目录”和“固定锚点目录”拆开。

### 2.4 原稿中的 API 设计与当前启动架构不完全匹配

原稿提出：

- `POST /api/storage/location/apply`
- `GET /api/storage/location/migration_progress`

但如果这些 API 依赖主服务启动后才能调用，就已经晚于路径绑定。

因此建议调整为：

- 首启选择与首次迁移：放在 launcher/pre-bootstrap 阶段
- REST API：保留给“启动后查看状态/清理旧备份/只读诊断”，不要作为首次选根的唯一入口

### 2.5 原稿对 fallback 策略还不够保守

若 `selected_root` 曾经被用户使用过，后续启动时它又离线或损坏，不能直接静默回退到 `anchor_root` 并继续正常写入，否则会造成新的数据分叉。

更合理的策略是：

- 进入 `ROOT_MODE_DEFERRED_INIT`
- 前端展示恢复/重试/临时切回的显式选择
- 在用户做出选择前，阻断高风险写入

## 3. 推荐的 v0.8 目标方案

### 3.1 决策门

#### 决策门 A：固定锚点目录

引入固定锚点目录 `anchor_root`，位置沿用当前平台标准应用数据目录：

- Windows：`%LOCALAPPDATA%/N.E.K.O`
- macOS：`~/Library/Application Support/N.E.K.O`
- Linux：`$XDG_DATA_HOME/N.E.K.O`，未设置则 `~/.local/share/N.E.K.O`

这里的“沿用当前平台标准应用数据目录”指的是对齐 `611ac...` 之后后续代码线已经出现的标准应用数据目录优先逻辑，而不是把 `c28a...` 发布版的 `Documents-first` 历史实现误写成当前默认。

#### 决策门 B：cloudsave 与运行时根关系

推荐采用方案 A，并明确写死：

- `cloudsave_root = anchor_root / "cloudsave"`
- `selected_root` 用于用户运行时数据
- 不让 `cloudsave_root` 跟随 `selected_root`

理由：

- 与当前 Steam Auto-Cloud 规则最兼容
- 跨设备行为可预测
- 避免自定义路径导致 Steam 同步失配

#### 决策门 C：切换时机

- 只允许“启动早期切换”
- 不支持运行中热切换

### 3.2 最小可落地的双根模型

为控制改造面，`v0.8` 采用“最小固定锚点”方案，而不是把所有目录都拆到锚点层。

`anchor_root` 固定承载：

- `state/`
- `cloudsave/`
- `.cloudsave_staging/`
- `cloudsave_backups/`
- 迁移检查点与跨进程锁文件

`selected_root` 承载运行时用户数据：

- `config/`
- `memory/`
- `plugins/`
- `live2d/`
- `vrm/`
- `mmd/`
- `workshop/`
- `character_cards/`
- `jukebox/`

以下内容在 `v0.8` 不强制迁入锚点层，以减少非必要重构：

- `logs/diagnostics/`
- `.rollback_tmp/`

它们继续跟随 `selected_root` / `app_docs_dir` 语义，后续版本再决定是否收敛。

但在迁移语义上应把它们视为“运行时派生目录”而不是核心用户资产：

- 切换到新 `selected_root` 后允许在新根按需重新创建
- `v0.8` 不要求把旧根中的历史 `logs/diagnostics/` 或 `.rollback_tmp/` 内容回填到新根
- 旧根中的这两类目录可以随 retained backup 一起保留，后续由 legacy cleanup 统一处理

明确不迁移、继续按外部路径处理的内容：

- Steam Workshop 实际安装目录
- `workshop_config.json` 中的 `user_mod_folder`
- `workshop_config.json` 中缓存的 `user_workshop_folder`

### 3.3 代码层兼容契约

为减少代码面冲击，`ConfigManager` 在 `v0.8` 必须保持以下兼容约定：

- `app_docs_dir` 保持存在，但语义改为 `selected_root`
- 新增 `anchor_root` 属性
- 新增 `runtime_root` 属性，值与 `app_docs_dir` 相同
- `config_dir`、`memory_dir`、`plugins_dir`、`live2d_dir`、`vrm_dir`、`mmd_dir`、`workshop_dir`、`chara_dir` 均继续从 `runtime_root` 推导
- `cloudsave_dir`、`cloudsave_staging_dir`、`cloudsave_backups_dir`、`local_state_dir` 改为从 `anchor_root` 推导

这条契约的意义：

- 大多数现有运行时代码仍可继续使用 `config_manager.app_docs_dir` / `memory_dir`
- 只有真正依赖 cloudsave 和本地状态的模块需要知道锚点层存在

禁止在 `v0.8` 采用的实现方式：

- 运行中对已有 `ConfigManager` 实例做根路径重绑定
- 靠 monkey patch / 全局变量热替换修复 import 阶段缓存路径

### 3.4 `storage_policy`、`root_state`、`storage_migration` 三份状态文件的职责

推荐引入三类状态文件，并明确职责边界。

`anchor_root/state/storage_policy.json`

- 保存稳定路由与用户选择
- 进程重启后仍然有效
- 不记录瞬时迁移阶段

推荐字段：

- `version`
- `anchor_root`
- `selected_root`
- `selection_source`: `default | user_selected | recovered`
- `cloudsave_strategy`: 固定写 `fixed_anchor`
- `first_run_completed`
- `updated_at`

`anchor_root/state/root_state.json`

- 保存运行态与最近一次结果
- 继续沿用当前项目已有职责

延续现有字段：

- `mode`
- `current_root`
- `last_known_good_root`
- `last_migration_source`
- `last_migration_result`
- `last_migration_backup`
- `last_successful_boot_at`

`v0.8` 建议新增字段：

- `legacy_cleanup_pending`

`anchor_root/state/storage_migration.json`

- 仅用于迁移过程中的检查点与恢复
- 迁移成功后可删除，失败时保留

推荐字段：

- `version`
- `status`: `preparing | copying | verifying | committing | retaining_source | rollback_required | failed`
- `txid`
- `source_root`
- `target_root`
- `retained_source_root`
- `retained_source_mode`: `standalone_backup | mixed_root_retained | pending_manual_cleanup`
- `copied_roots`
- `last_processed_path`
- `error_code`
- `error_message`
- `started_at`
- `updated_at`

设计原则：

- `storage_policy` 不维护 `last_migration_*`
- `storage_migration.json` 不承担读写栅栏职责
- `root_state.mode` 仍是全局写保护真源

### 3.5 路径合法性规则

`selected_root` 目标路径必须满足以下规则：

- 必须是绝对路径
- 规范化后不能为空
- 不能位于项目目录内
- 不能位于 `anchor_root/cloudsave`、`anchor_root/state`、`anchor_root/.cloudsave_staging`、`anchor_root/cloudsave_backups` 内
- 不能是 `anchor_root` 的子目录，除非它与 `anchor_root` 完全相同
- 不能是文件
- 必须可创建或可写

推荐的规范化规则：

- `Path.expanduser()`
- `resolve(strict=False)`
- Windows 下做 `normcase`
- 去除尾部分隔符
- 统一比较规范化后的字符串

这样可以避免：

- 把运行时根选进 cloudsave 目录
- 路径大小写差异被误判为不同目录
- symlink / junction 形成循环或伪不同路径

### 3.6 跨进程传播机制

这是 `v0.8` 必须显式设计的一项，否则方案无法稳定落地。

原因：

- 当前 launcher 通过 `multiprocessing` 拉起 `main_server.py` 与 `memory_server.py`
- 这两个模块都在 import 阶段执行 `get_config_manager()`
- Windows `spawn`、Linux/macOS `fork` 的子进程模型不同，仅依赖父进程内存态不可靠

推荐做法：

1. launcher 在用户完成首启选择后，先将 `storage_policy` 落盘
2. 同时把本次启动的存储布局写入环境变量
3. 子进程在 import 重型模块前先执行“从环境变量重载存储布局”

推荐环境变量：

- `NEKO_STORAGE_LAYOUT_VERSION`
- `NEKO_STORAGE_ANCHOR_ROOT`
- `NEKO_STORAGE_RUNTIME_ROOT`
- `NEKO_STORAGE_CLOUDSAVE_STRATEGY`

说明：

- 持久化真源仍然是 `storage_policy.json`
- 环境变量只是“本次启动的一致性桥接”，作用类似 launcher 当前对 `NEKO_*PORT` 的传播方式
- `multiprocessing.Process` 路径以“父进程先更新 `os.environ` + 子进程 import 前主动 reload storage layout”为主
- 若后续有 `subprocess` 拉起的辅助进程、修复器或独立工具，也应显式传入合并后的 `env`，不要假设它一定继承到最新布局

发布态要求：

- 这套机制必须在 `IS_FROZEN=True` 的 Steam/桌面启动器路径中同样生效
- 不能把 storage layout 传播仅实现为 source 模式分支
- 不能依赖 RemoteStorage bundle helper 或 source launch 检测

### 3.7 启动顺序（推荐实现）

推荐新增一个 pre-bootstrap 解析层，例如：

- `resolve_storage_layout_for_launch()`
- `configure_process_storage_layout()`

推荐顺序：

1. launcher 解析平台默认 `anchor_root`
2. 读取 `storage_policy.json`
3. 扫描当前 `selected_root` / 历史 legacy roots，判断是否有可迁移数据
4. 若缺少策略且需要首启交互，则先展示 launcher 阶段存储位置窗口
5. 落盘 `storage_policy`
6. 写入 `NEKO_STORAGE_*` 环境变量
7. 子进程启动前调用 `configure_process_storage_layout()`
8. 然后才允许首次 `get_config_manager()` 绑定路径
9. 进入 `cloud_apply_fence`
10. 执行 `bootstrap_local_cloudsave_environment()`
11. 执行 `CloudSaveManager.import_if_needed(...)`
12. 切回 `root_state.mode=normal`

不推荐的路线：

- 先 import `main_server.py` / `memory_server.py`，再试图在运行中调整根目录

### 3.8 `selected_root` 不可用时的策略

推荐分两类处理：

- 新装且尚未完成首启：允许直接使用默认 `anchor_root` 继续
- 已完成首启且 `selected_root` 失效：进入 `ROOT_MODE_DEFERRED_INIT`

在 `ROOT_MODE_DEFERRED_INIT` 下：

- 允许读取锚点层状态与恢复信息
- 阻断高风险写入
- 前端提示用户：
  - 重试连接原路径
  - 改选新路径并迁移
  - 显式确认“临时切回默认路径”，并将本次恢复动作记录为用户确认后的 `recovered` 选择

禁止静默在后台创建新的默认运行时根继续写入，否则会制造新的分叉数据集。

## 4. 首启状态机与场景化行为

### 4.1 场景状态机

`v0.8` 首启至少要明确处理以下场景：

1. 新装：无 `storage_policy`，无当前数据，亦无 legacy root
2. 升级但已在默认标准目录：无 `storage_policy`，运行时数据当前就在平台默认标准目录（即 `anchor_root`）且已有用户数据
   - 对应 `611ac...` 之后后续待上线代码线的升级来源
3. 升级自旧 Documents/legacy 根：无 `storage_policy`，`anchor_root` 下尚无当前运行时数据，但存在 legacy root
   - 对应 `c28a968...` 当前发布版的升级来源
4. 已配置正常启动：`storage_policy` 存在，`selected_root` 可用
5. 已配置但目标根离线/损坏：`storage_policy` 存在，`selected_root` 不可用
6. 上次迁移中断：`storage_migration.json` 存在且状态未完成

### 4.2 首启默认策略

推荐默认行为：

- 默认选中平台推荐路径，即 `anchor_root`
- 若检测到 legacy root，仅作为“可迁移源”展示，不要先隐式执行一次旧根导入
- 若用户保持默认路径：
  - 只有在用户明确确认继续默认路径之后，才允许沿用当前 phase-0 bootstrap 的导入/修复逻辑
- 若用户选择自定义路径并勾选迁移：
  - 直接从“最佳源根”迁移到该路径，不先导入到 `anchor_root` 再二次迁移

这样可以避免：

- legacy root -> `anchor_root` -> 自定义路径 的双重 copy
- 用户误以为软件在未经确认时已经移动了数据

### 4.3 “最佳源根”选择规则

为保证实现可判定，建议写死优先级：

1. 当前 `selected_root` / `anchor_root` 若已有用户内容，优先它
2. 否则从 `get_legacy_app_root_candidates()` 中选择“最丰富”的旧根
3. 若都没有，则视为新装

“最丰富”的判定可复用当前 `_runtime_root_summary(...).score` 思路。

但在真正复用前必须补一条实现约束：

- `jukebox/` 已被 `v0.8` 定义为运行时真源的一部分，因此 `_runtime_root_summary(...)`、`LEGACY_RUNTIME_DIR_NAMES`、`RUNTIME_ASSET_DIR_NAMES` 或等价评分/复制逻辑都必须把它纳入
- 否则会低估“只有音乐资源而缺少其他大目录”的旧根价值，导致源根选择与复制清单不一致

## 5. 迁移作业设计

### 5.1 迁移对象清单

`v0.8` 迁移只覆盖运行时数据层：

- `config/`
- `memory/`
- `plugins/`
- `live2d/`
- `vrm/`
- `mmd/`
- `workshop/`
- `character_cards/`
- `jukebox/`

显式不迁移：

- `cloudsave/`
- `.cloudsave_staging/`
- `cloudsave_backups/`
- `state/root_state.json`
- `state/storage_policy.json`
- `state/storage_migration.json`

默认不做历史回填、但会在新根懒创建的派生目录：

- `logs/diagnostics/`
- `.rollback_tmp/`

历史根中的可选状态文件处理建议：

- `cloudsave_local_state.json`
- `character_tombstones.json`

可在“legacy -> v0.8 首次迁移”时导入到 `anchor_root/state/`，但不要直接照搬旧 `root_state.json`。

### 5.2 迁移执行顺序

`v0.8` 默认应采用非破坏性的 `clone + verify + rebind + retain` 语义，而不是“迁移成功后立即删除源目录”。

推荐执行顺序：

1. 获取全局互斥锁并进入 `cloud_apply_fence`
2. 写入 `storage_migration.json`，记录 `source_root`、`target_root`、`txid`、当前阶段
3. 在 `target_root.parent` 下创建 `stage_root`，保证最终提交是同卷 rename/swap，而不是跨卷移动
4. 复制运行时目录到 `stage_root`
5. 完整性校验
6. 以 `safe_swap_directories()` 语义提交：
   - 不把单次 `os.replace(stage_root, target_root)` 视为“跨平台覆盖已有非空目录”的万能方案
   - 若 `target_root` 已存在，先 rename 到 `backup_slot`，再 rename `stage_root -> target_root`
   - 若切换失败且 `backup_slot` 存在，优先恢复旧 `target_root`
7. 更新 `storage_policy.selected_root`
8. 更新 `root_state.current_root`、`last_known_good_root`
9. 进入“保留旧数据”阶段，而不是默认删除 `source_root`
   - 先将 `storage_migration.status` 置为 `retaining_source`
   - 默认不得直接对旧根调用 `os.remove(...)`、`shutil.rmtree(...)`
   - 若 `source_root != anchor_root`，通常应将其视为“独立历史运行时根”；即使其中还保留旧版 `cloudsave/` / `state/`，这些也只是历史残留，不应单独阻止整根 backup rename
   - 仅当 `source_root == anchor_root`，也就是它仍是本次迁移后需要继续使用的 live anchor 时，才不能整根 rename；此时应仅把可迁移运行时目录收拢到 `anchor_root/legacy_runtime_backups/<txid>/`，或在失败时原地保留并记录待手动清理
10. 将保留结果写回 `root_state.last_migration_backup` / `legacy_cleanup_pending`
11. 清理检查点、阶段目录与已消费的 target-side backup

这里的时序约束需要明确：

- `selected_root` 的提交点发生在“新根已验证并完成 rebind”之后，而不是等旧数据保留完全结束之后
- “保留旧数据”属于提交后的 follow-up 阶段；其失败应收敛为 `legacy_cleanup_pending` 或恢复态告警，而不是反向否决已经成功的新根绑定

如果是“现有默认根 -> 自定义根”迁移：

- `anchor_root/state/` 与 `anchor_root/cloudsave/` 保持原地不动
- 只迁移可移动运行时目录

说明：

- Windows 下若 `target_root` 是非空目录，不能依赖 `os.replace(stage_root, target_root)` 直接覆盖成功
- `stage_root` 放在 `target_root.parent` 的主要价值，不只是避免 `EXDEV`，也是为了让“先备份旧目录、再切换新目录”的恢复路径保持简单稳定
- 源根保留目录名不建议写死成 `A_v0.7_backup` 一类版本耦合命名；更稳妥的是通用 `legacy_backup` 语义，并追加时间戳或 `txid`

### 5.3 预检项

迁移前必须检查：

- 目标路径合法性
- 目标路径及其阶段目录的深层可写性，而不是只探测根目录一次
- 磁盘剩余空间
- 源路径是否可读
- 源根是“独立运行时根”还是“混合锚点根”，以便决定后续保留/清理策略
- 待迁移树中是否存在 `symlink` / `junction` / 其他 reparse point
- `target_root` 不含冲突目录或已明确允许覆盖
- 当前无活跃写栅栏冲突
- 若目标路径位于 OneDrive / iCloud / Google Drive 等同步盘或受管目录，给出强警告，但不默认作为硬阻断

对 SQLite 文件：

- 延续当前 `time_indexed.db` 的 shadow copy / backup 思路
- 迁移必须发生在 memory server 打开句柄之前

对 `symlink` / `junction` 的 `v0.8` 保守策略建议为：

- 发现后默认阻止自动迁移该根并提示用户手动处理
- `v0.8` 不承诺“保留链接拓扑”或“自动穿透复制链接目标”
- 这样比静默解引用拷贝更安全，也更符合当前项目尚未建立链接对象复制契约的现实

### 5.4 失败与恢复

失败时：

- 不更新 `storage_policy.selected_root`
- 旧 `last_known_good_root` 保持不变
- `storage_migration.json` 保留
- 明确记录 `error_code`、失败路径、阶段信息
- 若已明确进入“需要回滚但尚未回滚完成”的提交中断态，建议写 `status = rollback_required`
- 若已明确失败并等待用户重试/人工处理，建议写 `status = failed`
- 对 `PermissionError`、Windows sharing violation、短暂 `OSError` 仅做有限次数的指数退避重试；超过上限再判定失败
- 若提交阶段已经生成 `backup_slot`，则失败恢复优先尝试把旧目录 restore 回 `target_root`
- 若“新根已完成 rebind，但旧根重命名/收拢失败”，应将其标记为 `legacy_cleanup_pending` 警告而不是整体回滚；只要新根校验通过，应用应继续从新根启动

下次启动：

- 若 `storage_migration.status == preparing`
  - 视为迁移尚未进入可提交阶段；清理残留阶段目录后返回“可重试/可取消”的首启恢复流程
- 若 `storage_migration.status` 为 `copying` / `verifying` / `committing` / `rollback_required`
  - 进入恢复分支
- 若 `storage_migration.status == retaining_source`
  - 若 `storage_policy.selected_root` 已经指向 `target_root`，则视为“新根已生效，但旧数据整理未完成”
  - 此时不回滚到旧根，而是补写 `legacy_cleanup_pending`、暴露手动清理入口，并清理残留阶段目录
- 若 `storage_migration.status == failed`
  - 保留失败信息并进入“可见、可重试”的恢复入口；若无法安全判定当前绑定状态，则进入 `ROOT_MODE_DEFERRED_INIT`
- 若可判定可回滚
  - 自动回滚并进入 `ROOT_MODE_DEFERRED_INIT`
- 若不可判定
  - 直接进入 `ROOT_MODE_DEFERRED_INIT`

### 5.5 迁移后提交条件

只有同时满足以下条件，才允许提交 `selected_root`：

- 目标目录结构完整
- 关键配置文件可读
- `characters.json` 可正常解析
- `user_preferences.json`、`core_config.json` 可正常解析
- `memory/` 目录可访问
- `path_fingerprint` 与落盘路径一致；其最低语义应是“规范化绝对路径一致”，平台可用时可附带 volume/device identity 辅助字段

提交后的后续要求：

- 若存在保留的旧数据，应在 `selected_root` 已生效后尽快写入 `last_migration_backup` 或等价元数据，并保证这些路径不会被重新识别为运行时真源
- 若该元数据写入或旧根收拢失败，不回滚已生效的新根绑定，而是标记 `legacy_cleanup_pending` 并暴露可见恢复/清理入口

提交完成后还应补一项兼容性收尾：

- 在 `anchor_root` 下生成 `_README_MIGRATED.txt` 或 `_数据已迁移说明.txt`
- 若旧运行时根被保留为空壳目录，也可在旧根再放一份同类提示文件
- 文案至少说明：运行时数据已迁移到哪个 `selected_root`、`v0.7` 及更早版本不识别 `storage_policy`、若要回退旧版本需先手动拷回数据

这不是运行时真源，但能显著降低“用户回退旧版本后误以为数据丢失”的风险。

### 5.6 源根保留决策表

| 场景 | 判定条件 | 默认保留动作 | 手动清理单元 | 失败后行为 |
| --- | --- | --- | --- | --- |
| 独立运行时根 | `source_root != anchor_root`；旧版单根目录中遗留的 `cloudsave/` / `state/` 不改变该判定 | best-effort 整根 rename 为同级 `legacy_backup` 目录 | 整个保留目录 | 保留 `selected_root = target_root`，标记 `legacy_cleanup_pending` |
| 混合锚点根 | `source_root == anchor_root`，即该根仍是迁移后继续生效的 live anchor | 不整根 rename；仅移动运行时目录到 `anchor_root/legacy_runtime_backups/<txid>/`，失败则原地保留 | 受跟踪的运行时目录集合 | 保留 `selected_root = target_root`，标记 `legacy_cleanup_pending` |
| 存在 symlink / junction / reparse point | 预检发现链接对象或循环风险 | 阻止自动迁移 | 无 | 直接转人工处理，不进入自动清理流程 |

### 5.7 推荐伪代码骨架

推荐把迁移主流程收敛成单个明确的 orchestration 函数，语义类似：

```python
def migrate_runtime_root(source_root, target_root, anchor_root):
    with cloud_apply_fence(...):
        write_storage_migration(status="preparing", source_root=source_root, target_root=target_root)

        stage_root = prepare_stage_root(target_root)
        copy_runtime_tree(source_root, stage_root)
        write_storage_migration(status="verifying", last_processed_path="...")
        verify_staged_runtime(stage_root)

        write_storage_migration(status="committing")
        commit_target_root(stage_root, target_root)

        persist_storage_policy(selected_root=target_root)
        persist_root_state(current_root=target_root, last_known_good_root=target_root)

        write_storage_migration(status="retaining_source")
        retain_result = retain_source_runtime_data(
            source_root=source_root,
            anchor_root=anchor_root,
            txid=...,
        )

        persist_retained_backup_metadata(retain_result)
        write_migrated_readme(anchor_root=anchor_root, target_root=target_root, backup=retain_result)
        clear_storage_migration_checkpoint()
        return retain_result
```

异常分支建议固定为：

- 若失败发生在 `persist_storage_policy(selected_root=target_root)` 之前：按“迁移失败”处理，可回滚 target-side swap
- 若失败发生在 `selected_root` 已切到 `target_root` 之后：按“迁移成功但旧数据保留/整理未完成”处理，不再把整单迁移回滚成旧根

## 6. 代码改造清单

### 6.1 新增模块

建议新增：

- `utils/storage_policy.py`

建议职责：

### 7.1 v0.8 必须收敛的读取点

以下配置读取必须改成“只读运行时真源”，不能再依赖 `get_config_path()` 回退：

- `characters.json`
- `core_config.json`
- `user_preferences.json`
- `tutorial_prompt.json`
- `autostart_prompt_state.json`
- `holiday_consumption.json`

最少要覆盖的模块：

- `ConfigManager.load_characters()`
- `utils/config_manager.py` 中 core config 环境加载逻辑
- `utils/preferences.py`
- `utils/tutorial_prompt_state.py`
- `utils/autostart_prompt_state.py`
- `utils/holiday_cache.py`

补充约束：

- 被保留的 `legacy_backup` / `legacy_runtime_backups/*` 只能作为诊断与手动清理对象，不能重新进入运行时读写真源候选集
- 即使 `selected_root` 离线，也不能静默回退去读取这些保留备份目录继续工作
- `logs/diagnostics/` 与 `.rollback_tmp/` 虽然继续跟随 `selected_root`，但在一致性语义上属于可重建派生目录，不纳入“历史内容必须迁移成功”的核心判定集

### 7.2 启动日志与诊断

启动日志至少记录：

- `anchor_root`
- `selected_root`
- `effective_runtime_root`
- `effective_cloudsave_root`
- `storage_policy_path`
- `root_state.mode`
- `core_config.json` 实际读取路径

记录方式约束：

- 以上属于系统事件与本地诊断，应通过统一 logger backend 记录，而不是绕开 `utils/logger_config.py` 自建日志链路
- 这些路径诊断只允许停留在本机日志/本机诊断接口，不应被写入 `cloudsave/` 快照、远端 bundle、遥测或其他跨设备同步载体

建议新增：

- `path_fingerprint`
- `selection_source`
- `storage_policy.version`
- `storage_layout_source`: `env | policy | fallback`
- `last_migration_backup`
- `legacy_cleanup_pending`

`path_fingerprint` 建议不要只保留成一个不可解释的 hash。更合适的诊断口径是：

- 最少包含规范化绝对路径
- 平台可用时附带 `volume_hint` / `device_id` 一类的 best-effort 辅助标识
- 若外置盘重新挂载后仅盘符变化，可先用辅助标识帮助判断是否还是同一存储
- 若仍无法确认，直接进入 `ROOT_MODE_DEFERRED_INIT`，不要静默把离线路径当成新空目录继续写

发现路径不一致时：

- 记录 `ERROR`
- 返回前端可见状态
- 必要时进入 `ROOT_MODE_DEFERRED_INIT`

若存在保留备份，建议输出类似诊断：

- `Legacy data retained at {path}; runtime is now bound to {selected_root}`

### 7.3 Memory 子系统说明

当前 memory 主链路仍然是：

- 角色目录 JSON 文件
- `time_indexed.db`（SQLite）

路径主要由 `config_manager.memory_dir` 推导。

当前未发现“需要在迁移时对持久化向量索引绝对路径做 rebase”的主链路，因此：

- `v0.8` 不把向量库路径重写列为必做项
- 但必须确保迁移发生在 memory server 打开句柄之前

## 8. 前端与接口设计

### 8.1 必做：launcher 阶段交互

首启迁移窗口建议使用 launcher 事件桥接或原生窗口，不建议依赖主服务首屏路由。

建议的前端事件/命令能力：

- `storage_location_required`
- `storage_location_apply`
- `storage_location_progress`
- `storage_location_failed`
- `storage_location_ready`

迁移 copy / verify / commit 不得阻塞 launcher 的 UI 渲染线程，建议放在后台 worker 线程或独立进程执行。

`storage_location_progress` 不应只有“正在迁移”文案，至少应包含：

- `stage`
- `current_path` 或当前文件名
- `bytes_done`
- `bytes_total`
- 或 `files_done` / `files_total`

确认迁移时建议增加一个默认不勾选的兜底项：

- `[ ] 迁移完成后自动清理旧路径下的源文件以节省空间（不建议）`

约束：

- 默认值必须是 `false`
- 只有当预检确认源根是“独立运行时根”且 cleanup 目标可判定时，才允许暴露该选项
- 若源根是 `anchor_root` 或混合锚点根，应改为只展示“迁移后保留旧数据，稍后在设置中手动清理”的说明，而不是提供危险的自动删除勾选框

迁移成功提示建议同时给出：

- `打开当前数据目录`
- `打开旧数据备份目录`
- `稍后去设置中清理`

推荐双入口：

- 首要入口放在“迁移成功”提示里，帮助用户立刻建立“新目录已生效、旧数据仍保留”的安全感
- 持久入口放在“设置-存储”页，承载稍后手动清理与空间回收

建议 `storage_location_required` 至少包含：

- `anchor_root`
- `recommended_runtime_root`
- `legacy_sources`
- `has_existing_runtime_content`
- `cloudsave_root`

发布态要求：

- 这套交互在 `frozen + Steam 启动` 和 `frozen + 桌面启动器启动` 下必须可用
- 不能要求用户先打开浏览器页面才能完成首次路径选择
- 若桌面壳不可用，launcher 只有在“新装、无 `storage_policy`、无现有用户数据、也无需迁移”的场景下，才可回退到“默认路径继续”
- 其余场景必须进入显式待恢复/待处理态，不能因为 UI 不可用就直接改写到新的默认运行时根
- 不能把首启选择依赖在 Steam Overlay、source helper 或手动打开某个 Web 页上

### 8.2 启动后的状态 API

主服务启动后，可新增只读或运维类接口：

- `GET /api/storage/location/status`
  - 返回当前绑定状态、`root_mode`、`first_run_completed`、`legacy_cleanup_pending`，以及前端确有必要展示的路径信息
  - 若后续需要暴露绝对路径，建议区分“状态摘要”和“详细诊断”两档；常规状态接口不应无约束扩散本机路径细节
- `POST /api/storage/location/apply`
  - 若后续支持“启动后手动迁移”入口，可接受 `auto_cleanup: bool = false`
  - 但它不能承担首次启动选根主流程
  - 它也不能直接对当前存活进程做根路径热切换；更合理的语义是“写入待执行迁移请求/策略变更意图，并要求受控重启后由 launcher/pre-bootstrap 执行”
  - 返回值应显式表明 `restart_required` / `handoff_to_launcher`，避免实现者误解为可在主服务内原地 apply
- `GET /api/storage/legacy_data`
  - 返回是否存在旧备份、备份路径、估算占用空间、是否允许一键删除
- `DELETE /api/storage/legacy_data`
  - 执行物理删除
  - 请求体不应直接接受任意绝对路径，建议仅删除受跟踪的 backup id / token
- `GET /api/storage/location/diagnostics`
  - 返回路径解析与恢复状态
  - 详细绝对路径、`path_fingerprint`、`volume_hint` / `device_id` 一类字段更适合放在这里，而不是默认混入通用状态接口

不建议把首次选根主逻辑完全依赖于：

- `POST /api/storage/location/apply`
- `GET /api/storage/location/migration_progress`

因为这会把关键路径放到“路径已经绑定之后”。

### 8.3 i18n

保留原稿中的 `storageMigration.*` key 设计即可，但语义上应改成“launcher 阶段弹窗”。

还需要补齐以下 i18n 约束：

- 8.1 中新增的 checkbox、成功页按钮、恢复态说明、失败提示、`restart_required` / `handoff_to_launcher` 提示，都必须补全 locale key，不能把中文文案硬写在实现里
- `storage_location_*` 事件/接口尽量传机器可判定字段（如 `stage`、`error_code`、`selection_source`），由前端按 locale key 渲染用户可见文本
- 若最终由 JS 渲染，仍应遵守项目现有 `window.t('key')` + 中文回退约定

## 10. 测试矩阵

### 10.1 必增自动化测试

- 首启无策略文件时的默认路径确认
- 首启选择自定义 `selected_root` 并迁移成功
- legacy root 直接迁移到自定义路径
- `selected_root` 离线时进入 `ROOT_MODE_DEFERRED_INIT`
- 迁移失败时 `storage_policy` 不更新、旧根保持有效
- 方案 A 下 `cloudsave_root` 始终固定在 `anchor_root`
- launcher 在 `get_config_manager()` 首次绑定前完成策略解析
- 子进程在 import 前通过 `NEKO_STORAGE_*` 命中正确布局
- `target_root` 已存在且为非空目录时，模拟 Windows 目录覆盖限制的 backup + swap 提交/恢复路径仍然正确
- copy / verify / commit 阶段遇到短暂锁定时，会按有限退避策略重试，而不是立刻把迁移永久打成失败
- 发现 `symlink` / `junction` / `reparse point` 时，迁移会进入显式受限路径，而不是静默穿透复制
- 迁移成功后会生成降级提示 README，明确旧版本不可直接识别新布局
- launcher 迁移进度事件会持续上报 `stage` 与字节级或文件级进度，而不是长时间静默
- 默认迁移成功后不会自动删除源目录；若有 retained backup，会正确记录 `legacy_cleanup_pending`
- 当 `source_root == anchor_root` 时，不会错误尝试整根 rename 锚点目录
- 旧数据保留阶段若 rename/收拢失败，只会留下可见告警，不会阻断新根启动
- 若进程中断在 `retaining_source` 阶段，下次启动不会误回滚到旧根，而是收敛为 `legacy_cleanup_pending`
- `GET /api/storage/legacy_data` 能返回备份体积与可清理状态，`DELETE /api/storage/legacy_data` 只能删除受跟踪备份
- `utils/preferences.py` 等带模块级缓存的调用点在新根下命中正确路径
- 旧根导入辅助函数在双根模型下不再复制 `cloudsave/`
- `IS_FROZEN=True` 时 launcher 仍走相同的 storage layout 解析和 bootstrap 顺序
- 发布态 `not_source_launch` 语义保持不变，不因存储迁移功能误触发 source-only helper

### 10.2 发布态验收矩阵

- `frozen + Steam 启动`
  - 必须正常完成 launcher phase-0
  - 必须命中正确 `storage_policy`
  - 必须继续使用 `anchor_root/cloudsave`

## 12. 对当前问题表述的校正

以下内容用于校正文档评审、代码 review 或验收阶段对“当前问题”的表述口径，避免把“文档硬约束”与“实现偏差但尚未被文档明文规定”的问题混在一起。

本节中的“旧 `Documents`”是相对于第 3 章 `v0.8` 目标布局或当前工作区模拟出来的新运行根而言的 legacy root，不等于第 1.1-1.2 节里历史发布侧默认主根本身。

### 12.1 关于“日志目录仍落在旧 Documents”

这条表述需要收窄，不宜直接写成：

- “整个日志目录仍然落在旧 `Documents`，因此违反 `v0.8` 设计”

更准确的口径应是：

- 当前文档**明确约束**的是 `logs/diagnostics/` 必须跟随 `selected_root / app_docs_dir`，并在当前根下懒创建；
- 当前文档**尚未单独强制规定**“全部通用日志文件都必须迁入 `selected_root`”；
- 因此，若通用 logger 仍优先写旧 `Documents`，它本身更适合作为“实现偏差 / 后续需统一的方向”记录，而不是单独作为文档违约项；
- 但若这种 logger 基目录选择进一步导致 `logs/diagnostics/` 与其他诊断输出分裂到旧根和新根两处，则这部分已经构成本文档范围内的真实问题。

换言之：

- “通用日志仍在旧 `Documents`”本身：属于**需进一步收敛的实现现状**
- “诊断目录发生分裂”：属于**本文档已覆盖的真实问题**

### 12.2 关于“诊断目录存在分裂”

这条问题是真实存在的，且属于本文档已显式覆盖的范围。

文档已有约束：

- `logs/diagnostics/` 继续跟随 `selected_root` / `app_docs_dir`
- `logs/diagnostics/` 必须在当前 `selected_root` 下懒创建，而不是在 import 阶段永久绑定旧根

因此，如果当前实现中：

- 一部分诊断文件跟随 logger 基目录
- 另一部分诊断文件直接跟随 `app_docs_dir / "logs" / "diagnostics"`

则应明确记为：

- 真实问题
- `v0.8` 前必须消除的路径一致性问题

### 12.3 关于 `characters.json` / `core_config.json` 的回退读取

这条问题是真实存在的，而且文档正文已将其列为必须收敛项。

更准确的描述应是：

- 当前风险点不一定表现为“直接读回旧根”；
- 当前实现里的直接回退目标首先是项目内 `config/`；
- 但在 `storage_policy` / 双根模型下，任何这类“非运行时真源回退”都会重新引入读写分裂风险，因此必须在 `v0.8` 中移除。

建议统一口径为：

- “当前仍保留非运行时真源回退读取语义；在 `v0.8` 双根模型下，这是不可接受的，需要改成只读运行时真源”

### 12.4 关于 `utils/preferences.py` 的 import 时路径缓存

这条问题是真实存在的，而且属于本文档已经点名的高风险点。

推荐固定表述为：

- `utils/preferences.py` 仍在模块 import 阶段缓存 `ConfigManager` 与活动路径；
- 当前单根基线下未必每次都会出错，但在 `launcher pre-bootstrap` 解析 `storage_policy`、子进程 import 前注入布局、以及后续双根实现下，这类缓存极易提前绑定旧路径；
- 因此这不是“可选优化”，而是 `v0.8` 必改项。

### 12.5 关于 autostart / tutorial / holiday 状态文件仍使用 `get_config_path()`

这条问题也是真实存在的，应继续保留为文档中的明确改造项。

推荐表述为：

- 当前这些状态文件在现有基线下通常仍能工作；
- 但它们本质上仍依赖 `get_config_path()` 的回退语义，不满足“只读运行时真源”的 `v0.8` 约束；
- 因此应按“真实存在的收敛缺口”记录，而不是仅写成“未来可能优化”。

### 12.6 建议在评审与验收中使用的统一分类

为避免后续讨论混乱，建议把相关问题分成两类：

- 文档已覆盖且可直接判定为真实问题：
  - `logs/diagnostics/` 分裂
  - `characters.json` / `core_config.json` 非真源回退读取
  - `utils/preferences.py` import 时路径缓存
  - autostart / tutorial / holiday 状态文件仍依赖 `get_config_path()`
- 需要收窄表述、避免过度下结论的问题：
  - 通用日志目录仍优先写旧 `Documents`

对于后一类，建议统一写成：

- “当前实现尚未完成日志基目录收敛；若它影响 `logs/diagnostics/` 一致性，则按真实问题处理；否则作为后续实现统一项继续跟踪”
