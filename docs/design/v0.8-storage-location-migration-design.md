# v0.8 存储位置首启迁移设计（首次正常进入，应用内选择，必要时关闭并自动重启）

> 目标：在 `v0.8` 引入“存储位置选择 + 可选迁移”，同时不破坏现有 cloudsave 启动链路。
>
> 本版是在“应用内选择、关闭当前实例后迁移、迁移完成自动重启”这条新流程下，对旧版设计文档做的一次完整补回与重整：
>
> - 保留旧版中仍然成立的发布态约束、跨平台目录事实、cloudsave 约束、多进程传播、读写收敛、测试与验收要求
> - 删除旧版中已经不再采用的“启动前阻断式首启选择”主流程
> - 把所有仍然必要的工程约束改写到新流程语义之下

---

## 0. 关联文档与总体约束

本设计与以下文档保持一致：

- `docs/design/cloud-save-sync-optimization-plan.md`
- `docs/design/v0.8-storage-location-migration-implementation-guide.md`
- `docs/design/v0.8-storage-location-migration-window-display-design.md`

当前需要先明确两件事。

第一，当前仓库里曾经存在一条“启动早期 probe gate / 首轮 setup 骨架”的探索路线，但它不再是本设计的正式用户流程。

第二，虽然首轮选择已经改成“应用内完成”，但真正的切根与迁移仍然必须发生在：

- 当前实例已经退出之后
- 新实例正式拉起之前

这意味着：

- 首轮交互时机变了
- 但路径绑定时机、读写一致性、cloudsave 锚点、多进程布局传播这些硬约束并没有消失

### 0.1 本版定下来的用户流程

本版采用的正式流程是：

1. 第一次启动时，应用先正常打开。
2. 前端在应用内立即显示存储位置选择弹窗或首屏覆盖页。
3. 用户在当前会话中完成选择。
4. 若选择结果与当前正在使用的路径一致：
   - 直接保存选择；
   - 关闭弹窗；
   - 当前会话继续运行。
5. 若选择结果与当前路径不同：
   - 只保存迁移意图，不在当前会话里切根；
   - 前端明确提示“将关闭当前实例并迁移数据”；
   - 用户确认后，当前实例受控关闭；
   - 迁移逻辑在实例关闭后、新实例拉起前执行；
   - 成功后自动重启进入新路径。

### 0.2 本版仍然必须遵守的总体约束

- 不破坏 Steam Auto-Cloud 现有主链路。
- 不支持运行中热切换根目录。
- 不在 `v0.8` 一次性重写所有历史调用点，只覆盖启动、迁移、关键读写链路。
- 首次选择可以放到应用内，但真正迁移执行必须早于新实例里任何服务进程首次 `get_config_manager()`。
- 发布态主验收路径仍是 `frozen + Steam 启动` 与 `frozen + 桌面启动器启动`，不是源码模式。
- 默认不自动删除旧数据。
- 迁移失败时必须保留旧根可恢复，不能把策略写成半有效状态。

### 0.3 发布态启动口径

文档必须区分以下几种启动方式，否则“是否正常”会被混淆成不同问题。

正式发布主路径：

- `frozen + Steam 启动`
- `frozen + 桌面启动器启动`

开发/兼容路径：

- `source + launcher.py`
- 手动三服务模式：`main_server.py` / `memory_server.py`

术语约定：

- “Steam 启动”：指 Steam 客户端拉起打包后的 launcher / 桌面应用入口
- “桌面启动器启动”：指用户直接双击打包后的桌面应用入口
- “源码 launcher 启动”：指开发环境执行 `python launcher.py`

本文若写“桌面端启动”，默认指“打包后的桌面启动器启动”。

### 0.4 对“正式发布后能否正常”的更新结论

在本版流程下，正式发布后通过 Steam 或桌面启动器启动，理论上可以正常工作，但必须同时满足以下条件：

1. 第一轮只负责在应用内收集用户意图，不在当前会话热切换根目录。
2. 如果需要迁移，新实例启动前必须先读取 `storage_migration` 检查点。
3. 新实例在任何服务 import `main_server.py` / `memory_server.py` 之前，必须先完成存储布局解析。
4. `cloudsave_root` 在发布态继续固定到 `anchor_root`。
5. 迁移成功后由系统自动重启，不要求用户手工再次打开应用。

如果这些条件没有落地，最容易出现的问题是：

- 首轮界面看起来能选路径，但当前会话和下次会话对“当前根”理解不一致
- 新实例服务进程先绑到旧根，再去读迁移检查点，导致迁移失效
- Steam Auto-Cloud 仍同步旧锚点，但运行时目录已经漂移，造成路径混乱
- 迁移失败后 `storage_policy` / `root_state` / 实际文件状态不一致

## 1. 现状基线（区分历史发布版与后续代码线）

本章分成两层口径：

- `1.1-1.2`：按历史发布基线回答“旧版本原本把数据放在哪里”
- `1.3-1.5`：按后续代码线回答“当前仓库可直接复用什么基础设施”

### 1.1 历史发布侧根目录解析与启动顺序

在历史发布基线里，`ConfigManager.__init__()` 会直接解析文档目录并立即绑定：

- `app_docs_dir`
- `config_dir`
- `memory_dir`
- `plugins_dir`
- `live2d_dir`
- `vrm_dir`
- `mmd_dir`
- `workshop_dir`
- `chara_dir`

也就是说，当时使用的是“初始化即绑定”的单根模型，不存在双根布局。

历史代码的真实解析顺序是：

1. Windows 主候选仍然是 `Documents`
   - 系统 API 返回的“我的文档”
   - 同盘符下的 `文档` / `Documents` / `My Documents`
   - 注册表 `Personal`
   - 同盘符根下枚举出的 `documents` / `文档` / `my documents`
   - `~/Documents`
   - `~/文档`
2. Windows 仅在 `Documents` 不可写时，才回退到 `%LOCALAPPDATA%`
3. 若仍不可用，再回退到：
   - 打包态：`Path(sys.executable).parent`
   - 开发态：`Path.cwd()`
4. 若仍失败，最终兜底仍是 `Path.cwd()`

平台补充：

- macOS 历史基线是 `~/Documents`，再退到 `cwd`
- Linux 历史基线是 `XDG_DOCUMENTS_DIR`（若设置）或 `~/Documents`，再退到 `cwd`

因此旧版本对应的原始存储位置应更准确地写成：

- 默认主根是 `Documents/N.E.K.O`
- Windows 在 `Documents` 不可写或受 CFA/反勒索防护影响时，可能回退到 `%LOCALAPPDATA%/N.E.K.O`
- 更低优先级还可能是打包目录或 `cwd` 下的 `N.E.K.O`

补充说明：

- Windows 历史代码已经有 CFA 兼容逻辑：若第一个可读候选与最终可写候选不同，会保留 `_readable_docs_dir`
- 这意味着 Windows 上的历史“原位置”不仅是默认主根，也可能在某些场景下成为只读来源
- 当时 `get_config_manager()` 已经是单例，因此首次绑定的根目录会被后续模块沿用

### 1.2 历史发布侧实际存储清单

历史发布侧默认把以下内容绑定在单一 `app_docs_dir` 下：

- `config/`
- `memory/`
- `plugins/`
- `live2d/`
- `vrm/`
- `vrm/animation/`
- `mmd/`
- `mmd/animation/`
- `workshop/`
- `character_cards/`
- `jukebox/`
- `jukebox/songs/`
- `jukebox/actions/`
- `logs/`
- `logs/diagnostics/`

其中还保留两类“非运行时主根，但会参与读取或初始化”的路径：

- 项目内回退源：`project_root/config/`、`project_root/memory/store/`
- 运行时外部引用：Steam Workshop 路径、`user_mod_folder`、`user_workshop_folder`

需要特别排除在“旧版原始存储位置”描述之外的目录：

- `cloudsave/`
- `.cloudsave_staging/`
- `cloudsave_backups/`
- `state/root_state.json`
- `state/cloudsave_local_state.json`
- `state/character_tombstones.json`

这些都是后续代码线才逐步出现的目录或机制。

### 1.3 后续代码线已具备的迁移与保护能力

当前工作区已经具备一批可复用基础设施，它们虽然最初并不是为“应用内选择后重启迁移”量身设计，但仍然值得直接继承：

- `import_legacy_runtime_root_if_needed()`
- `bootstrap_local_cloudsave_environment()`
- `cloud_apply_fence()`
- `ROOT_MODE_DEFERRED_INIT`
- `root_state.current_root`
- `root_state.last_known_good_root`
- `root_state.last_migration_source`
- `root_state.last_migration_result`
- `root_state.last_migration_backup`
- `root_state.last_successful_boot_at`

更具体地说：

- 旧根导入不是单纯覆盖，而是带 seeded target 修复和配置 merge 的导入流程
- `bootstrap_local_cloudsave_environment()` 能确保 `cloudsave/`、`state/`、manifest 以及 stale mode 恢复
- `cloud_apply_fence()` 提供跨进程互斥和 `root_state.mode` 写栅栏

这说明真正要补的不是“如何搬文件”，而是：

- 应用内选择后的状态衔接
- 当前实例关闭与下次实例迁移之间的 orchestration
- 多进程统一读到同一份最终布局

### 1.4 后续代码线读写路径语义

当前仍存在两套配置路径 API：

- `get_runtime_config_path(filename)`：始终返回运行时真源路径
- `get_config_path(filename)`：若运行时文件不存在，会回退到项目目录 `config/`

已经相对收敛的链路：

- `ConfigManager.save_characters()`
- `ConfigManager.save_workshop_config()`
- `ConfigManager.save_json_config()`
- `memory/recent.py`、`memory_server.py`、`main_routers/config_router.py` 中 `core_config.json` 的关键读取点
- `utils.preferences` 的关键写路径

仍然保留回退读取语义、需要在 `v0.8` 重点治理的链路：

- `ConfigManager.load_characters()`
- `utils/config_manager.py` 中 `core_config.json` 的环境读取逻辑
- `utils/preferences.py` 的部分读取逻辑
- `utils/autostart_prompt_state.py`
- `utils/tutorial_prompt_state.py`
- `utils/holiday_cache.py`

结论：

- `v0.8` 最大的“迁移后读写分裂”风险，不是保存路径，而是这些隐式回退读取

### 1.5 后续代码线测试基线

当前已有测试已经覆盖一部分关键事实：

- `tests/unit/test_cloudsave_config_manager.py`
  - 平台默认根目录
  - `cloudsave/`、`state/`、路径属性与基础目录创建
- `tests/unit/test_cloudsave_runtime.py`
  - bootstrap
  - 旧根目录导入/修复
  - stale mode 恢复
  - `cloud_apply_fence` 锁与写阻断
- `tests/unit/test_cloudsave_startup_flow.py`
  - 启动阶段 bootstrap 与 import 的调用顺序
- `tests/unit/test_steamworks_loader_paths.py`
  - source 模式下项目根定位不依赖 `cwd`

说明：

- 这些测试当前仍大量假设 `cloudsave_root == app_docs_dir / "cloudsave"`
- 一旦引入 `anchor_root / selected_root`，相关测试必须成组更新，不能只补新增测试

## 2. 针对新流程的差距评估

### 2.1 旧版里仍然成立的核心风险

即使首轮选择改到应用内，以下风险仍然成立：

- `ConfigManager` 是单例，首次绑定路径后难以安全重绑
- 多个模块存在 import 阶段路径缓存
- `cloudsave` 与运行时根历史上长期同根
- 多进程服务启动时的路径一致性不能靠“父进程内存态”保证

因此新流程虽然改变了首轮交互时机，但没有改变这些工程难点。

### 2.2 旧版里需要改写而不是直接删除的结论

旧版曾经把“首启选择必须早于首次 `get_config_manager()`”作为总前提。

在新流程下应改写为：

- 首轮“选择动作”可以发生在应用已运行的当前会话中
- 但真正“迁移执行 + 新布局生效”必须发生在下一次实例启动早期，且早于新的服务进程首次 `get_config_manager()`

也就是说：

- 首轮不再要求 pre-bootstrap
- 第二轮迁移执行仍然必须具备 pre-bootstrap 约束

### 2.3 旧版对单根假设的警告仍然有效

当前不仅 `ConfigManager` 使用单根模型，下列逻辑也默认 `cloudsave` 与运行时根同根：

- `ConfigManager.cloudsave_dir`
- `ConfigManager.local_state_dir`
- 锁文件位置与 mutex 命名
- `LEGACY_RUNTIME_DIR_NAMES`
- `_copy_runtime_root_entries()`
- `_replace_runtime_root()`
- 大量测试断言

因此若 `v0.8` 确认采用“固定锚点 + 可迁移运行时根”模型，就必须显式拆分：

- 哪些目录属于 `anchor_root`
- 哪些目录属于 `selected_root`
- 哪些辅助函数只能复用于“运行时目录搬运”，不能继续假设锚点同根

### 2.4 API 设计需要拆成两段

旧版里“首次交互”和“迁移执行”很多逻辑都围绕启动前桥接设计。

在新流程下，API 必须拆成两段：

第一段：当前会话内的选择与确认

- `GET /api/storage/location/bootstrap`
- `POST /api/storage/location/select`
- `POST /api/storage/location/restart`

第二段：关闭后的迁移与重启状态

- `GET /api/storage/location/status`
- 可选的 WebSocket / SSE / 本地进度桥

### 2.5 fallback 策略仍需保守

如果用户曾经选择过自定义路径，后续该路径离线或损坏，不能静默退回默认根继续写入，否则会制造新分叉。

更保守的策略仍然是：

- 进入 `ROOT_MODE_DEFERRED_INIT`
- 给出显式恢复、重试、改选路径或用户确认的回退动作
- 在用户完成恢复决策前阻断高风险写入

## 3. 推荐的 v0.8 目标方案

### 3.1 决策门 A：固定锚点目录

推荐继续采用固定锚点目录 `anchor_root`：

- Windows：`%LOCALAPPDATA%/N.E.K.O`
- macOS：`~/Library/Application Support/N.E.K.O`
- Linux：`$XDG_DATA_HOME/N.E.K.O`，未设置则 `~/.local/share/N.E.K.O`

这里的“平台标准应用数据目录”指的是对齐后续代码线已经出现的标准应用数据目录优先逻辑，而不是回退到历史发布版的 `Documents-first` 模型。

### 3.2 决策门 B：cloudsave 与运行时根关系

推荐继续采用“固定锚点 cloudsave”：

- `cloudsave_root = anchor_root / "cloudsave"`
- `selected_root` 仅承载用户运行时数据
- 不让 `cloudsave_root` 跟随 `selected_root`

理由：

- 与现有 Steam Auto-Cloud 规则最兼容
- 跨设备行为更可预测
- 避免自定义路径导致 Steam 同步失配

### 3.3 决策门 C：切换时机

新流程下，切换时机应明确写成：

- 当前会话允许选择
- 当前会话不允许切根
- 当前会话关闭后才允许迁移
- 新实例拉起前完成迁移并提交新布局

### 3.4 最小可落地的双根模型

为控制改造面，`v0.8` 采用“最小固定锚点”方案。

`anchor_root` 固定承载：

- `state/`
- `cloudsave/`
- `.cloudsave_staging/`
- `cloudsave_backups/`
- 迁移检查点
- 跨进程锁文件

`selected_root` 承载运行时用户数据：

- `config/`
- `memory/`
- `plugins/`
- `live2d/`
- `vrm/`
- `mmd/`
- `workshop/`
- `character_cards/`
- `jukebox/`

以下内容在 `v0.8` 不强制迁入锚点层：

- `logs/diagnostics/`
- `.rollback_tmp/`

它们继续跟随运行时根语义，但在迁移语义上应视为派生目录，而不是核心用户资产：

- 切换到新根后允许按需重新创建
- `v0.8` 不要求把旧根中的历史 `logs/diagnostics/` 或 `.rollback_tmp/` 全量回填到新根
- 旧根中的这两类目录可以随 retained backup 一起保留

明确不迁移、继续按外部路径处理的内容：

- Steam Workshop 实际安装目录
- `workshop_config.json` 中的 `user_mod_folder`
- `workshop_config.json` 中缓存的 `user_workshop_folder`

### 3.5 代码层兼容契约

为减少代码冲击，`ConfigManager` 在 `v0.8` 需要保持以下兼容约定：

- `app_docs_dir` 保持存在，但语义改为 `selected_root`
- 新增 `anchor_root`
- 新增 `runtime_root`，值与 `app_docs_dir` 相同
- `config_dir`、`memory_dir`、`plugins_dir`、`live2d_dir`、`vrm_dir`、`mmd_dir`、`workshop_dir`、`chara_dir` 继续从 `runtime_root` 推导
- `cloudsave_dir`、`cloudsave_staging_dir`、`cloudsave_backups_dir`、`local_state_dir` 改为从 `anchor_root` 推导

禁止的实现方式：

- 对已创建的 `ConfigManager` 实例做运行中热重绑定
- 依赖 monkey patch / 全局变量热替换修复 import 阶段缓存路径

### 3.6 `storage_policy`、`root_state`、`storage_migration` 三份状态文件职责

推荐明确区分三类状态文件。

`anchor_root/state/storage_policy.json`

- 保存稳定路由与最终选择
- 进程重启后仍然有效
- 不记录瞬时迁移阶段

推荐字段：

- `version`
- `anchor_root`
- `selected_root`
- `selection_source`: `default | user_selected | recovered`
- `cloudsave_strategy`: 固定写 `fixed_anchor`
- `first_run_completed`
- `updated_at`

关键要求：

- 若用户在首轮仅仅“请求迁移到新路径”，但迁移尚未成功，则不要提前把 `selected_root` 提交成新值
- `storage_policy` 只记录“当前已正式生效”的稳定根

`anchor_root/state/root_state.json`

- 保存运行态与最近一次结果
- 继续沿用当前已有职责

延续现有字段：

- `mode`
- `current_root`
- `last_known_good_root`
- `last_migration_source`
- `last_migration_result`
- `last_migration_backup`
- `last_successful_boot_at`

`v0.8` 建议新增：

- `legacy_cleanup_pending`

`anchor_root/state/storage_migration.json`

- 仅用于迁移过程中的检查点与恢复
- 描述“当前实例关闭后，新实例拉起前要执行什么迁移”
- 迁移成功后可删除，失败时保留

推荐字段：

- `version`
- `status`: `pending | preflight | copying | verifying | committing | retaining_source | rollback_required | failed | completed`
- `txid`
- `source_root`
- `target_root`
- `selection_source`
- `retained_source_root`
- `retained_source_mode`
- `last_processed_path`
- `error_code`
- `error_message`
- `requested_at`
- `started_at`
- `updated_at`

设计原则：

- `storage_policy` 不记录瞬时迁移阶段
- `storage_migration` 不承担读写栅栏职责
- `root_state.mode` 仍然是全局写保护真源

### 3.7 路径合法性规则

`selected_root` 目标路径必须满足：

- 必须是绝对路径
- 规范化后不能为空
- 不能位于项目目录内
- 不能位于 `anchor_root/cloudsave`、`anchor_root/state`、`anchor_root/.cloudsave_staging`、`anchor_root/cloudsave_backups` 内
- 不能是 `anchor_root` 的子目录，除非它与 `anchor_root` 完全相同
- 不能是文件
- 必须可创建或可写

推荐规范化规则：

- `Path.expanduser()`
- `resolve(strict=False)`
- Windows 下做 `normcase`
- 去除尾部分隔符
- 使用规范化后的路径做比较

额外路径警告建议：

- 目标路径位于 OneDrive / iCloud / Google Drive 等同步盘时给出强警告
- 目标路径位于外置盘、网络盘或权限受限目录时给出稳定性提示
- 对 symlink / junction / reparse point 目录做额外诊断，避免循环或伪不同路径

### 3.8 跨进程传播机制

即使首轮选择已经移到应用内，这一项仍然必须显式设计。

原因：

- 当前 launcher 仍然通过 `multiprocessing` 拉起 `main_server.py` 与 `memory_server.py`
- 两个服务都可能在 import 阶段触发 `get_config_manager()`
- Windows `spawn` 与 Linux/macOS `fork` 的子进程模型不同，不能依赖父进程内存态

推荐做法：

1. 当前会话里用户完成选择后：
   - 若保持当前路径，仅更新 `storage_policy`
   - 若请求迁移新路径，仅写 `storage_migration`
2. 当前实例关闭后，重启引导层先读取 `storage_migration`
3. 迁移成功后，再落盘最终 `storage_policy`
4. 新实例启动前，把本次正式生效的布局写入环境变量
5. 子进程在 import 重型模块前，先执行“从环境变量恢复存储布局”

推荐环境变量：

- `NEKO_STORAGE_LAYOUT_VERSION`
- `NEKO_STORAGE_ANCHOR_ROOT`
- `NEKO_STORAGE_RUNTIME_ROOT`
- `NEKO_STORAGE_CLOUDSAVE_STRATEGY`

说明：

- 持久化真源仍是 `storage_policy.json`
- 环境变量只是“本次启动一致性桥接”
- 对 `multiprocessing.Process` 和后续可能存在的 `subprocess` 辅助进程都应显式传递

### 3.9 启动顺序（按新流程重写）

新流程下推荐拆成两条链路。

第一条：当前会话首轮选择链路

1. 应用正常打开
2. 前端调用 `GET /api/storage/location/bootstrap`
3. 若需要首轮确认，则显示应用内选择弹窗/页面
4. 用户选择：
   - 若与当前路径相同，直接保存并继续当前会话
   - 若与当前路径不同，仅落迁移检查点并展示确认页
5. 用户确认后，当前实例受控关闭

第二条：关闭后的迁移与重启链路

1. 新实例拉起前，先解析 `anchor_root`
2. 读取 `storage_migration.json`
3. 若存在 `pending` 迁移，则先执行：
   - `preflight`
   - `copying`
   - `verifying`
   - `committing`
   - `retaining_source`
4. 成功后更新 `storage_policy`
5. 写入 `NEKO_STORAGE_*`
6. 子进程启动前先恢复最终布局
7. 然后才允许首次 `get_config_manager()` 绑定新路径
8. 再进入 `cloud_apply_fence`
9. 再执行 `bootstrap_local_cloudsave_environment()`
10. 再执行现有 cloudsave 相关 bootstrap/import
11. 最后进入正式服务启动

不推荐的路线：

- 当前会话选择完新路径后立刻在同一实例里切根
- 新实例里先起服务，再去处理待迁移检查点

### 3.10 `selected_root` 不可用时的策略

推荐分两类处理：

- 新装且尚未完成首启：允许直接使用默认根继续
- 已完成首启且 `selected_root` 失效：进入 `ROOT_MODE_DEFERRED_INIT`

在 `ROOT_MODE_DEFERRED_INIT` 下：

- 允许读取锚点层状态与恢复信息
- 阻断高风险写入
- 前端提示用户：
  - 重试连接原路径
  - 改选新路径并迁移
  - 显式确认“临时切回默认路径”

禁止静默在后台创建新的默认运行时根继续写入，否则会制造新的分叉数据集。

## 4. 场景状态机与默认行为

### 4.1 场景状态机

`v0.8` 至少要明确处理以下场景：

1. 新装：无 `storage_policy`，无当前数据，也无 legacy root
2. 升级但已在标准目录：无 `storage_policy`，当前运行时数据已在平台标准目录
3. 升级自旧 `Documents/legacy root`：无 `storage_policy`，存在历史旧根
4. 之前已选过自定义根，且当前路径仍可用
5. 之前已选过自定义根，但当前路径离线或损坏
6. 上次迁移中断，存在 `storage_migration` 未完成检查点

### 4.2 首轮默认策略

推荐默认策略：

- 新装且无历史数据：允许默认路径继续，但仍可要求用户在首轮确认
- 已有当前可用路径：首轮页面应优先显示“当前正在使用的路径”
- 存在 legacy root：首轮页面必须显式提示检测到旧数据
- 目标路径与当前路径一致：当前会话继续
- 目标路径与当前路径不同：必须走“关闭 -> 迁移 -> 自动重启”

### 4.3 “最佳源根”选择规则

若同时检测到多个候选 legacy root，推荐按以下顺序选最佳源根：

1. 当前正在使用的运行时根
2. 历史发布版主根 `Documents/N.E.K.O`
3. `%LOCALAPPDATA%/N.E.K.O` 或平台标准应用数据目录
4. 打包目录 / `cwd` 回退根

额外约束：

- 如果候选目录只是只读镜像或部分残留，不应优先于完整可写源根
- 若多个候选目录都存在，需结合时间戳、关键文件完整度和目录内容体量做诊断

## 5. 迁移作业设计

### 5.1 迁移对象清单

`v0.8` 的运行时迁移对象至少包括：

- `config/`
- `memory/`
- `plugins/`
- `live2d/`
- `vrm/`
- `mmd/`
- `workshop/`
- `character_cards/`
- `jukebox/`

默认不要求迁移到新根的内容：

- `cloudsave/`
- `.cloudsave_staging/`
- `cloudsave_backups/`
- `state/`

这些继续固定在 `anchor_root`。

派生目录按下列语义处理：

- `logs/diagnostics/`：允许在新根按需重建，不要求全量搬运
- `.rollback_tmp/`：允许不搬运

### 5.2 迁移执行顺序

推荐顺序：

1. 预检目标路径、空间、权限、链接目录、占用情况
2. 建立 `txid` 与迁移检查点
3. 拷贝或导入运行时目录
4. 校验目标完整性
5. 提交新策略与最终路径
6. 保留或整理旧源根
7. 清理迁移检查点并进入自动重启

核心原则：

- 先 verify 再 commit
- commit 成功后才允许新实例真正从新根启动
- 源根保留失败不应反向否决已经成功的新根提交，但必须留下明确告警

### 5.3 预检项

至少应检查：

- 目标路径是否与当前路径相同
- 目标路径是否合法、可创建、可写
- 目标路径是否位于非法子目录中
- 目标路径是否空间不足
- 源根或目标根是否被其它进程占用
- 是否命中 symlink / junction / reparse point
- 是否命中 OneDrive / iCloud / Google Drive 等同步目录
- 是否命中网络盘、外置盘、临时不可达卷

Windows 额外关注：

- CFA / 受控文件夹访问
- OneDrive 重定向 `Documents`
- 盘符变化
- `junction` / `reparse point`

macOS 额外关注：

- `~/Library/Application Support/N.E.K.O`
- iCloud Drive 同步目录
- 外置盘路径失效
- 授权体验与沙盒外目录访问

Linux 额外关注：

- `$XDG_DATA_HOME`
- `~/.local/share`
- 不同桌面环境下 `XDG_*` 变量缺失
- 挂载点权限与大小写敏感路径

### 5.4 失败与恢复

迁移失败时必须保证：

- 旧根仍可恢复
- `storage_policy` 仍指向最后一次已正式生效的根
- `storage_migration` 保留足够的恢复信息
- `root_state` 记录失败结果

恢复动作至少要支持：

- 重试同一路径
- 改选目标路径后再次迁移
- 用户确认保留当前旧根继续运行

### 5.5 迁移后提交条件

只有同时满足以下条件时才允许提交新根：

- 目标目录关键结构存在
- 关键配置文件与核心数据通过校验
- `storage_policy` 已可稳定落盘
- 新实例启动所需布局可稳定传播
- `root_state.last_known_good_root` 可更新

### 5.6 源根保留决策表

默认决策：

- 迁移成功后保留旧根
- 默认不自动删除旧根

建议保留模式：

- `standalone_backup`
- `mixed_root_retained`
- `pending_manual_cleanup`

`v0.8` 不建议默认启用自动清理；若未来提供，也必须是显式 opt-in。

### 5.7 推荐伪代码骨架

```text
first_session():
  bootstrap = get_storage_bootstrap()
  show_selection_ui_if_needed(bootstrap)
  result = user_selection()
  if result.target_root == current_root:
    save_storage_policy(current_root)
    continue_current_session()
  else:
    write_storage_migration_pending(current_root, result.target_root)
    confirm_restart()
    shutdown_current_instance()

restart_session():
  migration = load_storage_migration()
  if migration.status == pending:
    run_preflight()
    copy_runtime_entries()
    verify_target()
    commit_storage_policy(target_root)
    retain_source()
    mark_root_state_success()
  launch_services_with_committed_layout()
```

## 12. 对当前问题表述的校正

### 12.1 关于“首轮选择发生在应用内”

这句话成立，但不能误读为：

- 当前会话可以直接切根

正确含义是：

- 当前会话可以收集用户意图
- 真正切根只能发生在当前实例关闭之后

### 12.2 关于“迁移在关闭后执行”

这句话成立，但不能误读为：

- 关闭后就可以随意启动服务再做迁移

正确含义是：

- 迁移必须发生在新实例服务正式 import 重型模块之前

### 12.3 关于“logs/diagnostics 是否必须全部随迁”

`v0.8` 不要求把历史 `logs/diagnostics/` 全量视为核心用户资产迁入新根，但要求：

- 语义一致
- 新根可重建
- 失败时 diagnostics 有稳定落点

### 12.4 关于 `characters.json` / `core_config.json` 回退读取

这些问题并没有因为首轮改成应用内选择而消失。

只要读路径仍可能回退到项目目录或旧根，迁移后仍会读写分裂，因此它们仍然是 `v0.8` 的必改项。

### 12.5 关于 `utils/preferences.py` 等 import 时缓存路径模块

这些问题仍然存在，而且在“关闭后重启迁移”的流程中同样关键。

原因是：

- 新实例一旦 import 过早，就会再次把旧路径缓存下来

所以必须在新实例里把“最终布局恢复”放到首次重型 import 之前。

### 12.6 建议在评审和验收中使用的统一分类

评审时建议统一按下面四类讨论：

- 首轮应用内交互是否正确
- 关闭后迁移与自动重启是否正确
- 双根布局与 cloudsave 锚点是否正确
- 读写收敛与恢复策略是否正确

`v0.8` 的正确方案不是“当前会话里热切换路径”，也不是“丢掉旧版所有工程约束”；而是“应用内完成首轮选择，必要时关闭当前实例，在新实例拉起前完成迁移，并在固定锚点与读写收敛约束下自动重启到新路径”。
