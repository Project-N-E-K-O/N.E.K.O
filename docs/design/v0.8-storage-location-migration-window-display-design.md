## 0. 设计结论

本次窗口设计采用下面这条流程：

1. 应用先正常打开。
2. 打开后，先进入一个极早期的“存储状态确认”阶段。
3. 如果轻量状态探测判定当前无需阻断，则直接放行主业务界面。
4. 如果轻量状态探测判定当前必须阻断，再在应用内显示存储位置选择弹窗或全页覆盖层。
5. 如果用户选择的路径与当前路径一致，直接关闭弹窗，继续当前会话。
6. 如果用户选择的路径与当前路径不同，显示“即将关闭当前实例并迁移”的确认页。
7. 用户确认后，当前实例关闭。
8. 关闭后执行迁移。
9. 迁移完成后自动重启。

本次窗口设计只讨论应用打开后的应用内窗口与重启后的过渡显示。

但要明确当前边界：

- Web 启动模式的现阶段实现与验收，以实施文档为准
- 本文主要记录“后续桌面端启动补充设计”
- 与网页端完全相同的通用语义，本文只做引用，不重复扩展为当前实现范围

## 1. 文档边界

本文只回答显示层问题：

- 什么时候显示
- 显示什么
- 用户能点什么
- 桌面端启动时还需要额外补什么承载
- 重启迁移期间如何展示状态

本文不负责定义：

- 真实 copy / verify / commit 的底层算法
- `storage_policy` 和 `storage_migration` 的完整 schema
- 启动后设置页的完整管理能力

---

## 2. 总体原则

### 2.1 首次选择发生在应用内

窗口必须出现在应用已经打开之后。

但这里的“打开之后”不是指主业务已经完全初始化完毕，而是：

- 应用窗口已经出现
- 前端已进入一个极早期的存储状态确认阶段
- 若确认需要阻断，则在主业务继续初始化前切换到存储选择页

允许的承载形式：

- 强制选择 modal
- 全页覆盖层
- 首屏强制向导页

不建议的承载形式：

- 独立于当前应用界面的额外首启入口
- 用户需要自行跳转才能进入的首次选择页

### 2.2 当前会话不做热切换

显示层必须让用户清楚知道：

- 当前会话仍然使用当前路径
- 只有在关闭当前实例后，新路径才会真正生效

### 2.3 用户必须先理解“是否要重启”

窗口文案不能只说“已保存”。

必须明确区分：

- 选择结果等于当前路径：不需要重启
- 选择结果不同于当前路径：需要关闭当前实例、迁移并自动重启

### 2.4 窗口是应用内阻断

这里的“阻断”仅指：

- 在当前页面内，用户需要先完成存储位置选择

它的含义只是：

- 用户需要先在当前页面完成存储位置选择

## 3. 窗口生命周期

### 3.1 阶段总览

建议把显示层固定为下面几个阶段：

| 阶段 | 含义 | 允许动作 |
| --- | --- | --- |
| `startup_checking` | 应用已打开，正在确认当前是否必须进入存储引导 | 无 |
| `selection_required` | 第一次进入后要求用户选择存储位置 | `use_current` / `use_recommended` / `choose_custom` |
| `selection_confirm` | 用户已选中目标路径，等待确认是否继续 | `confirm_selection` / `back` |
| `restart_required` | 目标路径与当前路径不同，需要关闭当前实例 | `confirm_restart` / `back` |
| `shutting_down` | 当前实例正在退出 | 无 |
| `migrating` | 已退出当前实例，正在迁移 | 无 |
| `migration_failed` | 迁移失败 | `retry` / `stay_current` / `view_details` |
| `recovery_required` | 之前选定路径失效或上次迁移中断，需要显式恢复决策 | `retry_path` / `choose_new_path` / `use_recovered_default` |
| `migration_ready` | 迁移完成，准备或已经重启 | `continue` |

### 3.2 用户心智必须稳定

每个阶段都必须让用户知道三件事：

1. 当前是否还在旧路径上运行
2. 是否已经开始迁移
3. 下一步是否会关闭当前实例

如果处于 `startup_checking`，还必须回答：

4. 现在是在确认存储布局状态，不是在加载失败
5. 主业务界面会在确认完成后再继续初始化

如果处于恢复态，还必须额外回答：

4. 当前失效的是“上次正式生效路径”还是“一次未完成迁移”
5. 用户此刻做的是恢复决策，不是一次普通首轮选择

### 3.3 关闭按钮语义

关闭窗口不能等于“默默继续”。

建议规则：

- `selection_required` 阶段关闭：等价于留在当前路径，或弹二次确认
- `restart_required` 阶段关闭：回到选择页，不自动开始迁移
- `shutting_down` / `migrating` 阶段：不允许随意中断

## 4. 页面骨架与内容

### 4.1 首轮选择页

首轮页面必须至少展示：

- 当前路径
- 推荐路径
- 检测到的旧数据目录
- 锚点目录中的 cloudsave 将继续固定保留
- 是否建议迁移
- 是否需要重启

建议按钮：

- `保持当前路径`
- `使用推荐位置`
- `选择其他位置`

### 4.2 确认页

确认页需要把选择结果讲清楚。

如果用户保持当前路径，应显示：

- 当前会话将继续
- 不会关闭应用

如果用户改成不同路径，应显示：

- 当前实例会先关闭
- 关闭后迁移数据
- 迁移完成后自动重启
- 旧数据默认不会自动删除

### 4.3 关闭中页面

当用户确认切换到不同路径后，当前实例进入关闭流程。

此时显示层应只表达一件事：

- 正在关闭当前实例，随后会迁移并自动重启

不需要：

- 倒计时
- 无意义的等待秒数
- 让用户做额外操作

### 4.4 迁移中页面

迁移阶段建议展示：

- 当前阶段
- 总进度
- 当前处理对象摘要
- 是否仍保留旧数据

如果无法稳定显示细粒度进度，也至少应稳定显示：

- `正在迁移数据，请稍候`

### 4.5 成功页

成功页必须建立用户安全感，而不是只显示“完成”。

必须说明：

- 新路径已经生效
- cloudsave / 状态目录仍位于固定锚点目录
- 旧数据是否保留为备份
- 后续如果要清理，应去哪里操作

### 4.6 失败页

失败页必须至少回答：

1. 卡在什么阶段
2. 旧数据是否还安全
3. 用户下一步该做什么

建议动作：

- `重试`
- `保留当前路径`
- `查看详情`

### 4.7 恢复态页面

当主设计文档中的以下场景出现时，必须进入恢复态，而不是复用普通失败页：

- 上次迁移中断
- 之前已选过的 `selected_root` 当前不可用
- 系统进入 `ROOT_MODE_DEFERRED_INIT`

恢复态页面至少要展示：

- 上次正式生效路径
- 当前不可用的路径
- 是否检测到未完成迁移
- 当前是否允许临时回退到默认根

建议动作：

- `重试原路径`
- `重新选择位置`
- `确认临时切回默认路径`

## 5. 桌面端启动补充承载

网页端启动模式的承载方式，以实施文档为准。

本文第 5 章只补充桌面端启动时新增的窗口与交互要求。

### 5.1 桌面端启动

桌面端下：

- 用户先正常进入桌面应用窗口
- 桌面窗口出现后，先进入同样的“存储状态确认”阶段
- 若确认需要阻断，再在应用窗口内显示同一套选择页
- 若确认切换路径，关闭当前实例后可显示独立迁移进度窗口

桌面端可额外提供：

- 原生目录选择器
- 打开当前目录
- 打开旧备份目录

### 5.2 与网页端共享的核心语义

无论后续桌面端怎么承载，下列语义都必须与网页端保持一致：

- 首次选择发生在应用内
- 当前会话不热切换
- 切换到不同路径时，必须先关当前实例
- 迁移完成后自动重启

## 6. 显示层协议建议

### 6.1 启动后轻量状态探测

前端进入应用后，调用：

- `GET /api/v1/system/status`

建议返回：

- `ready`
- `status`
- `selection_required`
- `migration_pending`
- `recovery_required`

该接口的职责是：

- 尽早判断当前是否必须阻断主业务初始化
- 让显示层先进入 `startup_checking`
- 若无需阻断，则直接放行主业务
- 若需要阻断，再进入 bootstrap 阶段

### 6.2 启动后 bootstrap 数据

前端在状态探测确认需要阻断后，再调用：

- `GET /api/storage/location/bootstrap`

建议返回：

- `current_root`
- `recommended_root`
- `legacy_sources`
- `anchor_root`
- `cloudsave_root`
- `selection_required`
- `migration_pending`
- `recovery_required`
- `legacy_cleanup_pending`
- `last_known_good_root`

### 6.3 用户选择

前端提交选择时调用：

- `POST /api/storage/location/select`

建议返回：

- `continue_current_session`
- `restart_required`

### 6.4 重启动作

当前实例需要关闭并迁移时，前端调用：

- `POST /api/storage/location/restart`

前端据此切换显示阶段：

- `restart_required`
- `shutting_down`

### 6.5 迁移状态

迁移期间可通过：

- `GET /api/storage/location/status`
- 或 WebSocket / SSE

向显示层同步：

- `migrating`
- `migration_failed`
- `recovery_required`
- `migration_ready`

首轮正式显示协议应直接服务于应用内页面与重启后的过渡显示。

如果状态接口要支撑主设计里的恢复态与锚点语义，还应按需补充：

- `backup_root`
- `current_root`
- `last_migration_result`
- `error_code`
- `retained_source_mode`

## 7. 文案约束

### 7.1 必须说清楚的句子

文案必须明确表达下面这些意思：

- 你当前正在使用哪个路径
- 你刚刚选择的是哪个路径
- 如果两者不同，当前实例会先关闭
- 迁移完成后会自动重启
- cloudsave 和状态目录不会跟着一起迁走
- 旧数据默认不会自动删除

### 7.2 不要再出现的表达

文案里不要再出现：

- 把用户引导到其它入口完成首轮选择的提示
- 需要额外理解启动链路的内部术语
- 无意义倒计时秒数

---

## 8. 落地顺序建议

1. 先按实施文档补齐网页端 `startup_checking`、`selection_required` 页面与首轮接线。
2. 再以实施文档完成 `POST /api/storage/location/select` 与 Web 侧确认流。
3. 网页端闭环稳定后，再补桌面端启动下的承载差异。
4. 然后补桌面端独立迁移过渡页或进度窗口。
5. 再补桌面端 `recovery_required` 恢复页与目录操作能力。
6. 最后补成功页、失败页和清理入口的桌面端增强体验。

---

## 9. 验收要点

本章验收口径针对“后续桌面端启动补充”：

- 第一次启动时，桌面应用先正常打开。
- 打开后先进入存储状态确认阶段；若判定需要阻断，再立即在桌面应用窗口内出现存储位置选择弹窗或页面。
- 保持当前路径时，不关闭当前实例。
- 选择不同路径时，先出现明确确认页。
- 用户确认后，当前实例关闭，随后迁移并自动重启。
- 自定义路径失效或迁移中断时，会进入明确恢复态，而不是静默回退。
- 成功页能说明新路径、固定锚点 cloudsave 和旧备份的关系。
- 桌面端如提供额外窗口或原生目录能力，也不能改变与网页端一致的核心语义。

## 10. 一句话结论

网页端启动的当前实现看实施文档；本文只补“桌面端启动时，这套应用内选择与关闭后迁移流程该如何承载和增强显示”。
