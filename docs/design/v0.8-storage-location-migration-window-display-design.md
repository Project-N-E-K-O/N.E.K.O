## 0. 文档定位

本文回答四件事：

- 存储位置切换在当前网页端启动模式下，实际有哪些用户可见页面与状态
- 每个状态下显示什么、隐藏什么、允许用户点什么
- 哪些显示语义已经在代码里落地，哪些仍属于桌面端后续补充
- 后续桌面端启动如果补原生承载，哪些语义必须与网页端保持一致

本文不负责：

- 底层 copy / verify / commit 算法细节
- `storage_policy.json` / `storage_migration.json` 的完整 schema 定义
- 设置页里的长期存储管理能力

## 1. 当前代码已经确定的显示结论

### 1.1 当前网页端不是“多页面拼接”，而是一个阻断覆盖层加一个完成提示卡片

当前实现由两部分组成：

- 一个启动期阻断覆盖层，内部有 4 个子视图：
  - `loading`
  - `selection_required`
  - `maintenance`
  - `error`
- 一个非阻断的迁移完成提示卡片：
  - `completion_notice`

这意味着当前真实显示模型并不是文档早期版本里那种完全拆开的：

- `shutting_down`
- `migrating`
- `migration_ready`
- 独立成功页
- 独立失败页

网页端当前实现把“关闭当前实例”“关闭后迁移”“等待服务恢复”收敛成了同一个 `maintenance` 维护态页面，并用进度条和状态文案表达阶段变化。

### 1.2 当前正式流程

当前代码和两份主文档对齐后的正式流程是：

1. 应用先正常打开。
2. 前端先请求 `GET /api/v1/system/status` 做轻量状态探测。
3. 若无需阻断，则直接放行主业务初始化。
4. 若需要阻断，再请求 `GET /api/storage/location/bootstrap`。
5. 前端显示应用内覆盖层，要求用户先完成本次存储决策。
6. 若用户选择结果等于当前路径：
   - 保存选择；
   - 解除启动阻断；
   - 当前会话继续运行。
7. 若用户选择结果不同于当前路径：
   - 先显示重启前预检与确认卡片；
   - 用户确认后调用 `POST /api/storage/location/restart`；
   - 当前实例关闭；
   - 关闭后执行恢复布局或迁移；
   - 服务恢复后页面自动重新连接。

### 1.3 当前会话绝不热切根

显示层必须持续向用户传达这件事：

- 当前页面里做的只是“确认意图”
- 真正切换到新路径发生在当前实例退出之后
- 如果是普通迁移，则会在关闭后迁移并自动重启
- 如果是恢复态下的 `rebind_only`，则会关闭并重连原路径，但不会复制运行时数据

### 1.4 启动阻断是硬要求，不是提示

只要出现以下任一条件，主业务必须继续阻断：

- `selection_required=true`
- `migration_pending=true`
- `recovery_required=true`

当前前端测试已经把这条约束固化为验收标准：

- 覆盖层未解除前，`pageConfig` 不得先解析
- 覆盖层未解除前，独立启动请求不得抢跑

## 2. 当前网页端真实生命周期

### 2.1 阶段总览

建议以后统一按下面这套“代码事实优先”的显示生命周期理解：

| 前端阶段 | 后端状态来源 | 用户看到什么 | 允许动作 |
| --- | --- | --- | --- |
| `loading` | `GET /api/v1/system/status` 与 `GET /api/storage/location/bootstrap` 之间 | “正在确认存储布局状态” | 无 |
| `selection_required` | `blocking_reason=selection_required` 或 `recovery_required` | 路径总览、选择按钮、可展开的其他位置面板 | `保持当前路径` / `使用推荐位置` / `选择其他位置` / `提交该位置` |
| `selection_required + preview` | `POST /api/storage/location/select -> restart_required` | 重启前预检卡片 | `确认关闭并迁移` / `确认关闭并重连路径` / `返回重新选择` |
| `maintenance` | `blocking_reason=migration_pending` 或 `POST /restart -> restart_initiated` | 维护态进度页 | 无用户级业务动作 |
| `error` | bootstrap / status 请求异常 | 读取失败页 | `重试` |
| `completion_notice` | `GET /api/storage/location/status -> completion_notice.completed=true` | 迁移完成提示卡片 | `清理旧数据目录` / `关闭` |

### 2.2 不再建议沿用的旧阶段名

以下阶段名在设计语义上可以理解，但不应再当成当前网页端的真实页面清单：

- `startup_checking`
- `selection_confirm`
- `restart_required`
- `shutting_down`
- `migrating`
- `migration_failed`
- `migration_ready`

原因不是这些语义不存在，而是当前实现已收敛成更稳定的显示模型：

- `startup_checking` 对应 `loading`
- `selection_confirm` / `restart_required` 对应选择页里的预检确认卡片
- `shutting_down` / `migrating` 对应 `maintenance`
- “成功”对应非阻断完成提示卡片，而不是独立全页成功页
- “失败/恢复”当前优先回到 `recovery_required` 选择流，而不是单独再开一个失败页

### 2.3 关闭按钮语义

当前网页端覆盖层没有提供“右上角关闭并继续”的逃生语义。

后续桌面端如果补关闭按钮，规则也必须保持一致：

- 不能把关闭窗口解释成“默默继续使用”
- 在需要阻断时，关闭只能等价于“仍未完成选择”
- 在预检确认阶段，关闭或返回都只能回到选择页，不能自动开始迁移
- 进入维护态后，不应再提供可随意中断的关闭动作

## 3. 页面设计与展示要求

### 3.1 `loading`：启动期存储状态确认页

页面目标：

- 告诉用户这不是普通加载失败
- 明确当前正在确认存储布局状态
- 明确主业务会在存储状态确认完成后再继续加载

当前推荐文案方向：

- 标题：`正在确认存储布局状态`
- 副标题：`主业务界面会在存储状态确认完成后再继续加载。`

此页不需要：

- 错误恐吓文案
- 倒计时
- 用户操作按钮

### 3.2 `selection_required`：首轮选择页

当前实现的首屏最小展示集是：

- 当前路径
- 推荐路径
- 三个主要动作：
  - `保持当前路径`
  - `使用推荐位置`
  - `选择其他位置`

当前实现刻意不在首屏直接堆出以下内部信息：

- `anchor_root`
- `cloudsave_root`
- “固定锚点目录”说明块
- “固定 cloudsave 目录”说明块
- “已检测到的旧数据目录”独立标题

这样做的原因是：

- 首屏先让用户完成核心决策
- 内部布局概念和技术路径不要抢占首轮注意力
- 已检测到的旧数据目录只在用户主动展开“其他位置”后再作为复用候选出现

这点已经被前端测试固定为当前 UI 约束。

### 3.3 `selection_required` 下的“其他位置”面板

当用户点 `选择其他位置` 后，才展开补充能力：

- 展示可复用的旧数据路径候选
- 允许手动输入自定义路径
- 支持点击 `选择文件夹`

当前代码约束如下：

- 目录选择器优先走系统原生能力
  - macOS: `osascript`
  - Windows: PowerShell `FolderBrowserDialog`
  - Linux: `zenity` / `kdialog` / `yad`
- 原生选择器不可用时回退到 `tkinter`
- 仍不可用时，前端保留手动输入路径能力

因此文档里应把“选择文件夹”视为：

- 可用时优先体验
- 不可用时允许退化
- 不能成为唯一路径输入方式

### 3.4 重启前预检确认卡片

当 `POST /api/storage/location/select` 返回 `restart_required` 时，当前实现不会立刻进入维护态，而是先显示一个确认卡片。

这张卡片当前必须表达 4 类信息：

1. 来源与目标
   - 当前路径
   - 目标路径
2. 预检结果
   - `estimated_required_bytes`
   - `target_free_bytes`
   - `permission_ok`
   - `warning_codes`
3. 阻断错误
   - `blocking_error_code`
   - `blocking_error_message`
4. 行为边界
   - 本会话不会热切根
   - 确认后会按后端顺序执行关闭、必要迁移、自动重启与布局恢复

当前预检卡片的按钮必须按两类场景切换：

- 普通迁移：`确认关闭并迁移`
- 恢复重连：`确认关闭并重连路径`

只要存在阻断错误，例如：

- 目标路径不可写
- 目标卷剩余空间不足

确认按钮就必须禁用，不能让用户继续走到 `restart`。

### 3.5 `maintenance`：关闭、迁移、恢复服务的一体化维护页

这是当前网页端最重要的状态页。

它承担的是原文档中多个阶段的合并职责：

- 等待当前实例安全关闭
- 关闭后执行迁移或重连原路径
- 等待服务恢复
- 服务恢复后自动重新连接页面

当前页面元素至少包括：

- 标题：`正在优化存储布局...`
- 副标题：说明“当前实例即将关闭，数据会在关闭后迁移并自动重启”
- 状态说明
- 进度条
- 四步进度标签：
  - `关闭当前实例`
  - `处理存储目录`
  - `校验并生效`
  - `恢复服务`
- 一条稳定提示：
  - 连接短暂中断属于正常现象，页面会在服务恢复后自动重新连接

### 3.6 维护页的阶段表达

当前代码不是显示精确文件级进度，而是按迁移检查点状态映射阶段文本：

- `pending`
- `preflight`
- `copying`
- `verifying`
- `committing`
- `retaining_source`
- `completed`
- `failed`
- `rollback_required`

因此文档应明确：

- 当前网页端显示的是“阶段化进度”，不是实时字节流进度
- 这已经足够支撑用户心智
- 不需要伪造倒计时或秒级剩余时间

### 3.7 恢复态页面

恢复态在当前代码里不是独立新窗口，而是“选择页 + 恢复约束”。

典型触发场景：

- 已提交的 `selected_root` 当前不可用
- 上次迁移流程中断后系统仍处于恢复要求状态
- 后端进入 `ROOT_MODE_DEFERRED_INIT`

当前恢复态至少有三类不同分支，文档必须分开写：

#### A. 切回当前可用根继续当前会话

如果当前运行根可继续使用，且恢复逻辑允许直接在当前会话落回当前根，则：

- 用户可以走 `保持当前路径` 或切回推荐根
- 后端保存新的选择
- 当前会话解除阻断继续运行

#### B. 原始路径重新可用，但需要关闭后重连

如果之前不可用的原始路径重新可用，则：

- `select` 会返回 `restart_required`
- 同时 `restart_mode = rebind_only`
- 页面必须明确告诉用户：
  - 后续会关闭当前实例并自动重连原路径
  - 本次不会复制运行时数据

#### C. 恢复态下不允许任意切到第三条新路径

当前代码明确限制：

- 当恢复态源路径不可用时，用户不能直接改到一个完全不同的第三路径

此时前端应收到并展示类似错误：

- `recovery_source_unavailable`

文档里必须把这条规则写清楚，否则“恢复态”和“普通首轮选择”会被混成一件事。

### 3.8 完成提示卡片，不是独立成功页

当前迁移完成后的展示方式是：

- 主业务恢复为 `ready`
- 覆盖层消失
- 若旧数据目录仍保留，则弹出一个非阻断完成提示卡片

这张卡片当前至少展示：

- 原始路径
- 当前生效路径
- 当前保留目录
- 一句说明：
  - 新目录已经生效
  - 旧数据目录目前仍保留
  - 是否清理由用户决定

当前可见动作：

- `清理旧数据目录`
- `关闭`

因此文档不应再写成“迁移完成后进入独立成功页再点继续”，这和当前代码不一致。

### 3.9 失败显示

当前网页端没有单独定型的“失败页”。

真实行为应表述为：

- 若接口读取失败，进入 `error` 页
- 若迁移或恢复进入异常状态，优先通过：
  - 维护页状态文案
  - 重新进入 `recovery_required` 选择流
  - 选择页中的错误提示
  来完成恢复，而不是单独再开一张失败页

因此文档里若提“失败页”，必须降级成“后续桌面端可补充的增强体验”，不能写成当前网页端既有事实。

### 3.10 `error`：接口读取失败页

当前 `error` 页只负责一件事：

- 当 bootstrap / status 获取失败时，给用户一个明确重试入口

最小设计即可：

- 标题：无法读取存储位置引导信息
- 错误文本
- 按钮：`重试`

## 4. 显示层协议与字段使用建议

### 4.1 启动期轻量探测

前端首先调用：

- `GET /api/v1/system/status`

这里的职责是：

- 尽早判断是否必须阻断主业务初始化
- 决定是否继续请求 `/api/storage/location/bootstrap`

对显示层真正关键的不是接口字段越多越好，而是这几个判断：

- `ready`
- `status`
- `storage.selection_required`
- `storage.migration_pending`
- `storage.recovery_required`
- `storage.blocking_reason`

### 4.2 bootstrap

阻断成立后，前端再调用：

- `GET /api/storage/location/bootstrap`

当前显示层会实际消费的关键字段有：

- `current_root`
- `recommended_root`
- `legacy_sources`
- `selection_required`
- `migration_pending`
- `recovery_required`
- `blocking_reason`
- `last_known_good_root`
- `last_error_summary`
- `migration`
- `poll_interval_ms`

下面这些字段当前更适合作为协议层或后续增强预留，而不是首屏必须直出给用户：

- `anchor_root`
- `cloudsave_root`
- `legacy_cleanup_pending`

### 4.3 select

用户确认某个路径时调用：

- `POST /api/storage/location/select`

当前必须覆盖两类返回：

- `continue_current_session`
- `restart_required`

若返回 `restart_required`，前端还应消费：

- `restart_mode`
- `estimated_required_bytes`
- `target_free_bytes`
- `permission_ok`
- `warning_codes`
- `blocking_error_code`
- `blocking_error_message`

### 4.4 restart

真正开始关闭当前实例时调用：

- `POST /api/storage/location/restart`

成功后前端立即切入 `maintenance`，并开始轮询：

- `GET /api/storage/location/status`

### 4.5 status

维护态与完成提示主要依赖：

- `ready`
- `status`
- `lifecycle_state`
- `migration_stage`
- `maintenance_message`
- `poll_interval_ms`
- `blocking_reason`
- `last_error_summary`
- `migration`
- `completion_notice`

### 4.6 完成后旧目录清理

当 `completion_notice.completed=true` 且 `cleanup_available=true` 时，前端可提供：

- `POST /api/storage/location/retained-source/cleanup`

这是“迁移完成后的善后操作”，不是迁移主流程的一部分。

## 5. 当前网页端文案原则

必须说清楚：

- 当前路径是什么
- 目标路径是什么
- 当前会话不会热切换
- 若路径不同，必须先关闭当前实例
- 关闭后才会恢复布局，必要时再迁移数据
- 迁移完成后会自动重启或自动恢复服务
- 旧数据默认不会自动删除

尽量不要在首屏直接抛给用户：

- `anchor_root`
- `cloudsave_root`
- `storage_policy`
- `storage_migration`
- `ROOT_MODE_DEFERRED_INIT`

这些概念可以存在于：

- 调试信息
- 恢复态补充说明
- 桌面端高级详情

但不应成为首轮选择页的主叙事。

## 6. 桌面端启动的后续补充要求

本文在桌面端只补“承载差异”，不改变核心语义。

### 6.1 桌面端可以补的能力

桌面端后续可增加：

- 原生目录选择器的更稳定接入
- 独立迁移进度窗口
- 打开当前目录 / 打开保留目录
- 恢复态下更明确的详情区

### 6.2 但这些语义绝不能改

无论桌面端怎样扩展，以下语义必须保持与网页端一致：

- 应用先出现，再进入受限启动下的存储确认
- 当前会话不做热切根
- 路径不同必须先关闭当前实例
- 关闭后才执行迁移或重连
- 恢复态不是普通首轮选择的同义词
- 迁移完成后旧数据默认保留，清理必须显式确认

### 6.3 桌面端不应引入的新误导

桌面端即使补充原生窗口，也不要做成：

- 关闭窗口就默认继续当前路径
- 直接在当前会话里切根
- 把 cloudsave 锚点和运行时根混成同一件事
- 自动删除旧目录且不做明确确认

## 7. 验收清单

### 7.1 当前网页端必须满足

- 首次阻断时，先看到 `loading`，随后再进入选择页或维护页
- 覆盖层未解除前，主业务初始化不得先放行
- 首屏默认只突出当前路径、推荐路径和主要动作
- “其他位置”支持旧路径复用、手动输入和选择文件夹
- 选择不同路径后，必须先出现预检确认卡片
- 预检失败时，确认关闭按钮必须禁用
- 用户确认关闭后，进入维护页并持续轮询状态
- 服务恢复后页面自动重新连接
- 若迁移完成但旧目录仍保留，会显示非阻断完成提示卡片
- 用户可以显式清理旧数据目录

### 7.2 后续桌面端补充验收

- 桌面端新增窗口不改变网页端已经定下的状态语义
- 恢复态下能够区分“继续当前可用根”“关闭后重连原路径”“禁止切到第三路径”
- 桌面端若显示更多详情，也不能把内部实现细节直接变成首屏负担

---

## 8. 一句话结论

这条流程当前最准确的显示模型不是“多个抽象页面串联”，而是“启动阻断覆盖层 + 重启前预检卡片 + 维护态进度页 + 迁移完成提示卡片”；后续桌面端只补承载，不改语义。
