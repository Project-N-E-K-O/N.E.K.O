# 猫娘三态 GIF 可见扩展方案收口

> 本文是“请她离开”三态猫 GIF 的后续交互扩展判断稿。它不替代
> [cat-idle-states-feature.md](./cat-idle-states-feature.md) 和
> [cat-idle-states-reference.md](./cat-idle-states-reference.md)，也不改变当前
> goodbye / return / CAT1-CAT3 的既有语义。

## 结论

后续值得做的不是“更多交互逻辑”，而是**更明显的可见符号和物件设计**。

有价值的方向应该满足：

1. 用户一眼能看懂，不需要解释。
2. 主要靠图片资源、overlay 或最小化聊天框皮肤表达。
3. 不增加猫和聊天框之间的持续物理拉扯；如果移动，必须是一次性编排。
4. 不改变 return/goodbye 主链路。
5. 不干扰 CAT1 走路、CAT2/CAT3 停靠、拖拽、hover、return 的生命周期。

用户已提出的“睡觉猫头上 zzz”“最小化聊天框变成毛线球或鱼缸”属于正确方向：视觉价值高，语义直接，主要是资源和皮肤问题，不需要把系统改成复杂桌宠。

## 外部参考提炼

外部桌宠和养猫游戏里真正能迁移的不是追鼠标、拖窗口、随机打扰，而是“猫和物件的关系”。

1. Neko Atsume 的核心是摆放玩具、食物、家具吸引猫；有价值的是物件语义，而不是复杂控制。
2. 桌面宠物应用常见的有效表现是 nap、snack、play 这类一眼能懂的动作或道具。
3. 工具内宠物更适合低频反馈和可忽略陪伴，不适合抢主任务。
4. Cozy / idle 小组件的价值在于边缘陪伴和状态暗示，而不是强交互。

## 方案筛选标准

| 判断项 | 值得做 | 不值得做 |
| --- | --- | --- |
| 可见性 | 一眼看到，如头顶符号、道具、皮肤、垫子、猫带物件小移动 | 只有细微朝向、无语义的几 px 抖动、内部等待差异 |
| 实现方式 | 增加资源、overlay、CSS 层、皮肤映射 | 新增复杂调度、目标跟踪、拖拽分级 |
| 生命周期 | 能随 tier/return/drag 一次性清理 | 需要多处 timer、observer、跨端状态互相兜底 |
| 语义 | 强化“猫在睡/猫靠近玩具/猫带着物件移动” | 稀释“点击回来”或让聊天球变成持续物理绳子 |
| 跨端风险 | 网页端和桌面端都只是换视觉 | 需要重新设计窗口同步和坐标恢复 |

## 推荐方向 A：睡眠与情绪符号层

这类最适合 CAT2/CAT3，因为它不会动容器，也不碰聊天球。

可做方案：

1. CAT3 头顶 `zzz`：睡觉语义最直接。
2. CAT2 头顶小气泡：可以是 `...`、小月亮、半闭眼符号，表示打盹但没睡死。
3. CAT3 梦境小泡泡：偶尔出现小鱼、星星、小枕头这类图标，替代更复杂的动作。
4. hover 时符号轻微消失或淡出，避免和当前 click GIF 抢视觉。

实现边界：

1. 符号是 overlay，不改 GIF 本身。
2. 只跟随 return-ball art 的视觉位置，不参与 hit area。
3. tier 离开、return、drag start 时立即隐藏。
4. 不为符号新增业务状态；它只是当前 tier 的装饰层。

价值：可见性强，最少改动，能明显补足“CAT2/CAT3 静态循环不够像活物”的问题。

## 推荐方向 B：最小化聊天框物件皮肤

这类是高价值方向。聊天框最小化后不一定继续像普通 UI 球，而可以变成“猫会靠近的物件”。

用户已提出的毛线球、鱼缸属于这一类。后续还可以评估：

1. 纸箱：猫靠近纸箱非常直观，适合 CAT1 走向目标。
2. 小鱼干盘 / 食盆：强化“回来入口是猫在守着的东西”。
3. 猫窝 / 软垫：适合 CAT2/CAT3 停靠，让聊天球靠近猫时更自然。
4. 小铃铛 / 玩具鼠：比普通球更有猫相关语义。
5. 抱枕 / 方块垫：适合和当前圆形最小化入口做尺寸兼容。
6. 逗猫棒末端：可作为 hover 皮肤变化，但不应真的追鼠标。

实现边界：

1. 只替换最小化聊天框的视觉皮肤，不改变它作为 chat minimized shell 的业务身份。
2. 点击仍然恢复聊天框或走原有 return 语义，不新增“玩具点击玩法”。
3. CAT1 仍只是走向最小化目标，不因为物件类型改变寻路规则。
4. CAT2/CAT3 dock 仍用现有停靠规则，只是停靠对象更像猫道具。
5. 物件皮肤必须有固定安全区，避免桌面端折叠尺寸、命中区域和裁剪再次出问题。

价值：用户一眼能看到“聊天框和猫建立了关系”，比增加微小动作更有效。

## 推荐方向 C：CAT1 带最小化物件小移动（已接入第一版）

这类可以做，而且比猫单独小移动更有价值。前提是最小化聊天框已经有明确物件皮肤，例如毛线球、玩具、纸箱、鱼缸等；猫和物件一起小范围移动，用户能直接理解为“她带着东西挪了一下”。

关键规则：

1. 如果只是 CAT1 猫自己小移动，必须回到原位，避免污染 CAT1 和聊天球的距离关系。
2. 如果 CAT1 和最小化聊天框物件一起移动，不需要回原位；两者相对距离不变，最终位置可以成为新的停留位置。
3. 一起移动不是实时跟随，不是拖拽关系，而是一次完整的短编排。
4. 编排期间要抑制自身引起的距离重判，不能因为这次移动又触发 CAT1 重新走向聊天球。

适合场景：

1. CAT1 已经走到最小化聊天框物件旁边并 settled。
2. 当前没有 pending walk、walking、stretch、hover、drag、return、tier change。
3. 聊天框处于最小化物件皮肤状态。
4. 桌面端 bridge 没有 pending shrink/restore。

动作形式：

1. 猫和物件保持相对位置，一起水平移动一小段。
2. 可以是轻轻推着走、抱着走、绕一小弧线、挪到旁边一点。
3. 位移要明显到肉眼可见，但不能大到像用户拖走了窗口。
4. 结束后以新位置作为稳定位置，不自动回原点。

当前第一版实现边界：

1. 第一版作用于网页端 React chat 最小化球，因为猫和聊天球同在页面坐标体系内。
2. 桌面端仍等物件皮肤和窗口尺寸稳定后再评估，避免重新引入 shrink/restore、左上角闪图、坐标回跳问题。
3. 动作开始时记录 pair token；return、drag、tier change、chat expand、bridge restore 都必须取消 token。
4. 动作结束后只保留最终位置，不把中间每一帧当成用户拖拽或聊天球真实拖动。
5. 不能复用普通拖拽生命周期；这是自动编排动作，不是用户输入。
6. 随机间隔为几十秒到几分钟；移动时复用 CAT1 走路 GIF，结束后恢复 CAT1 默认图。

价值：这是少数“移动”也有意义的方向，因为它不是无语义小抖动，而是在强化猫和物件的关系。

## 推荐方向 D：猫周围的小场景底座

这类用于增强 CAT2/CAT3 的停留感，不需要新交互。

可做方案：

1. CAT2 脚边出现小软垫或半折毯子。
2. CAT3 身下出现猫窝、垫子、被子边缘。
3. CAT3 深睡一段时间后叠加小被角或夜帽，但不换 tier。
4. 当聊天球停靠到 CAT2/CAT3 左侧时，球和猫之间可以出现很小的地面阴影或垫子连接感。

实现边界：

1. 场景底座是 return-ball 内部/周边装饰，不改变容器坐标。
2. 不参与拖拽逻辑；拖拽时整体跟着 return-ball 视觉层走或直接隐藏。
3. 不做复杂场景系统，不根据屏幕内容变化。
4. 资源应保证透明背景、固定安全区、不同 tier 切换不跳尺寸。

价值：比“猫自己移动几 px”更明显，也更稳定。

## 推荐方向 E：低频自主动作

这一项仍然保留，但它不应该扩展成动作系统。

建议只评估：

1. CAT2 偶尔播放 `cat-idle-cat2-click.gif` 一轮，像打盹中半醒一下。
2. CAT3 极低频播放 `cat-idle-cat3-click.gif` 一轮，像睡梦中轻微反应。

不建议给 CAT1 加无语义的单独自主动作。CAT1 如果要动，优先做“带最小化物件一起小移动”；若只是猫自己动，则必须回到原位。

实现边界：

1. 只切当前 tier 的 click GIF 一轮。
2. 不移动 return-ball 容器。
3. 不移动聊天窗或聊天球。
4. return、tier change、drag start、hover enter 必须清理 timer。
5. 默认频率要低，不做活泼度配置，除非未来证明这项确实有价值。

## 推荐方向 F：扒边 / 挂边动作

这类可以做，但必须做成局部表演，不是真正爬屏幕或爬窗口。

可做形式：

1. 猫在 return-ball 边缘扒一下，像挂在边上。
2. 猫从最小化物件边缘探头，再缩回去。
3. 猫短暂坐到最小化聊天框物件上沿，但不改变点击区域。

实现边界：

1. 只作用在 return-ball / 最小化物件的视觉层，不改变页面或桌面窗口结构。
2. 不做屏幕边缘爬行，不依赖屏幕边界碰撞。
3. 不把普通窗口、输入框、页面元素当作可攀爬目标。
4. 动作结束后恢复当前 tier 或停在明确的新视觉姿态。
5. drag、return、tier change、chat expand 必须能立即打断并清理。

价值：它是“猫动作”，不是单纯换 GIF；用户能看出猫和当前入口发生了关系。

## 推荐方向 G：专注 / 休息联动

这类是完全不同方向，但要保持轻量。它不做养成数值，只利用已有 idle/goodbye 语义表达“她在陪你休息”。

可做形式：

1. 用户长时间没有回来，CAT2/CAT3 更安静，进入更明显的休息姿态或 overlay。
2. 用户点击回来时，先播放极短“醒来/抬头”视觉，再走原 return 链或在 return 链前做可取消过渡。
3. 如果系统已有工作/语音/任务忙碌状态，猫保持静止，不触发额外动作，避免打扰。
4. 未来如果做专注计时，只影响视觉频率，不引入宠物饥饿、心情、经验值这类系统。

实现边界：

1. 不新增 care meter、亲密度、任务奖励。
2. 不主动弹提醒，不遮挡输入。
3. 不把用户工作状态写进猫的永久状态。
4. 只消费已有 idle / busy / return 信号。

价值：它不是一个新动作，而是把已有三态猫变成更明确的“离开后陪伴/回来时醒来”体验。

## 推荐方向 H：轻微看向用户

这类可以做，但不能变成追鼠标。

可做形式：

1. CAT1/CAT2 偶尔抬头看屏幕外。
2. hover 前后短暂看向用户方向，再回默认状态。
3. 用户回来前，猫抬头一下，像注意到你。

实现边界：

1. 不实时追踪鼠标。
2. 不按鼠标每次移动重播 GIF。
3. 不改变 return-ball 坐标。
4. 最好使用单独短 GIF 或 overlay，不复用 CAT1 走路资源。
5. 触发频率必须低，且可被 drag/return/tier change 清理。

价值：它提供“她注意到你”的感觉，但不需要追鼠标这种高副作用设计。

## 资产型动作

以下不是工程方案，本质是美术 GIF 需求。可以做资源规格和接入规范，但不能靠工程本身产生高质量成品：

1. 洗脸 / 梳毛。
2. 打哈欠。
3. loaf 姿势。
4. 原地伸懒腰变体。
5. 被吵醒又趴回去。
6. 拖拽松手落地动作。

如果未来有对应 GIF，可以作为某个 tier 的短动作接入；在没有资源前，不应把它们写成待实现功能。

## 不推荐方向

以下方向仍然不建议做，因为它们副作用大，可见价值弱：

1. CAT1 等待期朝向、观察聊天球、无语义小抖动。
2. CAT1 贴近聊天球后继续加额外动作。
3. 拖拽分级、短等待概率。
4. 聊天球被拖动时让猫实时跟随或持续观察。
5. 摸头、追鼠标、实时看鼠标。
6. 抢鼠标、拖窗口、遮挡输入框、随机弹图。

## 优先级

建议优先级：

1. `P0`：CAT3 `zzz` / CAT2 小气泡 overlay。
2. `P0`：最小化聊天框物件皮肤第一版，优先选 1-2 个最清晰物件。
3. `P1`：CAT1 和最小化物件一起小移动，优先网页端验证。
4. `P1`：扒边 / 挂边动作，限定在 return-ball 和最小化物件视觉层。
5. `P1`：轻微看向用户，禁止实时追鼠标。
6. `P1`：CAT2/CAT3 小场景底座。
7. `P1`：CAT2/CAT3 低频 click GIF 自主动作。
8. `P2`：专注 / 休息联动，只消费已有 idle/busy/return 信号。
9. `P2`：更多物件皮肤主题，但必须先证明第一版皮肤不会破坏桌面端尺寸和命中区域。

## 验收标准

1. 用户不用解释也能看懂新增视觉含义。
2. 点击猫仍立即走原 return 事件。
3. 点击最小化聊天框仍走原聊天框恢复逻辑。
4. 拖拽猫或聊天框时，不残留 overlay、物件皮肤错位或旧 timer。
5. CAT1 走向聊天球、CAT2/CAT3 停靠、桌面端 shrink/restore 不受影响。
6. 猫单独小移动必须回原位；猫和最小化物件一起移动时，可以停在新位置，但不能触发额外寻路。
7. 所有新增资源都有固定安全区，不导致裁剪、缩放、命中区域漂移。
8. 默认不移动用户窗口、不抢鼠标、不遮挡输入。

## 不做的事

1. 不把 `cat-idle-cat4-*` 提升成 `CAT4`。
2. 不把桌宠动作塞进 goodbye / return 主业务链路。
3. 不把屏幕内容识别作为触发条件。
4. 不默认做恶作剧式桌宠行为。
5. 不为了“更像桌宠”增加看不明显但生命周期复杂的动作。
