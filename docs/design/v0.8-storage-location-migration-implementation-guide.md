# v0.8 存储位置首启迁移实施指导（前端骨架优先）

> 本文是 [docs/design/v0.8-storage-location-migration-design.md](docs/design/v0.8-storage-location-migration-design.md) 的落地指导文档。
>
> 主设计文档回答“目标方案是什么、约束是什么”；本文回答“代码应该按什么顺序改、第一阶段先做什么、哪些内容必须后置”。

---

## 0. 文档定位

本文只讨论“如何实施”，不重写主设计文档中的目标布局与术语定义。

### 0.1 当前实施状态

截至当前分支，本文 `Stage 1` 已完成首轮落地，完成范围如下：

- launcher 已在 `phase-0` 增加首启存储位置 probe gate
- probe 发生在 `_prepare_cloudsave_runtime_for_launch()` 之前
- 当前仓库内已提供可运行的原生启动窗口骨架与 `storage_location_*` 事件发射
- 本阶段仍保持 `probe_only` 语义，不执行真实迁移，也不切换最终运行根

后续 `Stage 2-Stage 5` 仍按本文顺序继续推进。

实施时应同时遵守以下约束：

- 首启路径选择必须发生在 launcher phase-0，早于任何服务进程首次 `get_config_manager()`
- 第一阶段先把“最初启动时的前端流程”搭出来，不要求立刻具备真实迁移能力
- 不支持运行中热切换根目录
- 发布态主验收路径仍是 `frozen + Steam 启动` 与 `frozen + 桌面启动器启动`
- 所有新增用户可见文案都必须走 i18n；若由 JS 渲染，遵守 `window.t('key')` + 中文回退
- 持久化日志仍只能走现有 `utils/logger_config.py` / `setup_logging(...)` 体系，不新增自定义文件 handler
- 详细绝对路径、失败堆栈、`path_fingerprint` 一类信息只进入诊断链路，不要默认混入通用状态或远程同步数据

## 1. 为什么要先做前端骨架

这项功能真正难的不是“复制目录”，而是“在正确的启动时机拿到用户选择，并把这次选择稳定传播给后续进程”。

如果一开始就直接实现 `storage_policy`、双根、迁移作业，最容易踩中的问题反而是：

- UI 还没出来，`main_server.py` / `memory_server.py` 已经按旧根完成 import
- 首启选择依赖主服务 Web 页面，结果路径绑定已经晚了
- 迁移逻辑和启动链路耦合过深，问题出现时无法区分是“流程错了”还是“迁移错了”

因此建议把实施拆成两个大里程碑：

1. 先搭出“启动最早阶段的前端流程骨架”，哪怕后端只是 no-op，也要先验证流程顺序正确
2. 再在这个已经可用的骨架上，逐步补齐 `storage_policy`、双根绑定、真实迁移和恢复逻辑

这能把风险从“同时改 UI、启动链路、配置绑定、迁移”拆成更容易验证的小步。

## 2. Stage 1：先把首启前端骨架跑通

### 2.1 阶段目标

这一阶段的目标只有一个：

- 在最初启动时，launcher 能停在 phase-0，桌面壳能显示存储位置选择前端，用户完成操作后流程再继续

这一阶段刻意不追求真实功能完整，重点是证明以下事情成立：

- 前端确实早于 `_prepare_cloudsave_runtime_for_launch()` 出现
- 服务进程不会在用户做选择前提前启动
- launcher 和桌面壳之间已经具备稳定的“请求/响应”握手机制

### 2.2 这一阶段明确不做什么

Stage 1 不做以下事情：

- 不落最终版 `storage_policy.json`
- 不改 `ConfigManager` 根目录绑定语义
- 不做真实目录迁移
- 不让“自定义路径”真正生效
- 不引入新的启动后 REST API

这一阶段允许采集“用户选了什么路径”，但该路径只作为流程演练结果，不改变当前运行根。

### 2.3 推荐实现锚点

建议直接复用当前已有的 launcher 事件桥：

- `launcher.py:_bootstrap_launcher_runtime(...)`
- `launcher.py:emit_frontend_event(...)`
- `launcher.py:main()`
- `launcher.py:_prepare_cloudsave_runtime_for_launch()`

推荐插入点：

1. `emit_frontend_event("startup_begin", ...)` 之后
2. `_prepare_cloudsave_runtime_for_launch()` 之前
3. 严格保证在任何服务子进程启动之前完成这段交互

换句话说，Stage 1 的“首启前端骨架”本质上是一个新的 launcher phase-0 gate。

### 2.4 推荐握手机制

现有 `emit_frontend_event(...)` 已经提供 launcher -> 桌面壳的单向事件桥；Stage 1 需要补的是桌面壳 -> launcher 的返回通道。

推荐优先使用“临时响应文件”而不是直接把首次交互绑到 HTTP 或主服务 IPC：

1. launcher 生成本次启动专属的 `response_file`
2. launcher 发出 `storage_location_required`
3. 桌面壳展示前端并等待用户操作
4. 桌面壳把选择结果写回 `response_file`
5. launcher 轮询或阻塞等待结果，再决定继续或退出

推荐原因：

- 不依赖主服务是否已经起来
- 不依赖额外端口
- 适合 `frozen + Steam` 与 `frozen + 桌面启动器` 两条主路径
- 出问题时更容易在 diagnostics 里还原一次启动到底停在了哪一步

Stage 1 的响应文件建议只作为一次性握手数据，不要直接承担长期持久化语义。

### 2.5 Stage 1 事件协议建议

建议新增事件：

- `storage_location_required`
- `storage_location_cancelled`
- `storage_location_probe_completed`

`storage_location_required` 的 payload 建议至少包含：

- `launch_id`
- `stage`: 固定为 `probe`
- `response_file`
- `recommended_runtime_root`
- `legacy_sources`
- `has_existing_runtime_content`
- `cloudsave_root`
- `allow_custom_directory_picker`

Stage 1 返回结果建议只保留很小的动作集：

- `action = use_default`
- `action = choose_custom`
- `action = cancel_startup`

如果选择 `choose_custom`，可以允许桌面壳调用原生目录选择器，返回一个真实路径字符串；但 launcher 在 Stage 1 只记录它，不真正切换到该路径。

### 2.6 前端页面应该放在哪里

这一阶段最重要的一条实现纪律是：

- 不要把首启前端做在 `templates/` 里并依赖 `main_server`

原因很简单：

- `templates/` 需要服务启动后才能访问
- 而首启路径选择必须发生在服务启动前

正确做法应是：

- 把页面放到桌面壳可直接加载的本地资源里
- 让它由 launcher stdout 事件驱动，而不是由主服务路由驱动

实现上可以复用现有桌面壳的 preload / IPC 体系，但页面本身应是“本地启动页”而不是“后端页面”。

### 2.7 前端页面范围建议

Stage 1 的 UI 只要覆盖以下最小闭环即可：

- 显示“推荐存储位置”
- 若检测到 legacy root，显示“发现旧数据”
- 提供“使用推荐位置”“选择其他位置”“取消启动”三个动作
- 显示当前处于“流程演练阶段”，不会立刻迁移数据

这时不需要出现：

- 迁移进度条
- 自动清理旧数据 checkbox
- 复杂恢复分支
- 设置页入口

### 2.8 i18n 与视觉约束

虽然只是骨架，这个启动前端仍然要按正式页面标准做：

- 文案进入 `static/locales/*.json`
- JS 文案走 `window.t('key')` + 中文回退
- 页面样式尽量复用项目现有视觉语言，不做一次性临时样式

如果本地启动页无法直接复用现有页面的 i18n 初始化逻辑，建议直接加载 `static/locales/*.json` 作为单一翻译源，并在启动页侧提供兼容 `window.t(...)` 的薄封装，而不是额外硬编码一套启动页文案。

建议在 Stage 1 就把以下 key 预留出来：

- `storageMigration.startup.title`
- `storageMigration.startup.subtitle`
- `storageMigration.startup.recommended`
- `storageMigration.startup.detectedLegacy`
- `storageMigration.startup.useDefault`
- `storageMigration.startup.chooseCustom`
- `storageMigration.startup.cancel`
- `storageMigration.startup.probeNotice`

### 2.9 Stage 1 验收标准

达到以下条件，就算 Stage 1 完成：

- `frozen + Steam 启动` 时能在启动最早阶段看到存储位置前端
- `frozen + 桌面启动器启动` 时行为一致
- 用户未完成操作前，launcher 不会启动服务进程
- 选择“取消启动”时，launcher 能干净退出
- 选择“推荐位置”或“自定义位置”后，launcher 能继续沿现有根目录完成启动
- 发布态下若本地首启 UI 无法拉起，launcher 不能静默跳过交互直接改写运行根；除“安全默认继续”被明确定义的窄场景外，应中止启动并给出可见错误
- 整个阶段不引入真实迁移副作用

## 3. Stage 2：补上 `StorageLayout` / `storage_policy` 与启动绑定

Stage 1 证明流程顺序正确之后，再开始落真正的存储布局模型。

### 3.1 这一阶段要交付什么

- 新增 `utils/storage_policy.py` 或等价模块
- 定义 `StorageLayout`
- 把 `anchor_root / selected_root / cloudsave_root` 的解析规则从 `ConfigManager` 中独立出来
- 让 launcher 在 phase-0 落盘 `storage_policy.json`
- 让父子进程通过 `NEKO_STORAGE_*` 共享同一份布局

### 3.2 `StorageLayout` 建议职责

建议把以下职责放进单独模块，而不是继续散在 `ConfigManager` 与 launcher 中：

- 解析 `anchor_root`
- 读取/写入 `storage_policy.json`
- 校验 `selected_root`
- 从环境变量恢复本次启动布局
- 向子进程导出 `NEKO_STORAGE_*`

这样 Stage 2 的核心目标不是“迁移”，而是“让所有进程在 import 前看到同一份布局”。

### 3.3 launcher 改造顺序建议

推荐顺序：

1. launcher 先解析 `anchor_root`
2. 读取 `storage_policy.json`
3. 若缺少策略且需要首启交互，则调用 Stage 1 已有前端流程
4. 用户确认后，生成正式 `StorageLayout`
5. 落盘 `storage_policy.json`
6. 写入 `NEKO_STORAGE_*`
7. 再进入 `_prepare_cloudsave_runtime_for_launch()`
8. 再启动 `main_server.py` / `memory_server.py`

注意：

- 这时才允许“推荐位置”或“用户自定义位置”真正参与运行根绑定
- 但即使 Stage 2 完成，也还没有要求真实迁移必须同时完成

### 3.4 `ConfigManager` 改造边界

这一阶段对 `ConfigManager` 的要求是：

- `app_docs_dir` 语义改为 `selected_root`
- `runtime_root` 与 `app_docs_dir` 对齐
- `anchor_root` 单独暴露
- `cloudsave_dir`、`local_state_dir` 等锚点相关目录改由 `anchor_root` 推导

禁止做法仍然不变：

- 不对已创建的 `ConfigManager` 实例做热重绑定
- 不靠 monkey patch 修补 import 时已经缓存下来的旧路径

### 3.5 Stage 2 验收标准

- 首次 `get_config_manager()` 前，`StorageLayout` 已经确定
- 父进程与子进程读取到相同的 `selected_root`
- `IS_FROZEN=True` 的发布态和源码态都走相同的布局解析主路径
- `cloudsave_root` 仍固定在 `anchor_root`

## 4. Stage 3：收敛关键读写路径到运行时真源

只有 Stage 2 做完，双根模型才真正开始有意义；否则很多模块仍会偷偷读回项目默认配置或旧语义路径。

### 4.1 这一阶段必须优先处理的点

- `utils/config_manager.py:load_characters()` 不再走 `get_config_path("characters.json")`
- `utils/config_manager.py` 里 `core_config.json` 读取改为只读运行时真源
- `utils/preferences.py` 去掉模块 import 时缓存的 `ConfigManager` / 路径
- `utils/autostart_prompt_state.py`
- `utils/tutorial_prompt_state.py`
- `utils/holiday_cache.py`

### 4.2 日志与诊断路径的收敛要求

这一阶段还要处理路径一致性问题：

- `logs/diagnostics/` 必须统一跟随当前 `selected_root`
- 诊断文件不能一部分跟 logger 基目录，一部分跟 `app_docs_dir / logs / diagnostics`
- 若通用日志目录暂时仍未完全迁入 `selected_root`，也不能再让诊断目录继续分裂

实现约束：

- 诊断文件继续走统一 logger/diagnostics 体系
- 不允许为迁移功能单独引入新的诊断写盘实现

### 4.3 Stage 3 验收标准

- 不出现“写新根、读项目默认配置”的分裂
- 不出现 `preferences` 这类模块在 import 时先绑错根
- `logs/diagnostics/` 在新根下统一懒创建

## 5. Stage 4：补齐真实迁移、恢复与启动后续流程

到这一步，才开始实现真正的数据迁移。

### 5.1 推荐顺序

应直接复用主设计文档里的总语义：

- `clone + verify + rebind + retain`

默认语义仍然是：

- 迁移成功后不自动删除旧根
- `cloudsave/`、`state/` 继续留在 `anchor_root`
- 旧数据保留失败不反向否决已经成功的新根绑定

### 5.2 这一阶段的关键实现件

- `storage_migration.json` 检查点
- 迁移 worker / orchestration 函数
- `storage_location_progress`
- `storage_location_failed`
- `storage_location_ready`
- 启动中断后的恢复分支

实现时要继续保持一个底线：

- 首启 UI 负责“收集选择和展示进度”
- 真正的迁移提交逻辑仍由 launcher/pre-bootstrap 侧掌控

### 5.3 迁移期间的前端要求

这时才需要把 Stage 1 的骨架扩成完整前端：

- 预检结果展示
- 复制/校验/提交/保留旧数据的阶段文案
- 可观测的进度条
- 失败后的重试或恢复入口
- 成功后的“打开当前数据目录 / 打开旧备份目录 / 稍后清理”入口

### 5.4 `POST /api/storage/location/apply` 的正确语义

即使将来要补启动后的设置入口，也不要把它设计成“主服务内热切换根目录”。

推荐语义应始终是：

- 记录迁移意图
- 返回 `restart_required` / `handoff_to_launcher`
- 真正执行仍在下次 launcher/pre-bootstrap 中完成

### 5.5 Stage 4 验收标准

- 自定义 `selected_root` 能真实生效
- legacy root 能被正确迁入新根
- 失败时不会把 `storage_policy` 落成坏状态
- 中断后能进入可恢复状态，而不是静默退回旧逻辑继续写

## 6. Stage 5：补齐设置页、清理入口与诊断接口

当前端骨架、双根绑定和真实迁移都稳定后，再补完整的启动后运维能力。

### 6.1 这一阶段建议补的内容

- `GET /api/storage/location/status`
- `GET /api/storage/location/diagnostics`
- `GET /api/storage/legacy_data`
- `DELETE /api/storage/legacy_data`
- 设置页“当前存储位置 / 旧数据备份 / 手动清理”入口

### 6.2 这一阶段的约束

- 通用状态接口默认不扩散全部绝对路径
- 物理删除只允许删除受跟踪 backup，不接受任意绝对路径
- 旧数据清理失败只留下可见告警，不改变当前 `selected_root`

---

## 7. 建议的提交拆分

为了降低回归风险，建议按下面顺序拆 PR 或 commit。

### 7.1 Commit 1：首启前端占位页

- 新增桌面壳本地启动页
- 接入 i18n key
- 只渲染页面，不要求阻塞 launcher

### 7.2 Commit 2：launcher phase-0 gate

- 新增 `storage_location_required`
- 新增响应文件握手
- 让用户操作真正决定“继续启动还是取消”
- 仍不改变运行根

### 7.3 Commit 3：`StorageLayout` 与 `storage_policy`

- 新增 `utils/storage_policy.py`
- launcher 落正式策略
- 子进程通过 `NEKO_STORAGE_*` 恢复布局

### 7.4 Commit 4：关键读写收敛

- 收敛 `characters.json` / `core_config.json`
- 清理 import 时路径缓存
- 收敛 diagnostics 路径

### 7.5 Commit 5：真实迁移与恢复

- 接入 `storage_migration.json`
- 接入迁移 worker 与阶段进度
- 接入恢复分支

### 7.6 Commit 6：设置页与清理

- 新增状态 API
- 新增 legacy cleanup
- 补足手工回归与发布态验收

## 8. 实施时最容易做错的几件事

- 把首启前端放进 `templates/`，导致它依赖主服务启动后才能访问
- 在 Stage 1 就提前落长期 `storage_policy`，把未定型 schema 固化
- 让 launcher 在收到用户选择前就开始 `_prepare_cloudsave_runtime_for_launch()` 或启动子进程
- 只在 source 模式下测试通过，没有验证 `frozen + Steam` 主路径
- 迁移功能自己另起一套日志写盘
- 忘记处理 `utils/preferences.py` 这类 import 时路径缓存模块
- 让 `POST /api/storage/location/apply` 变成“服务内立即切根”

---

## 9. 最终完成定义

当以下条件同时成立时，可以认为这份实施指导已经被完整落地：

- 启动最早阶段存在独立于主服务的存储位置前端
- launcher 在首次 `get_config_manager()` 前完成布局决策
- `storage_policy`、`root_state`、`storage_migration` 三者职责边界清晰
- 父子进程共享同一份 `StorageLayout`
- 关键配置读取链路只读运行时真源
- `logs/diagnostics/` 不再分裂
- 发布态首启、重启、迁移失败恢复都能通过验收
