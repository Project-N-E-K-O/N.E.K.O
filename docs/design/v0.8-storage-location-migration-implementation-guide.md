# v0.8 存储位置首启迁移实施指导（NEKO 网页端与后端主链路）

> 本文是 [docs/design/v0.8-storage-location-migration-design.md](docs/design/v0.8-storage-location-migration-design.md) 的实施文档。
>
> 本文只讨论这条已经定下来的实施主线：
>
> - 第一次启动先正常进入应用
> - 应用打开后先做轻量状态探测，必要时再显示存储位置选择弹窗或页面
> - 若用户改到不同路径，则关闭当前实例
> - 关闭后执行迁移
> - 迁移完成后自动重启
>
> 当前文档范围再补一条硬边界：
>
> - 本文覆盖 NEKO 主项目内的网页端覆盖层、后端接口、迁移检查点、启动前迁移和跨进程布局传播
> - 桌面端 Electron 窗口承载、loadingWindow 与卫星窗口编排，已移动到 NEKO-PC 项目内的 `docs/v0.8-storage-location-migration-window-display-design.md`

本文回答三件事：

- 基于当前仓库事实，这条流程在 NEKO 主项目里已经如何落地

## 1. 当前仓库已确认现状

### 1.1 当前启动链路的关键事实

当前仓库整体启动链路会经过 `launcher.py` 与 cloudsave 启动准备，当前实现已经把“网页端应用内选择”和“关闭后启动前迁移”接在同一条主线上：

- 前端页面正常打开
- Web UI 先用轻量状态接口判断当前是否必须进入存储引导
- 若需要进入引导，再在应用内请求 bootstrap 并完成首轮选择
- 若用户选择不同路径，前端先展示预检确认，再调用 restart 接口
- 后端写入检查点并受控关闭当前实例
- `launcher.py` 在新服务启动前执行待迁移检查点并传播最终布局

桌面端启动下的 Electron 窗口承载差异，暂不在本文展开。

这意味着：

- “迁移执行”仍然适合放在“当前实例退出后、新实例正式拉起前”
- “首次让用户选择路径”则应发生在当前应用界面里

本次实施的核心转向是：

- 首次选择改到应用内
- 迁移动作保留在重启前后边界

### 1.2 当前已经具备双根布局与检查点模型

当前 `ConfigManager` 仍以 `app_docs_dir` 作为大量历史调用点的兼容入口，但实际代码已经具备双根布局：

- `anchor_root` 固定在平台标准应用数据目录
- `selected_root` 承载运行时用户数据
- `cloudsave_root = anchor_root / "cloudsave"`
- `cloudsave_root` 不跟随 `selected_root` 迁移

已落地模块包括：

- `utils/storage_policy.py`
- `utils/storage_migration.py`
- `utils/storage_layout.py`
- `main_routers/storage_location_router.py`
- `static/app-storage-location.js`

已落地状态文件包括：

- `anchor_root/state/storage_policy.json`
- `anchor_root/state/storage_migration.json`
- `anchor_root/state/root_state.json`

这意味着本文里的“关闭后迁移”不能理解为“把整个当前根原样挪到新位置”，而必须理解为：

- 运行时目录迁移到新的 `selected_root`
- 锚点层目录继续固定在 `anchor_root`

### 1.3 当前已经接入的基础设施

当前仓库已经具备并接入的能力：

- `bootstrap_local_cloudsave_environment(config_manager)`
- `import_legacy_runtime_root_if_needed(config_manager)`
- `cloud_apply_fence(...)`
- `ROOT_MODE_DEFERRED_INIT`
- `ROOT_MODE_MAINTENANCE_READONLY`
- `storage_policy.first_run_completed`
- `storage_migration.status`
- `run_pending_storage_migration(config_manager)`
- `resolve_storage_layout(config_manager)`
- `export_storage_layout_to_env(layout)`
- `root_state.current_root`
- `root_state.last_known_good_root`
- `root_state.last_migration_source`
- `root_state.last_migration_result`
- `root_state.last_migration_backup`
- `root_state.legacy_cleanup_pending`

这意味着 NEKO 主项目当前已经具备：

- 应用启动后如何把“当前根 / 推荐根 / 旧数据根”交给前端展示
- 用户确认不同路径后，如何把当前实例平滑关掉
- 关闭后如何根据检查点执行迁移并自动重启
- 新实例如何在任何服务 import 前恢复最终布局

剩余更偏向持续收敛与桌面端承载：

- 继续治理隐式回退读取链路
- 扩展 diagnostics 覆盖面
- 在 NEKO-PC 中补齐 Electron loadingWindow 与卫星窗口保护

### 1.4 为什么不能在当前会话里直接切根

当前实例一旦启动，很多模块已经绑定当前根目录：

- `get_config_manager()` 是进程内单例
- `main_server.py` / `memory_server.py` import 链路会用到运行时路径
- 部分模块仍存在 import 时缓存路径的情况

因此不能采用：

- 当前页面点一下后，当前实例立即切到新根继续跑

可接受做法只有：

- 当前实例继续使用当前根
- 若用户改选到不同路径，则当前实例退出

这里要再强调一层主设计里的硬约束：

- “下一次会话中正式生效”不等于“新实例起来后再慢慢修”
- 正确语义是：新实例里任何服务首次 `get_config_manager()` 前，就必须先恢复最终布局

否则即使用户流程正确，后端仍会再次绑到旧路径。

## 2. 目标实施流程

### 2.1 第一次进入应用

应用先正常启动并打开 UI。

网页端当前收口后的推荐顺序是：

1. 页面尽早请求轻量哨兵接口（如 `GET /api/v1/system/status`）
2. 若返回 `ready`，则放行页面配置与主业务初始化
3. 若返回需要阻断的存储状态，则继续请求存储初始化接口

随后前端拿到：

- `current_root`
- `recommended_root`
- `legacy_sources`
- `selection_required`
- `migration_pending`
- `recovery_required`

若任一阻断条件成立：

- `selection_required=true`
- `migration_pending=true`
- `recovery_required=true`

则立即显示弹窗或全页覆盖层，并阻止用户进入普通使用流程。

### 2.1.1 前移弹出时机的真实语义：受限启动，而不是完整业务启动

这里要补一条很容易被误解的约束：

- “应用窗口已经出现”不等于“本次启动已经允许主业务完整运行”

在网页端方案下，前移后的首轮存储页本质上仍然是一种“受限启动”：

- 允许先启动：
  - 网页端壳本身
  - 静态资源与基础 i18n
  - `GET /api/v1/system/status`
  - `GET /api/storage/location/bootstrap`
  - 只读的 `storage_policy` / `root_state` / legacy source 诊断
- 明确禁止先启动：
  - `pageConfig` 放行后的主业务初始化
  - 会在 import 或启动阶段绑定路径并长期持有的重型模块
  - 依赖当前运行根持续写入的后台任务、记忆链路与模型初始化
  - 任何会把“当前根已经正式可用”当成既成事实的写入流程

因此，实现上应把以下状态视为“网页端受限模式”：

- `selection_required=true`
- `migration_pending=true`
- `recovery_required=true`
- 或等价的生命周期状态：`MIGRATION_PENDING`

只有当存储引导明确放行后，才允许：

- 页面配置开始加载
- 主功能 UI 开始初始化
- 高风险写链路恢复

这条语义即使在“应用先正常打开，随后才弹页”的新流程下也不能丢，否则会退化回“窗口先开了，但后端其实已经先把旧路径绑死”的旧问题。

### 2.2 用户确认保持当前路径

如果用户最终选择的路径与 `current_root` 一致：

- 后端写入 `storage_policy.json`
- 标记首启选择完成
- 前端关闭弹窗
- 当前会话继续运行

### 2.3 用户确认切换到不同路径

如果用户确认的目标路径与 `current_root` 不一致：

- 后端写入 `storage_migration.json(status=pending)`
- 同时保存 `source_root` 与 `target_root`
- 前端展示“将关闭当前实例并迁移数据”的确认页

用户点击确认后：

1. 后端进入受控关闭流程
2. 当前实例退出
3. 迁移器根据检查点执行真实迁移
4. 迁移成功后自动重启
5. 新实例按新路径运行

## 3. 当前落地情况与后续核对点

原实施文档里的 Stage 1-5 已经基本落到当前代码中。现在更准确的口径不是“建议新增”，而是“确认现有链路和剩余边界”。

### 3.1 网页端应用内覆盖层

当前已落地：

- `static/app-storage-location.js` 在页面启动早期挂载存储位置覆盖层。
- 前端先调用 `GET /api/v1/system/status`。
- 若 `selection_required`、`migration_pending`、`recovery_required` 任一成立，再调用 `GET /api/storage/location/bootstrap`。
- 覆盖层包含 `loading`、`selection_required`、`maintenance`、`error` 四个阻断子视图。
- `completion_notice` 作为非阻断完成提示卡片显示。
- 目录选择优先使用可选的通用宿主能力 `window.nekoHost.pickDirectory({ startPath })`；宿主能力不可用或失败时，回退后端 `POST /api/storage/location/pick-directory`。
- `static/app.js` 等待 `window.appStorageLocation.init()` 后才启动主业务。
- `static/js/index.js` 等待 `__nekoStorageLocationStartupBarrier` 后才开始 `pageConfig`。

当前约束：

- 首屏只突出当前路径、推荐路径和主要动作。
- 旧数据目录、手动路径和文件夹选择放在“其他位置”面板。
- 覆盖层未解除前，主页主功能、React Chat 独立启动请求等不得抢跑。

### 3.2 `storage_policy` 与首轮提交接口

当前已落地：

- `utils/storage_policy.py`
- `GET /api/storage/location/bootstrap`
- `POST /api/storage/location/select`

`storage_policy` 当前保存：

- `anchor_root`
- `selected_root`
- `selection_source`
- `cloudsave_strategy=fixed_anchor`
- `first_run_completed`
- `updated_at`

`POST /api/storage/location/select` 当前返回两类结果：

- `continue_current_session`
- `restart_required`

关键行为：

- 选择当前路径时写入稳定策略并释放受限启动闸门。
- 选择不同路径时只返回预检与 `restart_required`，不提前提交新根。
- 恢复态下区分“继续当前可用根”“关闭后重连原路径”“禁止切到第三路径”。

### 3.3 迁移检查点、预检与受控关闭

当前已落地：

- `utils/storage_migration.py`
- `anchor_root/state/storage_migration.json`
- `POST /api/storage/location/restart`
- `GET /api/storage/location/status`

检查点当前支持：

- `pending`
- `preflight`
- `copying`
- `verifying`
- `committing`
- `retaining_source`
- `rollback_required`
- `failed`
- `completed`

预检当前返回：

- `estimated_required_bytes`
- `target_free_bytes`
- `permission_ok`
- `warning_codes`
- `blocking_error_code`
- `blocking_error_message`

当前后端路径归一化约束：

- `selection_source=custom` 且用户选择的目录名不是 `N.E.K.O` 时，最终 `selected_root` / `target_root` 会归一化为 `<用户选择目录>/N.E.K.O`。
- 若用户已经选择 `N.E.K.O` 目录，则按该目录作为最终运行根。
- 最终目标根已有运行时数据时，`POST /api/storage/location/select` 就必须返回 `blocking_error_code=target_not_empty`，前端确认按钮禁用，不能等到重启后的迁移阶段才失败。

受控关闭当前通过 `main_routers.shared_state.get_request_app_shutdown()` 接入 `main_server.py` 的关闭流程。

### 3.4 关闭后迁移、自动重启与布局传播

当前已落地：

- `launcher.py` 启动早期调用 `_resolve_storage_layout_for_launch()`。
- `_resolve_storage_layout_for_launch()` 会先执行 `run_pending_storage_migration(config_manager)`。
- 迁移完成后通过 `resolve_storage_layout(config_manager)` 解析最终布局。
- `export_storage_layout_to_env(layout)` 写入：
  - `NEKO_STORAGE_SELECTED_ROOT`
  - `NEKO_STORAGE_ANCHOR_ROOT`
  - `NEKO_STORAGE_CLOUDSAVE_ROOT`
- `ConfigManager` 会在初始化时读取这些环境变量，确保服务首次绑定最终布局。
- 迁移或 rebind 需要重启时，`launcher.py` 会发出 `storage_migration_restart` 事件并拉起新的 launcher。

这已经覆盖主设计中最关键的约束：

- 迁移晚于当前实例退出。
- 迁移早于新服务重型 import。
- 新实例首次 `get_config_manager()` 前拿到最终布局。

### 3.5 迁移完成提示与旧目录清理

当前已落地：

- `GET /api/storage/location/status` 返回 `completion_notice`。
- 网页端在 ready 后轮询完成提示。
- 若旧目录仍保留，显示非阻断完成提示卡片。
- `POST /api/storage/location/retained-source/cleanup` 负责显式清理旧目录。

仍然保持的设计约束：

- 旧数据默认保留。
- 清理必须由用户明确触发。
- 清理旧目录不是迁移主流程的一部分。

## 4. 前后端职责划分

### 4.1 前端负责什么

前端负责：

- 在应用打开后立即请求存储状态
- 显示首轮选择弹窗或页面
- 展示“将关闭当前实例并迁移”的确认页
- 在服务真空期展示维护页，并保持自动重连
- 在重启后展示成功 / 失败结果

在本文当前范围里，这里的“前端”特指 NEKO 主项目网页端前端，不包含 NEKO-PC 的 Electron 专属窗口外壳。

目录选择这里要保持一条额外边界：

- 网页端只识别通用 `window.nekoHost.pickDirectory` 能力，不识别 NEKO-PC 或 Electron 专有对象。
- 宿主返回的路径只作为用户输入填入“其他位置”，不能被视为已通过校验。
- 后续预检、保存策略、写迁移检查点和触发重启，仍然必须经过 NEKO 后端接口。
- 迁移完成卡片若检测到通用 `window.nekoHost.openPath`，可以提供“打开当前目录 / 打开旧目录”便利按钮；路径来源必须是后端 `completion_notice`，清理旧目录仍只走后端 cleanup 接口。

### 4.2 后端负责什么

后端负责：

- 返回当前根、推荐根、旧数据根
- 保存 `storage_policy`
- 保存 `storage_migration`
- 触发受控关闭
- 在关闭后执行迁移
- 自动重启
- 在新实例里于首次重型 import 前恢复最终布局

### 4.3 明确禁止什么

不允许让前端自己判断：

- 当前实例是否已经正式切到新根
- 当前是否可以绕过关闭流程直接迁移
- 迁移失败时是否还能继续沿用半切换状态

## 5. 当前代码接入点

### 5.1 主服务接口

当前已整理并接入以下接口：

- `GET /api/storage/location/bootstrap`
- `POST /api/storage/location/select`
- `POST /api/storage/location/restart`
- `GET /api/storage/location/status`
- `POST /api/storage/location/pick-directory`
- `POST /api/storage/location/retained-source/cleanup`

### 5.1.1 网页端必须可见的字段契约

为了让网页端真正承担“意图捕获器”和“状态哨兵”，以下信息不能只存在于后端内部日志里，必须能通过接口返回给前端展示。

`GET /api/storage/location/bootstrap` 最低应提供：

- `current_root`
- `recommended_root`
- `legacy_sources`
- `selection_required`
- `migration_pending`
- `recovery_required`
- `blocking_reason`
- `last_error_summary`

当用户已经选定目标路径、准备进入关闭迁移前，`POST /api/storage/location/restart` 或等价预检接口最低应返回或缓存以下可展示字段：

- `target_root`
- `estimated_required_bytes`
- `target_free_bytes`
- `permission_ok`
- `warning_codes`
- `blocking_error_code`
- `blocking_error_message`

其中 `warning_codes` 至少应覆盖：

- 同步盘目录
- 外置盘 / 网络盘
- Windows CFA / 权限受限目录
- symlink / junction / reparse point
- 目标空间不足

`GET /api/storage/location/status` 最低应提供：

- `lifecycle_state`
- `migration_stage`
- `maintenance_message`
- `poll_interval_ms`
- `effective_root`
- `last_error_summary`

要求：

- “目标卷剩余空间”“预估迁移体量”“权限/风险警告”必须能被网页端明确展示
- 这些检查既是迁移预检，也属于用户可见的决策依据，不能只做内部校验不回传前端

### 5.2 配置与检查点模块

当前已新增：

- `utils/storage_policy.py`
- `utils/storage_migration.py`
- `utils/storage_layout.py`

职责分别是：

- `storage_policy.py`
  - 保存稳定选择
- `storage_migration.py`
  - 保存待迁移检查点和结果
- `storage_layout.py`
  - 统一布局计算与路径规范化

这里应再补一层与主设计一致的职责拆分：

- `storage_policy.py`
  - 只保存当前已正式生效的稳定布局
- `storage_migration.py`
  - 只保存迁移过程检查点、失败信息与恢复信息
- `storage_layout.py`
  - 统一 `anchor_root / selected_root / cloudsave_root` 计算
  - 导出本次启动布局环境变量

路径规范化与路径合法性校验当前主要位于 `utils/storage_policy.py`。

### 5.3 前端入口

前端建议在“应用壳 ready”之后尽快挂载存储位置引导逻辑。

建议形式：

- 全局阻断 modal
- 或专门的 first-run route / overlay

不建议：

- 单独做一个独立于应用主界面的首启入口
- 让用户自己从设置页深处再去找首启入口

当前范围补充：

- 这里默认入口就是网页端主 UI
- 不把桌面端专属启动壳纳入 NEKO 主项目实现范围；该部分在 NEKO-PC 文档中维护

## 6. 迁移执行约束

### 6.1 迁移必须晚于当前实例退出

这是本次实施最重要的技术约束。

原因：

- 当前实例里已经持有旧根相关句柄和缓存
- 运行中切根很容易造成读写分裂

因此迁移必须在：

- 当前实例完全退出后
- 新实例正式拉起前

补充要求：

- 新实例正式拉起后，也不能立刻让服务 import 重型模块
- 必须先读取 `storage_migration`
- 必须先恢复最终布局
- 必须先完成子进程布局传播
- 在页面内触发 restart 后，网页端必须稳定停留在维护页，直到新实例明确返回 `ready`
- 冷启动时若迁移发生在后端页面可用之前，则由 NEKO-PC 的 Electron `loadingWindow` 承担等待展示

### 6.2 成功语义

迁移成功至少意味着：

- 目标根数据完整可用
- `storage_policy.selected_root` 已更新
- 新实例将从新根启动
- 旧根如保留，已被标记为备份

### 6.3 失败语义

迁移失败时必须保证：

- 旧根仍然可回退
- 不把 `storage_policy` 写成半有效状态
- 下次启动仍能给出明确恢复选择

若用户之前已经选过自定义路径，而该路径在后续启动时离线或损坏，也应按主设计进入恢复态，而不是静默回退默认根继续写入。

---

## 7. 最容易做错的点

- 把“应用内弹窗”重新做成独立的首启入口
- 用户选了不同路径后，在当前会话里直接切根
- 把 cloudsave 跟随运行时根一起迁移
- 新实例里先起服务，再去处理 `storage_migration`
- 只写前端页面，不补受控关闭和自动重启
- 迁移失败时把策略写坏，导致新旧路径都不稳定
- 把旧数据自动删除作为默认行为
- 在 NEKO-PC 中重新实现一套迁移状态机，绕过 NEKO 后端接口

---

## 8. 最终完成定义

当下面这些条件全部成立时，才算这份实施文档真正落地：

- 应用第一次启动时先正常打开
- 打开后先做轻量状态探测；若判定需要阻断，则立即显示存储位置选择弹窗或页面
- 选择当前路径时，不重启，当前会话继续
- 在存储页未放行前，主页主功能不会继续初始化
- 选择不同路径时，当前实例关闭后才开始迁移
- 迁移完成后自动重启，不要求用户手工再次打开
- `cloudsave_root` 固定在 `anchor_root`
- 新实例里任何服务首次 `get_config_manager()` 前已拿到最终布局
- 整个流程都围绕当前应用界面、关闭流程和自动重启展开
- 当前 NEKO 主项目交互与验收以网页端覆盖层、后端接口和启动前迁移为准

桌面端启动相关的显示补充、窗口编排和专属体验，不作为本文当前完成定义的一部分，已移动到 NEKO-PC 项目文档维护。

一句话收敛：

- 实施主线就是“应用内完成选择，必要时关闭当前实例，迁移后自动重启”。
