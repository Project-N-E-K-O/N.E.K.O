# v0.8 存储位置首启迁移实施指导（当前仅网页端启动）

> 本文是 [docs/design/v0.8-storage-location-migration-design.md](docs/design/v0.8-storage-location-migration-design.md) 的实施文档。
>
> 本文只讨论这条已经定下来的实施主线：
>
> - 第一次启动先正常进入应用
> - 应用打开后先做轻量状态探测，必要时再显示存储位置选择弹窗或页面
> - 若用户改到不同路径，则关闭当前实例
> - 关闭后执行迁移
> - 迁移完成后自动重启
>
> 当前文档范围再补一条硬边界：
>
> - 本文当前只覆盖“网页端启动模式”下的实现与接线
> - 桌面端启动相关的显示承载与补充交互，放在 `docs/design/v0.8-storage-location-migration-window-display-design.md` 后续补充

## 0. 文档边界

本文回答三件事：

- 基于当前仓库事实，这条流程该怎么在网页端启动模式下落地
- 哪些能力已经可以复用
- 代码应按什么顺序改，才能把风险压低

本文以“应用打开后再选择”的单一方案为前提，不再展开其它入口形态。

本文当前明确不负责：

- 打包后的桌面启动器首启承载方式
- 桌面端原生目录选择器、独立迁移窗口、桌面窗口恢复体验
- Steam / 桌面启动器发布态的最终交互补完

## 1. 当前仓库已确认现状

### 1.1 当前启动链路的关键事实

当前仓库虽然整体启动链路仍会经过 `launcher.py` 与 cloudsave 启动准备，但本文当前实施口径只关心网页端启动后的应用内接线：

- 前端页面正常打开
- Web UI 先用轻量状态接口判断当前是否必须进入存储引导
- 若需要进入引导，再在应用内请求 bootstrap 并完成首轮选择

桌面端启动下的窗口承载差异，暂不在本文展开。

这意味着：

- “迁移执行”仍然适合放在“当前实例退出后、新实例正式拉起前”
- “首次让用户选择路径”则应发生在当前应用界面里

本次实施的核心转向是：

- 首次选择改到应用内
- 迁移动作保留在重启前后边界

### 1.2 当前仍是单根模型

当前 `ConfigManager` 仍以单一 `app_docs_dir` 作为运行根。

当前尚不存在：

- `storage_policy.json`
- `storage_migration.json`
- 正式的 `anchor_root / selected_root / cloudsave_root` 双根模型
- 进程级稳定的 `StorageLayout`

但根据主设计文档，本实施线虽然采用“应用内选择，关闭后迁移”的新流程，最终目标布局仍然已经明确：

- `anchor_root` 固定在平台标准应用数据目录
- `selected_root` 承载运行时用户数据
- `cloudsave_root = anchor_root / "cloudsave"`
- `cloudsave_root` 不跟随 `selected_root` 迁移

这意味着本实施文档里的“关闭后迁移”不能理解为“把整个当前根原样挪到新位置”，而必须理解为：

- 运行时目录迁移到新的 `selected_root`
- 锚点层目录继续固定在 `anchor_root`

### 1.3 当前可以直接复用的基础设施

当前仓库已经具备的可复用能力：

- `bootstrap_local_cloudsave_environment(config_manager)`
- `import_legacy_runtime_root_if_needed(config_manager)`
- `cloud_apply_fence(...)`
- `ROOT_MODE_DEFERRED_INIT`
- `root_state.current_root`
- `root_state.last_known_good_root`
- `root_state.last_migration_source`
- `root_state.last_migration_result`
- `root_state.last_migration_backup`

这意味着后端真正缺的不是“如何 copy 目录”，而是：

- 应用启动后如何把“当前根 / 推荐根 / 旧数据根”交给前端展示
- 用户确认不同路径后，如何把当前实例平滑关掉
- 关闭后如何根据检查点执行迁移并自动重启
- 新实例如何在任何服务 import 前恢复最终布局

### 1.4 为什么不能在当前会话里直接切根

当前实例一旦启动，很多模块已经绑定当前根目录：

- `get_config_manager()` 是进程内单例
- `main_server.py` / `memory_server.py` import 链路会用到运行时路径
- 部分模块仍存在 import 时缓存路径的情况

因此不能采用：

- 当前页面点一下后，当前实例立即切到新根继续跑

可接受做法只有：

- 当前实例继续使用当前根
- 若用户改选到不同路径，则当前实例退出

这里要再强调一层主设计里的硬约束：

- “下一次会话中正式生效”不等于“新实例起来后再慢慢修”
- 正确语义是：新实例里任何服务首次 `get_config_manager()` 前，就必须先恢复最终布局

否则即使用户流程正确，后端仍会再次绑到旧路径。

## 2. 目标实施流程

### 2.1 第一次进入应用

应用先正常启动并打开 UI。

网页端当前收口后的推荐顺序是：

1. 页面尽早请求轻量哨兵接口（如 `GET /api/v1/system/status`）
2. 若返回 `ready`，则放行页面配置与主业务初始化
3. 若返回需要阻断的存储状态，则继续请求存储初始化接口

随后前端拿到：

- `current_root`
- `recommended_root`
- `legacy_sources`
- `selection_required`
- `migration_pending`
- `recovery_required`

若任一阻断条件成立：

- `selection_required=true`
- `migration_pending=true`
- `recovery_required=true`

则立即显示弹窗或全页覆盖层，并阻止用户进入普通使用流程。

### 2.1.1 前移弹出时机的真实语义：受限启动，而不是完整业务启动

这里要补一条很容易被误解的约束：

- “应用窗口已经出现”不等于“本次启动已经允许主业务完整运行”

在网页端方案下，前移后的首轮存储页本质上仍然是一种“受限启动”：

- 允许先启动：
  - 网页端壳本身
  - 静态资源与基础 i18n
  - `GET /api/v1/system/status`
  - `GET /api/storage/location/bootstrap`
  - 只读的 `storage_policy` / `root_state` / legacy source 诊断
- 明确禁止先启动：
  - `pageConfig` 放行后的主业务初始化
  - 会在 import 或启动阶段绑定路径并长期持有的重型模块
  - 依赖当前运行根持续写入的后台任务、记忆链路与模型初始化
  - 任何会把“当前根已经正式可用”当成既成事实的写入流程

因此，实现上应把以下状态视为“网页端受限模式”：

- `selection_required=true`
- `migration_pending=true`
- `recovery_required=true`
- 或等价的生命周期状态：`MIGRATION_PENDING`

只有当存储引导明确放行后，才允许：

- 页面配置开始加载
- 主功能 UI 开始初始化
- 高风险写链路恢复

这条语义即使在“应用先正常打开，随后才弹页”的新流程下也不能丢，否则会退化回“窗口先开了，但后端其实已经先把旧路径绑死”的旧问题。

### 2.2 用户确认保持当前路径

如果用户最终选择的路径与 `current_root` 一致：

- 后端写入 `storage_policy.json`
- 标记首启选择完成
- 前端关闭弹窗
- 当前会话继续运行

### 2.3 用户确认切换到不同路径

如果用户确认的目标路径与 `current_root` 不一致：

- 后端写入 `storage_migration.json(status=pending)`
- 同时保存 `source_root` 与 `target_root`
- 前端展示“将关闭当前实例并迁移数据”的确认页

用户点击确认后：

1. 后端进入受控关闭流程
2. 当前实例退出
3. 迁移器根据检查点执行真实迁移
4. 迁移成功后自动重启
5. 新实例按新路径运行

## 3. 推荐实施阶段

### 3.1 Stage 1：先做网页端应用内弹窗 / 页面

这一阶段的目标只有一个：

- 让首次选择稳定发生在网页端应用内

建议实现：

1. 前端应用启动后，先调用轻量哨兵接口（如 `GET /api/v1/system/status`）
2. 若哨兵返回 `ready`，则不显示存储页，直接继续主页初始化
3. 若哨兵返回阻断状态，再调用 `GET /api/storage/location/bootstrap`
4. 如果需要首次确认或恢复确认，则显示：
   - 强制选择弹窗
   - 或首屏覆盖页
5. 页面展示：
   - 当前路径
   - 推荐路径
   - 已检测到的旧数据路径
   - 保持当前路径
   - 使用推荐路径
   - 选择其他位置
6. 在用户完成选择前，主功能区域不可继续操作
7. 页面配置加载、模型初始化以及主业务入口必须等待存储引导放行后再继续

本阶段当前只要求 Web UI 形态成立：

- 由现有网页前端壳子直接挂载弹窗或覆盖层
- 不要求桌面端独立窗口
- 不要求原生文件夹选择器
- 不要求桌面端重启过渡页

这一阶段明确不做：

- 真实迁移
- 运行根切换
- 自动重启

验收标准：

- 应用先正常打开
- 打开后若哨兵判定需要阻断，则立即出现选择弹窗或页面
- 用户可在当前界面内直接完成首轮选择
- 在存储页未放行前，主页主功能不会继续初始化
- 以上验收以网页端启动模式为准

### 3.2 Stage 2：补 `storage_policy` 与首轮提交接口

这一阶段目标：

- 把“只是显示”变成“能真正保存用户选择”

建议新增：

- `utils/storage_policy.py`
- `GET /api/storage/location/bootstrap`
- `POST /api/storage/location/select`

这一阶段同时要把主设计里已经定稿的稳定布局约束固化下来：

- `storage_policy` 保存“当前已正式生效”的稳定根
- 若用户请求迁移到不同路径，但迁移尚未成功，不得提前把 `selected_root` 提交为新值
- `storage_policy` 中要能恢复：
  - `anchor_root`
  - `selected_root`
  - `cloudsave_strategy=fixed_anchor`
  - `selection_source`

其中 `POST /api/storage/location/select` 应返回两类结果：

- `continue_current_session`
- `restart_required`

约束：

- 若用户保持当前路径，则直接 `continue_current_session`
- 若用户选择不同路径，则只返回 `restart_required`
- 不允许在当前会话里直接切根
- 若当前处于恢复态或其他阻断态，Stage 2 仍必须停留在存储选择页，不允许因为 `selection_required=false` 而错误放行主页

### 3.3 Stage 3：补迁移检查点与受控关闭

这一阶段目标：

- 把“需要重启”真正变成“当前实例关闭后可执行迁移”

建议新增：

- `utils/storage_migration.py`
- `storage_migration.json`
- `POST /api/storage/location/restart`

这一阶段不只要有“待迁移”标记，还要让检查点足够支撑重启后的恢复与诊断。

最低建议字段：

- `status`
- `source_root`
- `target_root`
- `selection_source`
- `requested_at`
- `error_code`
- `error_message`
- `backup_root`

建议控制流：

1. 前端先通过 `POST /api/storage/location/select` 保存目标路径
2. 若后端返回 `restart_required`
3. 前端展示确认页
4. 用户确认后调用 `POST /api/storage/location/restart`
5. 后端写下最终迁移检查点并启动受控关闭

这一阶段还要同时补齐网页端在“服务真空期”里的表现：

1. `POST /api/storage/location/restart` 一旦成功返回，前端立即切换到静态维护页
2. 维护页禁止展示“连接失败”或通用报错 toast
3. 文案应明确类似：
   - `正在优化存储布局...`
   - `当前实例即将关闭，数据会在关闭后迁移并自动重启`
4. 当前端检测到后端连接中断时，应维持维护页而不是退回主页
5. 前端随后轮询：
   - `GET /api/storage/location/status`
   - 或回退轮询 `GET /api/v1/system/status`
6. 一旦探测到新实例恢复并返回 `ready`，前端自动重连并放行主页

验收时不能只看“后端有没有关掉”，还要看：

- 关闭到重启之间，网页端是否稳定停留在维护页
- 服务恢复后是否自动重连，而不是要求用户手工刷新

验收标准：

- 选择不同路径时，当前实例会优雅退出
- 退出前不会尝试在当前会话里切根

### 3.4 Stage 4：补真实迁移与自动重启

这一阶段目标：

- 让“关闭后迁移、迁移后自动重启”完整打通

推荐迁移语义：

- `preflight`
- `copying`
- `verifying`
- `committing`
- `retaining_source`
- `failed`
- `completed`

迁移执行要求：

- 只在当前实例完全退出后执行
- 成功后更新 `storage_policy`
- 成功后清理或完成 `storage_migration` 检查点
- 成功后自动重启
- 失败时保留旧根可恢复

这一阶段还必须显式补上主设计文档中的“重启前布局恢复”步骤：

1. 新实例拉起后，先解析平台默认 `anchor_root`
2. 读取 `storage_migration`
3. 若存在 `pending` 迁移，则先完成迁移与提交
4. 迁移成功后再落盘最终 `storage_policy`
5. 把本次正式生效的布局写入环境变量或等价桥接
6. 子进程在 import 重型模块前先恢复最终布局
7. 然后才允许首次 `get_config_manager()`

如果这一步缺失，就会出现主设计文档已经明确指出的回归：

- 新实例服务进程再次先绑定旧根
- 之后再去读迁移检查点，导致迁移链路失效

### 3.5 Stage 5：补迁移完成页与清理入口

这一阶段再补：

- 启动后的“迁移完成”提示
- 旧数据备份查看入口
- 手动清理旧备份入口

这里要明确：

- 旧数据默认保留
- `v0.8` 不要求自动删除旧数据

## 4. 前后端职责划分

### 4.1 前端负责什么

前端负责：

- 在应用打开后立即请求存储状态
- 显示首轮选择弹窗或页面
- 展示“将关闭当前实例并迁移”的确认页
- 在服务真空期展示维护页，并保持自动重连
- 在重启后展示成功 / 失败结果

在本文当前范围里，这里的“前端”特指网页端前端，不包含桌面端专属窗口外壳。

### 4.2 后端负责什么

后端负责：

- 返回当前根、推荐根、旧数据根
- 保存 `storage_policy`
- 保存 `storage_migration`
- 触发受控关闭
- 在关闭后执行迁移
- 自动重启
- 在新实例里于首次重型 import 前恢复最终布局

### 4.3 明确禁止什么

不允许让前端自己判断：

- 当前实例是否已经正式切到新根
- 当前是否可以绕过关闭流程直接迁移
- 迁移失败时是否还能继续沿用半切换状态

## 5. 建议的代码接入点

### 5.1 主服务接口

建议新增或整理以下接口：

- `GET /api/storage/location/bootstrap`
- `POST /api/storage/location/select`
- `POST /api/storage/location/restart`
- `GET /api/storage/location/status`

### 5.1.1 网页端必须可见的字段契约

为了让网页端真正承担“意图捕获器”和“状态哨兵”，以下信息不能只存在于后端内部日志里，必须能通过接口返回给前端展示。

`GET /api/storage/location/bootstrap` 最低应提供：

- `current_root`
- `recommended_root`
- `legacy_sources`
- `selection_required`
- `migration_pending`
- `recovery_required`
- `blocking_reason`
- `last_error_summary`

当用户已经选定目标路径、准备进入关闭迁移前，`POST /api/storage/location/restart` 或等价预检接口最低应返回或缓存以下可展示字段：

- `target_root`
- `estimated_required_bytes`
- `target_free_bytes`
- `permission_ok`
- `warning_codes`
- `blocking_error_code`
- `blocking_error_message`

其中 `warning_codes` 至少应覆盖：

- 同步盘目录
- 外置盘 / 网络盘
- Windows CFA / 权限受限目录
- symlink / junction / reparse point
- 目标空间不足

`GET /api/storage/location/status` 最低应提供：

- `lifecycle_state`
- `migration_stage`
- `maintenance_message`
- `poll_interval_ms`
- `effective_root`
- `last_error_summary`

要求：

- “目标卷剩余空间”“预估迁移体量”“权限/风险警告”必须能被网页端明确展示
- 这些检查既是迁移预检，也属于用户可见的决策依据，不能只做内部校验不回传前端

### 5.2 配置与检查点模块

建议新增：

- `utils/storage_policy.py`
- `utils/storage_migration.py`
- `utils/storage_layout.py`

职责分别是：

- `storage_policy.py`
  - 保存稳定选择
- `storage_migration.py`
  - 保存待迁移检查点和结果
- `storage_layout.py`
  - 统一布局计算与路径规范化

这里应再补一层与主设计一致的职责拆分：

- `storage_policy.py`
  - 只保存当前已正式生效的稳定布局
- `storage_migration.py`
  - 只保存迁移过程检查点、失败信息与恢复信息
- `storage_layout.py`
  - 统一 `anchor_root / selected_root / cloudsave_root` 计算
  - 统一路径规范化与路径合法性校验

### 5.3 前端入口

前端建议在“应用壳 ready”之后尽快挂载存储位置引导逻辑。

建议形式：

- 全局阻断 modal
- 或专门的 first-run route / overlay

不建议：

- 单独做一个独立于应用主界面的首启入口
- 让用户自己从设置页深处再去找首启入口

当前范围补充：

- 这里默认入口就是网页端主 UI
- 不把桌面端专属启动壳纳入本阶段实现范围

## 6. 迁移执行约束

### 6.1 迁移必须晚于当前实例退出

这是本次实施最重要的技术约束。

原因：

- 当前实例里已经持有旧根相关句柄和缓存
- 运行中切根很容易造成读写分裂

因此迁移必须在：

- 当前实例完全退出后
- 新实例正式拉起前

补充要求：

- 新实例正式拉起后，也不能立刻让服务 import 重型模块
- 必须先读取 `storage_migration`
- 必须先恢复最终布局
- 必须先完成子进程布局传播
- 在这段窗口期内，网页端必须稳定停留在维护页，直到新实例明确返回 `ready`

### 6.2 成功语义

迁移成功至少意味着：

- 目标根数据完整可用
- `storage_policy.selected_root` 已更新
- 新实例将从新根启动
- 旧根如保留，已被标记为备份

### 6.3 失败语义

迁移失败时必须保证：

- 旧根仍然可回退
- 不把 `storage_policy` 写成半有效状态
- 下次启动仍能给出明确恢复选择

若用户之前已经选过自定义路径，而该路径在后续启动时离线或损坏，也应按主设计进入恢复态，而不是静默回退默认根继续写入。

---

## 7. 最容易做错的点

- 把“应用内弹窗”重新做成独立的首启入口
- 用户选了不同路径后，在当前会话里直接切根
- 把 cloudsave 跟随运行时根一起迁移
- 新实例里先起服务，再去处理 `storage_migration`
- 只写前端页面，不补受控关闭和自动重启
- 迁移失败时把策略写坏，导致新旧路径都不稳定
- 把旧数据自动删除作为默认行为

---

## 8. 最终完成定义

当下面这些条件全部成立时，才算这份实施文档真正落地：

- 应用第一次启动时先正常打开
- 打开后先做轻量状态探测；若判定需要阻断，则立即显示存储位置选择弹窗或页面
- 选择当前路径时，不重启，当前会话继续
- 在存储页未放行前，主页主功能不会继续初始化
- 选择不同路径时，当前实例关闭后才开始迁移
- 迁移完成后自动重启，不要求用户手工再次打开
- `cloudsave_root` 固定在 `anchor_root`
- 新实例里任何服务首次 `get_config_manager()` 前已拿到最终布局
- 整个流程都围绕当前应用界面、关闭流程和自动重启展开
- 当前交互与验收以网页端启动模式为准

桌面端启动相关的显示补充、窗口编排和专属体验，不作为本文当前完成定义的一部分。

一句话收敛：

- 实施主线就是“应用内完成选择，必要时关闭当前实例，迁移后自动重启”。
