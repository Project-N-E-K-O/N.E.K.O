# 插件误触发修复：修改原因与收益说明

## 一、为什么要修改

本次问题来自一次 `turn_end` 后的后台 Agent 分析：UserPlugin 评估模型在没有用户明确要求调用尖塔插件的情况下，选择并执行了 STS2 插件的快照读取入口。

日志中的关键现象是：

- 后台分析在 `turn_end` 阶段启动，且 `user_plugin_enabled` 为启用状态。
- UserPlugin 评估返回 `has_task=true` 和 `can_execute=true`。
- 被选择的入口是 `sts2_autoplay` 插件中的 `sts2_get_snapshot`。
- 插件执行成功后，前端只看到类似“插件任务已完成：done”的结果。

这暴露出两个独立但相关的问题：

1. 自动分派误触发：诊断/只读入口被暴露给 Agent 自动路由，模型容易把“读取快照”理解为有帮助的动作，从而在用户没有明确授权时调用插件。
2. 结果展示不完整：STS2 的若干只读入口声明了 `llm_result_fields=["summary"]`，但实际返回值缺少 `summary` 字段，导致结果解析回退成默认完成文本 `done`。

## 二、为什么这样修改

### 1. 引入入口级自动路由隐藏机制

修改 [`brain/task_executor.py`](brain/task_executor.py)，新增入口级隐藏判断 [`DirectTaskExecutor._is_plugin_entry_agent_hidden()`](brain/task_executor.py#L1000)。当插件入口 metadata 中存在以下标记时，该入口不参与 Agent 自动分派：

- `metadata={"agent_auto": False}`
- `metadata={"agent_exposed": False}`
- `metadata={"llm_exposed": False}`
- `metadata={"agent_hidden": True}`

同时修改：

- [`DirectTaskExecutor._build_plugin_desc_lines()`](brain/task_executor.py#L1039)：构造给 LLM 的插件入口描述时跳过隐藏入口。
- [`DirectTaskExecutor._find_plugin_entry()`](brain/task_executor.py#L1014)：LLM 决策校验和回退入口选择时也跳过隐藏入口。

这样做的原因是：

- 比把整个插件设为 passive 更细粒度。整个 STS2 插件仍可以保留自然语言控制能力。
- 只隐藏不适合自动触发的诊断/只读入口，不影响真正需要用户授权的动作入口。
- 框架层能力可复用，其他插件后续也可以用相同 metadata 控制哪些入口参与 Agent 自动路由。

### 2. 将 STS2 诊断/只读入口标记为不参与自动分派

修改 [`plugin/plugins/sts2_autoplay/__init__.py`](plugin/plugins/sts2_autoplay/__init__.py)，对以下入口加入 `metadata={"agent_auto": False}`：

- [`sts2_health_check`](plugin/plugins/sts2_autoplay/__init__.py#L75)
- [`sts2_refresh_state`](plugin/plugins/sts2_autoplay/__init__.py#L82)
- [`sts2_get_status`](plugin/plugins/sts2_autoplay/__init__.py#L89)
- [`sts2_get_snapshot`](plugin/plugins/sts2_autoplay/__init__.py#L100)
- [`sts2_get_history`](plugin/plugins/sts2_autoplay/__init__.py#L156)

这样做的原因是：

- 这些入口本质上是状态读取、健康检查、历史查看或调试辅助，不应该因为后台 `turn_end` 分析而自动调用。
- 真正会改变游戏行为或需要用户授权的入口仍保留给 Agent 自动路由，例如 [`sts2_play_one_card_by_neko`](plugin/plugins/sts2_autoplay/__init__.py#L115) 和 [`sts2_start_autoplay`](plugin/plugins/sts2_autoplay/__init__.py#L123)。
- 这能避免模型在“插件可能有帮助”的情况下选择最低风险的查询入口，从而造成无意义插件调用。

### 3. 补齐 STS2 只读入口返回值中的 summary

修改 [`plugin/plugins/sts2_autoplay/service.py`](plugin/plugins/sts2_autoplay/service.py)，让只读/诊断接口返回结构中包含 `summary` 字段：

- [`STS2AutoplayService.health_check()`](plugin/plugins/sts2_autoplay/service.py#L127)
- [`STS2AutoplayService.refresh_state()`](plugin/plugins/sts2_autoplay/service.py#L136)
- [`STS2AutoplayService.get_status()`](plugin/plugins/sts2_autoplay/service.py#L140)
- [`STS2AutoplayService.get_snapshot()`](plugin/plugins/sts2_autoplay/service.py#L169)
- [`STS2AutoplayService.get_history()`](plugin/plugins/sts2_autoplay/service.py#L735)

这样做的原因是：

- 这些插件入口在 [`plugin/plugins/sts2_autoplay/__init__.py`](plugin/plugins/sts2_autoplay/__init__.py) 中声明了 `llm_result_fields=["summary"]`。
- 结果解析函数 [`parse_plugin_result()`](brain/result_parser.py#L170) 会优先读取声明字段。
- 如果返回值没有 `summary`，解析器无法生成有效摘要，只能回退到默认完成文本。

补齐 `summary` 后，即使用户明确调用这些只读入口，也能得到有意义的简短结果，而不是只显示 `done`。

### 4. 收紧 UserPlugin 评估提示词

修改 [`config/prompts_agent.py`](config/prompts_agent.py)，在中文和英文 UserPlugin 评估提示词中加入更严格的触发条件：

- 只有用户最新请求明确要求插件能力、控制外部服务、查询插件状态/数据，或明确授权插件行动时，才允许返回 `has_task=true`。
- 禁止仅因为后台 `turn_end` 分析、插件可能有帮助、普通闲聊、未明确授权、或 AI 提到某功能而主动调用插件。

这样做的原因是：

- 入口隐藏解决的是“哪些入口可见”的问题。
- 提示词收紧解决的是“什么时候应该调用插件”的问题。
- 两者结合后，既减少误触发，也减少模型把普通对话解释成插件任务的概率。

### 5. 增加回归测试

修改 [`tests/test_agent_rewrite_regression.py`](tests/test_agent_rewrite_regression.py)，新增 [`test_task_executor_hides_agent_auto_disabled_plugin_entries()`](tests/test_agent_rewrite_regression.py#L342)。

测试覆盖：

- 带 `metadata={"agent_auto": False}` 的入口不会出现在给 LLM 的插件描述中。
- 当首选入口被隐藏时，入口查找逻辑不会继续选择隐藏入口，且返回的 entry 为空（即 `(plugin, None)`）。

这样做的原因是：

- 防止后续重构时重新把诊断入口暴露给自动路由。
- 固化本次修复的关键行为，避免回归。

## 三、解决了什么问题

本次修改解决了以下问题：

1. 解决 STS2 快照入口被后台自动分析误触发的问题。
2. 解决 `sts2_get_snapshot` 等只读入口执行后结果显示为 `done`、缺少有效摘要的问题。
3. 解决插件入口缺少“是否允许 Agent 自动路由”这一粒度控制的问题。
4. 降低 UserPlugin LLM 在普通对话、后台分析、未授权场景下主动调用插件的概率。
5. 保留 STS2 插件的核心自然语言控制能力，不因为修复误触发而关闭整个插件。

## 四、相比之前的优点

### 修改前

- 所有 STS2 entry 都暴露给 UserPlugin LLM。
- 诊断/只读入口和真正动作入口没有区分。
- LLM 在后台 `turn_end` 分析时可能选择 `sts2_get_snapshot` 这类“看似安全、有帮助”的入口。
- 返回值字段和 `llm_result_fields` 不匹配时，用户只能看到 `done`。
- 如果用插件级 passive 规避误触发，会连真正有用的自动游玩入口也一起禁用。

### 修改后

- 诊断/只读入口可以通过 metadata 隐藏，不参与 Agent 自动路由。
- 真正需要用户授权的动作入口仍可被正常选择。
- 插件结果返回结构和 `llm_result_fields` 对齐，展示更清晰。
- UserPlugin prompt 更明确要求“用户明确授权/明确请求”才调用插件。
- 框架层新增入口级控制能力，可被其他插件复用。
- 增加回归测试，降低未来改动引入同类问题的风险。
